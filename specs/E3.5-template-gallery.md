## Ziel

Das Design-Dashboard bekommt einen Bereich **„Templates"** mit 8 mitgelieferten, fertig gestalteten Designs (800×480): Wetter-Dashboard, Familienkalender, Foto+Uhr, Wochenplan, News-Briefing, Minimal-Uhr, Countdown, System-Monitor. Jede Karte zeigt eine **server-gerenderte, panel-echte Vorschau** (`POST /api/preview_live` ohne `raw` — der Nutzer sieht von der ersten Sekunde an, was sein Panel wirklich anzeigen wird, inkl. Live-Wetter und echter Uhrzeit). Ein Klick auf die Karte erstellt eine **Kopie als neues, NICHT aktives Design** (kollisionssicherer Name) und öffnet sie im Designer. Die Templates werden als statische JSONs über den bestehenden `go:embed static/*`-Mechanismus ausgeliefert — kein Build-Step, kein neuer Endpoint, kein zusätzlicher RAM.

**Warum:** Das ist der wichtigste Wow-Faktor (Manager-Auftrag): Ein frisch installiertes System hat heute genau ein leeres „Default Design" und eine leere Canvas — der Weg zum ersten hübschen Panel-Bild ist lang (Widgets platzieren, Properties verstehen, Layout bauen). Mit der Galerie ist das erste vorzeigbare Display 30 Sekunden entfernt, und die Templates dienen zugleich als lebende Dokumentation dafür, wie gute E-Ink-Layouts in diesem System aussehen.

### Verifizierte Fakten: Bestandsaufnahme (main, `5a093eb`)

1. **DesignV2 ist das Template-Format, der Renderer kann alles Nötige:** `models/design.go:113–171` — Element `{id, type, x, y, width, height, rotation, zIndex, locked, visible, properties{}}`, Design `{name, version:2, canvas{width,height,background}, elements[]}`. Der Server rendert (`services/preview.go:334–378`): `text`/`i-text`/`textbox`, `image`, `shape` (fill/stroke/strokeWidth/rx), `widget_clock`, `widget_weather`, `widget_forecast`, `widget_calendar`, `widget_news`, `widget_timer`, `widget_custom`, `widget_system`. Das Frontend baut aus exakt denselben Typen Fabric-Objekte (`element-factory.js:266–318`). Referenzbeispiel: `server/internal/services/testdata/designs/basic.json`.
2. **Widget-Layout-Presets und real konsumierte Properties:** `services/widgets/layouts.go` definiert die Layout-IDs (clock: `digital_large|digital_with_seconds|digital_with_date|date_only|full|custom`; weather: `compact|standard|detailed|minimal|custom`; forecast: `vertical|compact_row|detailed_list|custom`; calendar: `list|agenda|compact|detailed`*; news: `headlines|summary|single|custom`; system: `vertical|horizontal|compact|custom`; timer: `countdown_large|countdown_compact|label_above|days_only|custom`). Die `fill*`-Methoden (`preview.go:448–841`) konsumieren: clock `layout,timezone,customTemplate`; weather/forecast `latitude,longitude,layout,days,customTemplate`; calendar `icalUrl,maxEvents,daysAhead,layout,title`; news `feedUrl,maxItems,layout,title,showDescription`; timer `targetDate,layout,label,finishedText,customTemplate`; system `layout,showLabels,customTemplate`. *Achtung: calendar-Layout `detailed` und die System-Platzhalter `%uptime%`/`%hostname%` sind in layouts.go gelistet, aber im Renderer NICHT implementiert (`applySystemPlaceholders` kennt nur `%cpu%|%memory%|%temperature%`, calendar-switch kennt nur `agenda|compact|list`) — Templates dürfen sie nicht verwenden.
3. **Leerzustände sind verifiziert (exakte Strings, als normaler Widget-Text gerendert):** weather ohne Koordinaten → Default **Berlin 52.52/13.41** (`preview.go:525–526`), bei Fetch-Fehler `"No data"` (30-min-Cache + stale-on-error, `weather.go:139–147`); forecast `"No forecast data"`; calendar ohne `icalUrl` → **`"No calendar URL"`** (`preview.go:621–623`), URL-Fehler → `"No events"`; news ohne `feedUrl` → `"No feed URL"`, Fetch-Fehler → `"No news"`; timer: ungültiges `targetDate` → `"Invalid timer target"`, abgelaufen → `finishedText`; system auf Nicht-Linux → `"Load: N/A"`-Zeilen. Alle Fallbacks rendern in Font/Farbe/Ausrichtung des Widgets — die Templates müssen diese Zustände bewusst mitgestalten (Richtung 2), Layout bricht nie (Render-Fehler gibt es nur bei PNG-Encode/Context-Abbruch).
4. **„Template übernehmen" braucht KEINEN neuen Endpoint:** `POST /api/designs {name, elements, canvas}` (handlers/design.go:215–236 → `CreateDesign`, design.go:570–611) erstellt ein Design mit generierter ID und **`Active: false`** — Aktivierung ist ein separater `POST /api/designs/{id}/activate`. Rückgabe ist das volle Design inkl. `id`; `DesignDashboard.openDesign(id, name)` (design-dashboard.js:125–140) lädt es in den Designer. **Namenskollisionen verhindert der Server NICHT** (CreateDesign dedupliziert nicht; die Legacy-Endpoints `Get(name)`/`SetActive(name)` nehmen den ERSTEN Treffer, design.go:322–334) → die Kollisionsvermeidung muss clientseitig passieren (Richtung 4).
5. **`preview_live` taugt für Template-Vorschauen:** `POST /api/preview_live` rendert ein DesignV2 aus dem Request-Body OHNE zu speichern (handlers/preview.go:64–85); ohne `raw`-Param quantisiert gegen die Panel-Palette des konfigurierten Displays. Render-Semaphore global, Default 1, Abbruch → 503 (E5.6). Das E3.6-Muster in `toolbar.js` (Session-Zähler, Objekt-URL-Cache, In-Flight-Guard) ist die Vorlage für die Galerie-Queue (Richtung 3).
6. **Statische Auslieferung ist fertig verdrahtet:** `//go:embed static/*` (main.go:25–26) embeddet rekursiv, `GET /static/` wird ohne Auth ausgeliefert (middleware/auth.go:44–45, 85; FileServer main.go:241–242). Neue Dateien unter `server/static/templates/` sind damit ohne eine Zeile Go unter `/static/templates/…` erreichbar (Content-Type `application/json` via Extension). Templates enthalten keine Nutzerdaten → public ist ok. Die CLAUDE.md-Regel „server/static/ niemals bearbeiten" ist durch die E3.x-Praxis überholt (alle Frontend-Tasks editieren static/js) — hier kommen ausschließlich NEUE Dateien hinzu.
7. **Dashboard-Karten existieren, Thumbnails nicht:** `#dashboard-overlay` mit Sections „Recently Edited"/„All Designs" (designer.html:401–417); `DesignDashboard.createCard()` rendert `.design-card` mit `.no-preview`-Platzhalter (design-dashboard.js:77–123, E3.6-Fakt 3 bestätigt). Das CSS ist vollständig vorhanden — inkl. **`.design-card-preview img` (media-dashboard.css:403)**, ein Karten-`<img>` ist also bereits styliert. Dashboard öffnet per Logo-Klick (designer.js:292–294); `DesignDashboard.isOpen` existiert als Flag.
8. **Fonts & Roundtrip-Fallen:** Frische Installationen haben keine hochgeladenen Fonts; `loadFontFace` fällt über System-Noto/DejaVu auf das embedded `goregular` zurück (preview.go:1126–1157; Docker installiert font-noto/ttf-dejavu). Templates dürfen deshalb **kein `fontFamily`** setzen. Zweite Falle: `storage.js extractProperties` (Z. 117–152) baut Text-Properties beim Speichern aus dem Fabric-Objekt neu auf — `verticalAlign` geht bei `text`-Elementen VERLOREN, `textbox` wird beim Laden zu `text` degradiert (element-factory.js fromElement). Widget-Properties werden dagegen 1:1 durchgereicht (inkl. unbekannter Keys wie `templateSlot`).
9. **Test-Harness ist wiederverwendbar:** `golden_test.go` liefert `setupGoldenServices` (gepinnte Settings, deterministische Font/Bild-Umgebung) und **`assertPaletteExactness`** (Z. 220–280: Output ist `*image.Paletted`, Palette == `DisplayConfig.Colors` in Länge/Reihenfolge, jeder Pixel ∈ Driver-Palette, ≥ 2 Farben genutzt) — beides package-lokal in `services` nutzbar. Display-Konstanten: `waveshare_7in5_v2` (2 Farben), `waveshare_7in3_e` (6: `#000000/#FFFFFF/#FF0000/#FFFF00/#00FF00/#0000FF`, models/display.go:24–45). Die Golden-PNGs selbst dürfen sich nicht ändern.
10. **Farbphysik für die Gestaltung:** Exakte Driver-Palettenfarben quantisieren auf dem 6-Farb-Panel zu sich selbst — Flächen und Text in `#FF0000` etc. bleiben **knackscharf, ohne Dithering**. Auf dem B/W-Panel dithern alle Nicht-B/W-Farben nach Luminanz (Näherung: Gelb ≈ hell, Grün ≈ mittel, Rot/Blau ≈ dunkel). Konsequenz: Templates verwenden ausschließlich Palettenfarben (Richtung 2) — 6-Farb-Rendering ist dann pixelscharf, B/W bleibt kontrastlesbar.

### Bindende Architektur-Richtung

1. **Auslieferung — statisch, null Go:** Templates liegen als `server/static/templates/<id>.json` + Manifest `server/static/templates/index.json` im Repo und wandern über den bestehenden `go:embed static/*` ins Binary. Manifest-Schema (bindend):
   ```json
   { "templates": [ { "id": "weather-dashboard", "name": "Weather Dashboard",
       "description": "Current conditions with 3-day forecast", "file": "weather-dashboard.json",
       "setup": [] } ] }
   ```
   `setup` listet nötige Nutzer-Eingaben als kurze englische Labels (z. B. `["Calendar ICS URL"]`, `["Your photo"]`) und steuert den Karten-Badge. Kein Build-Step, kein neuer Endpoint, kein Thumbnail-Verzeichnis.

2. **Template-Katalog (bindend: 8 Stück) und Gestaltungsregeln.** Die Feinarbeit (exakte Koordinaten, Größenabstufungen) ist Design-Arbeit des frontend-designers, aber diese Regeln sind Pflicht: Canvas exakt 800×480, `background: "#FFFFFF"`; **alle** Farbwerte (`color`, `fill`, `stroke`, Canvas-Background) sind exakte 6-Farb-Driver-Farben (Fakt 10; `transparent` für fill/stroke erlaubt); genau EINE Akzentfarbe pro Template, Fließtext immer Schwarz auf Weiß bzw. Weiß auf Schwarz, farbiger TEXT nur ≥ 28 px (B/W-Dithering, Fakt 10); Außenränder ≥ 24 px, viel Weißraum, klare Typo-Hierarchie (max. 3 Schriftgrößen pro Template); **kein `fontFamily`**, **kein `verticalAlign`**, **nur Element-Typ `text`** für Texte (Fakt 8), Rotation 0; Element-IDs `tpl_<templateid>_<n>` eindeutig; alle Element-Boxen vollständig innerhalb der Canvas; `widget_custom`, calendar-Layout `detailed`, `%uptime%`/`%hostname%` verboten (Fakt 2); Leerzustands-Strings (Fakt 3) müssen im Layout eingeplant sein (keine Winz-Boxen, in denen „No calendar URL" clippt). Der Katalog:

   Sprachregelung: Karten-/UI-Texte Englisch (konsistent zum Dashboard), Panel-Inhalte der Templates Deutsch (konsistent zu `formatGermanDate`, das Datum/Wochentage hart deutsch rendert — preview.go:486–491).

3. **Galerie-UI + sequenzielle, gecachte Live-Previews (bindend):** Neue Section im `#dashboard-overlay` zwischen „Recently Edited" und „All Designs": `<div class="templates-section"><h2>Templates</h2><div class="designs-grid" id="templates-grid"></div></div>`. Karten nutzen die Bestands-Klassen (`.design-card`, `.design-card-preview`, `.design-card-body`, `.design-card-name`, `.design-card-date` für die description); Setup-Badge über `.design-card-badge` + neue neutrale Variante (einzige CSS-Ergänzung). `TemplateGallery.ensureLoaded()` (vom `DesignDashboard.open()`-Hook gerufen) lädt das Manifest EINMAL pro Page-Session und rendert sofort alle Karten mit `.no-preview`-Ladezustand („Rendering…"). Danach lädt eine **strikt sequenzielle Queue** die Vorschauen in Kartenreihenfolge: pro Template genau EIN `POST /api/preview_live` (Body = Template-JSON nach Preview-Substitution, s. Richtung 4; OHNE `raw` — panel-echt), Antwort-Blob als Objekt-URL in einem **Page-Session-Cache** (`{id: url}`), `<img>` in die Karte. Nie mehr als ein Preview-Request in flight (E5.6-Vertrag); die Queue pausiert, wenn `DesignDashboard.isOpen` false wird, und läuft beim nächsten Öffnen weiter; erneutes Öffnen mit vollem Cache löst NULL neue Requests aus. Fehler/503 pro Karte: `.no-preview` zeigt den Template-Namen als Fallback, **kein Auto-Retry** (nächste Page-Session versucht es neu), die Queue macht mit der nächsten Karte weiter. Kein `localStorage`, keine Persistenz, nur `click`-Handler (E3.1-Disziplin).

4. **Use-Flow + Instanziierungs-Pass (bindend):** Klick auf die Karte = Template übernehmen (title-Attribut „Use this template"). Ablauf: (a) Deep-Copy des Template-JSONs; (b) **Substitution 1 — Timer-Token:** der String `__NEXT_NEW_YEAR__` in String-Property-Werten wird zu `"<Jahr+1>-01-01 00:00:00"` (lokales Jahr); (c) **Substitution 2 — Foto-Slot:** Elemente mit `properties.templateSlot === "photo"`: existiert mindestens ein Bild in der Media-Library (`GET /api/media/images?page=1&limit=1`, neuestes zuerst) → `properties.image` = dessen `filename`; sonst wird das Element ENTFERNT (der gestaltete Platzhalter darunter bleibt); `templateSlot` wird danach aus den Properties gelöscht; (d) **Namenskollision:** Name = Manifest-`name`; solange er in `DesignDashboard.designs` (oder unter den in dieser Session erzeugten) existiert, wird ` 2`, ` 3`, … angehängt (Fakt 4 — der Server dedupliziert nicht, Legacy-Endpoints sind namensbasiert); (e) `POST /api/designs {name, elements, canvas}` → **nicht aktiv** (Fakt 4), danach `DesignDashboard.openDesign(id, name)` (schließt Dashboard, lädt Canvas, refresht Widgets) + Notification. **Keine Auto-Aktivierung** — „Set Active" bleibt bewusster Nutzerklick. Für die **Karten-Vorschau** wird nur Substitution 1 angewendet (der leere Foto-Slot rendert serverseitig als No-op, der Platzhalter darunter ist sichtbar — ehrliche Vorschau).

5. **Go-Render-Test (bindend, test-engineer):** Neue Datei `server/internal/services/template_render_test.go` (package `services`, Templates via `filepath.Join("..", "..", "static", "templates")`). Zwei Teile: **(a) Statik-Lint** pro Template: Manifest ↔ Dateibestand konsistent (jede gelistete Datei existiert, keine ungelistete `*.json` außer `index.json`), IDs eindeutig, `version == 2`, Canvas 800×480 + `#FFFFFF`, Element-Typen ∈ erlaubter Menge (Richtung 2, ohne `widget_custom`/`textbox`/`i-text`), Element-IDs eindeutig, Rotation 0, Boxen innerhalb der Canvas, alle Farb-Properties ∈ 6-Farb-Driver-Palette ∪ {`transparent`}, kein `fontFamily`/`verticalAlign`, timer-`targetDate` ist `__NEXT_NEW_YEAR__` oder ein parsebares Zukunftsdatum. **(b) Render:** für jedes Template × beide Display-Profile (`setupGoldenServices`, quality high): Token-Substitution wie Richtung 4b (Helper via `strings.ReplaceAll` auf dem Roh-JSON), `Render(ctx, design, false)` → **kein Fehler**, dann `assertPaletteExactness` (Fakt 9). KEINE Golden-Bytes für Templates (Uhrzeit/Live-Daten variieren) — offline degradieren die Widgets auf ihre Fallback-Strings (Fakt 3) und der Test bleibt grün; er darf Netz nutzen, aber nie davon abhängen.

6. **Stack-Invarianten:** Vanilla JS, kein npm, keine Dependency, kein neuer Endpoint/Query-Param, kein Go-Produktionscode, kein localStorage, keine `mouse*`-Listener, Golden-PNGs unverändert. Kein gebündeltes Foto-Asset und keine Änderung der Empty-State-Strings in Go.

## Non-Goals

- **Kein Template-Editor / keine Template-Verwaltung** — Templates sind read-only Ship-Assets; Ändern/Löschen/Eigene-hochladen ist ein eigener Task.
- **Keine Nutzer-Uploads als Templates** und kein „Design als Template speichern".
- **Keine Auto-Aktivierung** — übernehmen ≠ aufs Panel schieben (bewusste Produktentscheidung, Manager-Vorgabe).
- **Kein Thumbnail-Cache-System und kein neuer Endpoint** — lazy + sequenziell + Page-Session-Cache genügt (E3.6-Non-Goal „Dashboard-Thumbnails" wird hier bewusst NUR für Template-Karten eingelöst, weil deren Anzahl fix (8) und ihr Inhalt quasi-statisch ist; Design-Karten bleiben ohne Thumbnails). Vorab gerenderte, committete Thumbnail-PNGs wurden verworfen: sie zeigten weder das konfigurierte Display-Profil noch Live-Daten.
- **Kein gebündeltes Beispielfoto** — der Foto-Slot arbeitet mit Platzhalter bzw. dem neuesten Nutzer-Upload (Richtung 4c).
- **Keine i18n-Infrastruktur** — Sprachregelung nach Richtung 2 (UI Englisch, Panel-Inhalte Deutsch).
- **Keine Go-Produktionsänderungen** — auch nicht an den Empty-State-Strings (Fakt 3); hübschere Leerzustände sind Gestaltungsaufgabe der Templates.
- **Keine Preview-Persistenz** über die Page-Session hinaus (kein localStorage, keine Server-Caches).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Live-Daten variieren / offline: Wetter „No data", News „No news", Kalender ohne URL | Fallback-Strings sind verifiziert (Fakt 3) und in den Layouts eingeplant (Richtung 2); AC5/AC6 beschreiben die Zustände explizit. Render schlägt dadurch nie fehl. |
| Externe Calls beim Rendern (open-meteo, tagesschau-Feed) — Galerie-Previews erzeugen Fetches | Wetter: 30-min-Cache + stale-on-error (Fakt 3). News: ungecacht, aber durch die Queue + Session-Cache max. 1 Feed-Fetch pro Page-Session; 10-s-HTTP-Timeout begrenzt Hänger. Default-Feed ist im Manifest-Text als austauschbar deklariert. |
| Pi Zero: 8 Renders beim ersten Dashboard-Öffnen dauern (Semaphore 1, ~Sekunden/Render) | Bewusst sequenziell + progressiv (Karten füllen sich nacheinander, Ladezustand sichtbar); Cache macht jedes weitere Öffnen kostenlos; ein konkurrierender Client-Refresh schiebt sich dazwischen → einzelne Karte fällt auf 503-Fallback, kein Sturm (AC6a). |
| Countdown-Token wird nicht substituiert (Preview oder Use) → „Invalid timer target" auf dem Panel | Doppelt abgesichert: L2-Assert U4 (kein Literal im erstellten Design) + Go-Test substituiert wie das Frontend und schlüge bei Format-Drift fehl (Richtung 5). |
| Namenskollision trifft Legacy-Endpoints (namensbasiert, erster Treffer) | Kollisions-Suffix clientseitig (Richtung 4d, AC3); IDs bleiben ohnehin eindeutig. Restrisiko: Nutzer benennt manuell auf Duplikat — Bestandsverhalten, nicht Scope. |
| Roundtrip-Verluste beim ersten Speichern im Designer (fontFamily → „Arial", verticalAlign weg, textbox → text) | Templates verwenden die verlustfreie Teilmenge (Richtung 2, per Go-Lint erzwungen); „Arial" fällt serverseitig auf die Default-Font-Kette zurück (Fakt 8) — Panel-Bild bleibt stabil. |
| CLAUDE.md sagt „server/static/ niemals bearbeiten" | Überholte Regel, E3.x-Praxis editiert static/js längst; hier ausschließlich NEUE Dateien (Fakt 6). Im Merge-Request explizit benennen; CLAUDE.md-Korrektur ist ein separater Chore. |
| Öffentliche Erreichbarkeit der Template-JSONs (`/static/` ist auth-frei) | Ship-Assets ohne Nutzerdaten — unkritisch (Fakt 6); die Galerie-Funktionen (preview_live, designs-API) bleiben session-geschützt. |
| Paralleles designer.html-Editing (E3.7 u. a.) | Templates-Section liegt isoliert im Dashboard-Overlay-Block (Z. 401–417-Region); trunk-based, klein mergen, rebase (Koordination). |
| Ästhetik ist nicht vollautomatisch prüfbar | Bewusste Doppelstrategie: Go-Test erzwingt das Mechanische (Palette, Bounds, Regeln), AC5 erzwingt eine beschreibende Sichtprüfung mit Merge-Veto — „rendert fehlerfrei" allein reicht ausdrücklich nicht. |
