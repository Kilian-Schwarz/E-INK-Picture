# E1.3: Konsistenter Display-Default Server↔Client (`EINK_DISPLAY_TYPE`)

**Status:** Draft
**Abhängigkeiten:** keine (unabhängig von E1.1/E1.2 umsetzbar)
**Blockiert:** verlässliche Frisch-Installationen; ohne diesen Task rendert der Server B/W während der Client 6-Farb-Hardware treibt
**Zuständig:** go-backend (nur Server-Teil + Compose/Env-Dateien); Changelog-Notiz: docs-writer

## Ziel

Eine frische Installation (keine `settings.json`) rendert nie mit dem falschen Panel-Profil, weil der Server-Default für `display_type` per Env-Var konfigurierbar wird und sein Fallback auf dasselbe Panel zeigt wie der Client-Fallback (Treiber `epd7in3e`).

## Kontext

### Begriffsklärung (bindend — hier ging in der Discovery schon einmal etwas durcheinander)

Zwei verschiedene Namensräume, nicht verwechseln:

| Ebene | Werte | Quelle |
|---|---|---|
| **display_type** (Server, `settings.json`, neue Env-Var) | `waveshare_7in5_v2`, `waveshare_7in3_e` | `server/internal/models/display.go:7-8` |
| **driver** (Client, Waveshare-Modulname) | `epd7in5_V2`, `epd7in3e` | `display.go:32,42` (Feld `Driver`), `client/config.py:5` |

Der Client-Fallback ist der **Driver** `epd7in3e` (`client/config.py:5`). Das serverseitige Äquivalent ist der **display_type** `waveshare_7in3_e` (dessen `Driver`-Feld exakt `epd7in3e` ist). Der Fallback der neuen Env-Var ist daher **`waveshare_7in3_e`** — der String `epd7in3e` ist KEIN gültiger `DisplayType` und würde von `models.GetDisplayConfig` stillschweigend auf das B/W-Profil zurückfallen (`display.go:56-61`).

### Ist-Zustand (verifiziert)

- Server-Default ist an **sechs Stellen** hartcodiert `models.DisplayWaveshare75V2` (B/W): `server/internal/services/settings.go` Z. 40 und 50 (`GetSettings`-Defaults bei fehlender/kaputter Datei), Z. 56 (leeres Feld), Z. 84 (`GetDisplayConfig`-Fehlerpfad), Z. 123 (`TriggerRefresh`), Z. 149 (`RecordClientRefresh`).
- **Verschärfung:** `TriggerRefresh` und `RecordClientRefresh` **persistieren** den Default: ein `POST /api/trigger_refresh` oder Client-Heartbeat auf frischer Installation schreibt `settings.json` mit `display_type: waveshare_7in5_v2` — der falsche Default wird dauerhaft festgenagelt, bevor der Nutzer je die Settings-UI gesehen hat.
- Client übernimmt beim Poll den Treiber vom Server: `fetch_display_config()` in `client/client.py` (liest `display.driver` aus `GET /settings`, lädt ggf. Treiber neu). Ergebnis heute: frische Installation → Server rendert/meldet B/W-Profil, obwohl Client-Default und Compose-Default (`docker-compose.yml:42`, `EINK_DISPLAY_DRIVER=epd7in3e`) 6-Farb-Hardware annehmen.
- Env-Konfiguration des Servers läuft zentral über `server/internal/config/config.go` (`Config`-Struct + `getEnv(key, fallback)`, Z. 16-34); `main.go:57` baut `services.NewSettingsService(cfg.DataDir)`.
- `NewSettingsService` wird außerdem an 12 Stellen in Tests aufgerufen (`settings_test.go`, `preview_test.go` u. a.) — Signaturänderung zieht Test-Anpassungen nach sich.
- Env-Beispiele: Root-`.env.example` (dokumentiert bereits Server- und Client-Variablen; ein `server/.env.example` existiert NICHT), `client/.env.example` (nur Client). Compose: `docker-compose.yml` (Server-Service Z. 10-19 `environment`), `docker-compose.cloud.yml` (Override, merged `environment` per Variable).

### Bindende Richtung (Manager-Entscheid)

Single Source of Truth bleibt der Server. Der Server-Default wird per `EINK_DISPLAY_TYPE` konfigurierbar; Fallback `waveshare_7in3_e` (= Client-Fallback-Panel). Env greift NUR, wenn `settings.json` keinen Wert hat — bestehende Installationen ändern sich nicht. `client/config.py` bleibt unangetastet.

### Umsetzungsskizze

1. `config.go`: neues Feld `EInkDisplayType` mit `getEnv("EINK_DISPLAY_TYPE", "")` (leer = nicht gesetzt; Fallback-Logik gehört in den Settings-Service, nicht ins Config-Package).
2. `NewSettingsService(dataDir string, defaultDisplayType models.DisplayType)`: validiert den Parameter gegen `models.DisplayProfiles`; leer → `models.DisplayWaveshare73E`; nicht-leer aber unbekannt → `slog.Warn` + `models.DisplayWaveshare73E`. Ergebnis wird als Feld gespeichert.
3. Alle sechs Hardcode-Stellen in `settings.go` ersetzen durch das Service-Feld (eine zentrale Stelle, kein weiteres `DisplayWaveshare75V2` in `services/`).
4. `main.go:57`: `services.NewSettingsService(cfg.DataDir, models.DisplayType(cfg.EInkDisplayType))`.
5. `docker-compose.yml` Server-Service: `- EINK_DISPLAY_TYPE=${EINK_DISPLAY_TYPE:-waveshare_7in3_e}`. Root-`.env.example` dokumentiert die Var mit beiden gültigen Werten.

## Akzeptanzkriterien

1. **AC1 — Env-Var nach Haus-Muster:** `config.go` liest `EINK_DISPLAY_TYPE` über `getEnv` in ein neues `Config`-Feld; kein `os.Getenv` außerhalb von `config.go`.
   *Beweis:* Code-Review + `grep -rn "EINK_DISPLAY_TYPE" server/ --include="*.go"` trifft (außer Tests) nur `config.go`.

2. **AC2 — Frisch + Env gesetzt → Env gewinnt:** Neuer Test: `SettingsService` mit `t.TempDir()` ohne `settings.json`, Default-Parameter `waveshare_7in5_v2` → `GetSettings().DisplayType == models.DisplayWaveshare75V2` und `GetDisplayConfig().Colors` hat Länge 2.
   *Beweis:* neuer Test in `settings_test.go` (Name z. B. `TestSettingsService_DefaultDisplayTypeFromEnv`).

3. **AC3 — Frisch ohne Env → `waveshare_7in3_e`:** Gleicher Aufbau mit leerem Default-Parameter → `GetSettings().DisplayType == models.DisplayWaveshare73E`, `GetDisplayConfig().Driver == "epd7in3e"`, `len(Colors) == 6`.
   *Beweis:* neuer Test.

4. **AC4 — Bestehende `settings.json` gewinnt immer:** Vorhandene Datei mit `{"display_type":"waveshare_7in5_v2", ...}` + Default-Parameter `waveshare_7in3_e` → `DisplayType` bleibt `waveshare_7in5_v2`; auch nach `TriggerRefresh()` und `RecordClientRefresh()` steht in der Datei weiterhin `waveshare_7in5_v2`.
   *Beweis:* neuer Test inkl. Re-Read der Datei nach beiden Aufrufen.

5. **AC5 — Ungültiger Env-Wert fällt sicher zurück:** Default-Parameter `epd7in3e` (der klassische Verwechsler) oder `banana` → `slog.Warn` + effektiver Default `waveshare_7in3_e`. Kein Panic, kein stiller B/W-Fallback.
   *Beweis:* neuer Test (Warn-Log-Assertion optional; entscheidend ist der effektive Default).

6. **AC6 — Kein Festnageln des falschen Defaults mehr:** Frische Installation (keine `settings.json`), Default `waveshare_7in3_e`: nach `TriggerRefresh()` bzw. `RecordClientRefresh()` enthält die persistierte `settings.json` `display_type: waveshare_7in3_e` — nicht mehr `waveshare_7in5_v2` (heutiges Verhalten via settings.go:123/149).
   *Beweis:* neuer Test, liest die geschriebene Datei.

7. **AC7 — Grep-AC (zentraler Default):** Nach Umsetzung liefert
   `grep -rn "DisplayWaveshare75V2" server/ --include="*.go" | grep -v _test.go`
   Treffer NUR noch in `server/internal/models/display.go` (Profil-Definition Z. 25-34 und Last-Resort-Fallback in `GetDisplayConfig` Z. 60 — siehe Non-Goals). In `server/internal/services/` und `server/main.go`: null Treffer.
   *Beweis:* Grep-Kommando unter „Verifikation".

8. **AC8 — Compose + Env-Doku konsistent:** `docker-compose.yml` gibt dem **Server**-Service `EINK_DISPLAY_TYPE=${EINK_DISPLAY_TYPE:-waveshare_7in3_e}`; Root-`.env.example` dokumentiert `EINK_DISPLAY_TYPE=waveshare_7in3_e` mit Kommentar (gültige Werte: `waveshare_7in3_e` = 6-Farb/epd7in3e, `waveshare_7in5_v2` = B/W/epd7in5_V2; greift nur ohne `settings.json`-Wert). `docker-compose.cloud.yml` braucht keinen Eintrag (Compose merged `environment` per Variable), das ist im MR kurz zu verifizieren.
   *Beweis:* `grep -n EINK_DISPLAY_TYPE docker-compose.yml .env.example` trifft beide Dateien.

9. **AC9 — Bestand grün + Hygiene:** Alle bestehenden Tests (inkl. der 12 angepassten `NewSettingsService`-Aufrufe, die den bisherigen B/W-Default erwarten und explizit `models.DisplayWaveshare75V2` übergeben dürfen, wo das Testverhalten davon abhängt) laufen durch; `gofmt -l .` leer, `go vet ./...` sauber.
   *Beweis:* Kommandos unter „Verifikation".

10. **AC10 — Changelog-Notiz (docs-writer):** `CHANGELOG.md` (Keep a Changelog, „Changed") dokumentiert: Frische Installationen ohne `settings.json` starten jetzt mit dem 6-Farb-Profil `waveshare_7in3_e` statt B/W; Nutzer mit 7,5"-V2-Panel ohne persistierte Settings müssen `EINK_DISPLAY_TYPE=waveshare_7in5_v2` setzen oder den Typ in der Settings-UI wählen.
    *Beweis:* `grep -n EINK_DISPLAY_TYPE CHANGELOG.md`.

## Non-Goals

- **`client/config.py` bleibt unangetastet** (bindend) — der Client-Fallback `EINK_DISPLAY_DRIVER=epd7in3e` und die Treiber-Übernahme vom Server (`client/client.py`, `fetch_display_config`) bleiben wie sie sind.
- **Keine Migration/Umschreibung bestehender `settings.json`** — auch keine „Reparatur" ungültiger persistierter Werte: eine Datei mit unbekanntem `display_type` (z. B. Alt-Casing `waveshare_7in5_V2`) fällt weiterhin über `models.GetDisplayConfig` (display.go:56-61) auf das B/W-Profil zurück. Dieses Verhalten zu ändern könnte reale Alt-Installationen kippen und ist bewusst außen vor (deshalb die Ausnahme in AC7).
- Kein Umbenennen oder Deprecaten von `EINK_DISPLAY_DRIVER` (Client-Var).
- Keine UI-Änderungen (Settings-Seite kennt den Typ bereits).
- Kein zusätzliches `server/.env.example` — die Root-Datei ist der dokumentierte Ort.
- Keine Validierung von per API gesetzten Werten über das Bestehende hinaus.

- keine `client/`-Änderungen im Diff (Non-Goal).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| **Verhaltensänderung für 7,5"-V2-Nutzer ohne `settings.json`:** nach Update rendert der Server 6-Farb-Profil; Client lädt via `fetch_display_config` ggf. den falschen Treiber (ImportError → Preview-only) oder das B/W-Panel bekommt 6-Farb-PNGs | Bewusst akzeptiert (Manager-Entscheid: Konsistenz mit Client-Default). Mitigation: Changelog-Eintrag (AC10), `.env.example`-Doku (AC8), Fix per einmaligem `EINK_DISPLAY_TYPE=waveshare_7in5_v2` oder Settings-UI. Betroffen sind nur Installationen, die noch NIE Settings gespeichert haben und kein Env setzen. |
| Tippfehler/Driver-String in `EINK_DISPLAY_TYPE` (z. B. `epd7in3e`) | Durch AC5 abgefangen: Warnung + definierter Fallback `waveshare_7in3_e` statt stillem B/W-Fallback von `models.GetDisplayConfig`. |
| Signaturänderung `NewSettingsService` bricht parallele Branches/Agents | Kleine, mechanische Anpassung (13 Aufrufstellen inkl. main.go); beschreibende Commit-Message nach Konvention; kein API-/Wire-Format betroffen. |
| Docker-Bestandsinstallationen kippen beim Image-Update | Nein: `settings.json` liegt im gemounteten `./data`-Volume und gewinnt immer (AC4). Nur wirklich frische Volumes sind betroffen. |
| `docker-compose.cloud.yml` überschreibt Server-`environment` und verliert die neue Var | Compose merged `environment` pro Variable; im MR per `docker compose -f docker-compose.yml -f docker-compose.cloud.yml config` gegenprüfen (AC8-Hinweis). |
| **Rollback** | Ein Server-Commit (+ Compose/Env-Doku): `git revert <commit>` stellt den alten B/W-Default wieder her. Vorsicht: In der Zwischenzeit durch AC6-Verhalten frisch persistierte `settings.json` mit `waveshare_7in3_e` bleiben bestehen — das ist korrekt (explizit gespeicherter Zustand), muss beim Rollback nicht bereinigt werden. |
