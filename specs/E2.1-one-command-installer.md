# E2.1: One-Command-Installer — `curl -fsSL …/install.sh | bash`

**Status:** Draft
**Abhängigkeiten:** keine harten. E6.1 (CI) ist spezifiziert und baut Cross-Binaries, aber es gibt **keine Releases/Tags** — der Installer arbeitet deshalb quellbasiert (git clone + lokaler Build via setup.sh). Umstellung auf Release-Binaries ist E6.2.
**Blockiert:** E2.3 (Setup-Wizard im Browser), E2.4 (Self-Update-API), E2.5 / Gate L3 (E2E-Beweis auf frischem OS); außerdem die Umstellung des Test-Pi von Docker auf nativ (siehe Offene Punkte).
**Zuständig:** pi-client — Schreibrecht NUR: `install.sh` (NEU), `setup.sh`, `systemd/eink-server.service`, `systemd/eink-client.service`, `scripts/test-setup.sh` (NEU), `client/requirements.txt` (nur Versions-Constraints), `INSTALL.md`, `README.md` (nur Installations-Abschnitt). Kein Go-/Python-Produktivcode, keine Docker-Dateien, kein `eink.sh`.

Dieser Task umfasst die Epic-Items **E2.1 + E2.2** (Bootstrap-Installer + Härtung des nativen Setups).

## Ziel

Eine frische Raspberry-Pi-OS-Installation wird mit genau einem Befehl zu einem laufenden E-Ink-Picture-System:

```sh
curl -fsSL https://raw.githubusercontent.com/Kilian-Schwarz/E-INK-Picture/main/install.sh | bash
```

Ohne Terminal-Interaktion, ohne stille Degradation: Der Installer klont das Repo, baut den Server, richtet venv + gepinnten Waveshare-Treiber ein, aktiviert SPI, installiert systemd-Services mit `enable --now` — und **schlägt laut fehl**, statt unbemerkt im Preview-only-Modus zu landen. Zweiter Lauf = Update, nicht Schaden.

## Kontext

### Betroffene Dateien

| `setup.sh` | Härtung: Headless-Modus, Waveshare-Pin, Fehler-Sichtbarkeit, SPI nonint, Idempotenz/`--update`, systemd-Template-Rendering, `--dry-run` |

### Verifizierte Fakten

1. **setup.sh ist user-zentriert, nicht system-zentriert:** Alles hängt an `SCRIPT_DIR` (Z. 6–7): venv (Z. 153), Server-Binary (Z. 142), `.env` (Z. 196–202), logrotate-Pfade (Z. 206–217), Daten/Logs (Z. 189–193); systemd-Units laufen als `User=$USER` (Z. 238), Go-PATH landet in `~/.profile` (Z. 119–121). `sudo` wird nur punktuell genutzt (apt, tar nach /usr/local, tee nach /etc, systemctl). → **Zielverzeichnis-Entscheidung: `$HOME/E-INK-Picture`** (überschreibbar via `EINK_INSTALL_DIR`), nicht `/opt`: /opt würde chown-Gymnastik oder Root-Build erfordern und den Annahmen von setup.sh, eink.sh und INSTALL.md (Quick Start Z. 5–10: Nutzer-Checkout) widersprechen.
2. **Interaktive Blocker:** `read -p` bei SPI (setup.sh Z. 72) und systemd (Z. 224). Bei `curl … | bash` ist stdin die Pipe — `read` konsumiert Skripttext oder schlägt fehl. Headless ist daher nicht optional, sondern Voraussetzung für den Einzeiler.
3. **Waveshare ungepinnt + Fehler verschluckt:** Z. 181–183: `pip install "git+https://github.com/waveshare/e-Paper.git#subdirectory=…" 2>/dev/null || warn` — (a) installiert git-HEAD (nicht reproduzierbar), (b) pip macht dabei einen **vollen Clone des Riesen-Repos** (Arduino/STM32/Bilder) ins Tmp — RAM/SD-kritisch auf Pi Zero (512 MB), (c) jeder Fehler wird zu einer WARN-Zeile → Client läuft still im Preview-only-Modus (client.py Z. 44–46 fängt den `ImportError`). `github.com/waveshare/e-Paper` redirectet auf `waveshareteam/e-Paper` (kanonisches Repo).
4. **Recherchierter initialer Pin:** master-HEAD von `waveshareteam/e-Paper` am 2026-07-14: `500fa7c6f57b786102cccb866682f8cc43e08996` („Update 3.5-inch e-Paper G and 7.5-inch e-Paper H“). Die benötigten Treiber (`epd7in3e`, `epd7in5_V2` unter `RaspberryPi_JetsonNano/python`) sind darin enthalten. Der Pin ist ein **Startwert** — funktionale Bestätigung auf Hardware ist Teil von Gate L3 (Offene Punkte); Bumps sind bewusste Einzel-Commits.
5. **Weitere verschluckte Fehler:** GPIO-Lib-Zoo (Z. 171–177: `RPi.GPIO spidev gpiod gpiozero lgpio` als ein Block mit `2>/dev/null || warn`) und der apt-Fallback (Z. 59–63), der mit `|| true` endet — ein Totalausfall der Systempakete bliebe heute unbemerkt und würde erst Minuten später als kryptischer Folgefehler auftauchen.
6. **systemd-Duplikat + schwaches Restart:** setup.sh erzeugt Units per Heredoc (Z. 230–273), obwohl `systemd/`-Templates mit `%USER%`/`%INSTALL_DIR%`-Platzhaltern existieren und inhaltlich fast identisch sind — klassische Drift-Falle. Beide Quellen: `Restart=on-failure`; `systemctl enable` ohne `--now` (Z. 276), der Erststart bleibt dem Nutzer überlassen.
7. **Keine Releases:** CI (E6.1) lädt Build-Artefakte nur mit 7-Tage-Retention hoch; es gibt keine Tags/Releases. Der Einzeiler muss daher aus dem Raw-Content von `main` funktionieren (`raw.githubusercontent.com/Kilian-Schwarz/E-INK-Picture/main/install.sh`); Binary-Distribution ist E6.2.
8. **Docker-Kollisionsrealität:** Der Test-Pi läuft im Docker-Modus (PROGRESS.md Z. 98). `docker-compose.yml` belegt Port 5000 (Z. 7) und die SPI-/GPIO-Devices exklusiv-konfliktträchtig (Z. 50–57, `privileged`). Nativ + Docker parallel kollidiert doppelt (Port-Bind schlägt fehl, zwei Prozesse am selben SPI-Bus). Der Installer muss das erkennen und abbrechen, nicht „mal sehen“.
9. **Python-Versionen ohne Obergrenze:** `client/requirements.txt` (`requests>=2.31.0`, `Pillow>=10.0.0`) und setup.sh Z. 165–168 haben keine oberen Schranken — ein Pillow-Major-Bump (13.x) auf frischen Installationen wäre unbemerkt live. E6.1-CI installiert dieselben Untergrenzen; Constraints-Änderungen hier müssen die Untergrenzen unverändert lassen (CI bleibt kompatibel, Drift-Risiko siehe Risiken).
10. **Treiber-Bereitstellung:** client.py importiert `waveshare_epd` als installiertes Paket (Z. 35–46). `pip install <clone>/RaspberryPi_JetsonNano/python` installiert exakt dasselbe Paket wie heute `pip install git+…#subdirectory=…` (gleiches setup.py) — der Wechsel ist für den Client unsichtbar.
11. **Fetch-by-SHA ist auf GitHub möglich:** `git fetch --depth 1 origin <full-sha>` funktioniert gegen GitHub (uploadpack.allowAnySHA1InWant); mit `--filter=blob:none` + sparse-checkout lässt sich der Download auf das Python-Subdir reduzieren. Debian-12/13-git (≥ 2.39) beherrscht Partial Clone; Fallback ohne `--filter` für ältere git-Versionen.
12. **Mac-Verifikationsrealität:** Auf dem Mac liegt `/bin/bash` 3.2 — gesourcte setup.sh-Funktionen und `scripts/test-setup.sh` dürfen keine bash-4-Features nutzen (keine assoziativen Arrays, kein `${var,,}`), sonst ist Gate L1 nicht ausführbar.

### Bindende Richtungsentscheidungen (Manager)

- `install.sh` ist ein **dünner Bootstrap** — Klonen + Delegation an `setup.sh --headless`; keine zweite Implementierung von Arch-Erkennung & Co. (nur das Minimum, um bis zum Checkout zu kommen).
- Zielverzeichnis `$HOME/E-INK-Picture` (Fakt 1), Override via `EINK_INSTALL_DIR`.
- Waveshare-Treiber wird **gepinnt** (initial Fakt 4) und per schlankem git-Fetch bezogen — nie wieder `pip install git+…` von HEAD.
- Treiber-Fehler auf Pi-Hardware sind **fatal** (Exit ≠ 0), außer `--allow-preview-only` ist gesetzt. Auf Nicht-Pi-Linux ist Preview-only automatisch und wird deutlich benannt (kein Fehler).
- `--dry-run` ist ein Feature, kein Debug-Hack: druckt alle mutierenden Aktionen ohne Ausführung — das macht den Installer auf dem Mac testbar.
- Headless-Defaults: SPI aktivieren = ja, systemd installieren + `enable --now` = ja.

## Akzeptanzkriterien

1. **AC1 — `install.sh` Bootstrap (curl|bash-tauglich):** Neue Datei im Repo-Root. `set -euo pipefail`; gesamte Logik in Funktionen, letzte Zeile `main "$@"` (Schutz vor abgebrochenem Download); **kein** `read`/TTY-Zugriff. Ablauf: (a) Gate OS = Linux und Arch ∈ {aarch64, armv7l, armv6l, x86_64}, sonst Exit ≠ 0 mit klarer Meldung; (b) früher `sudo -n true`-Check mit verständlicher Meldung, falls sudo ein Passwort bräuchte; (c) git via apt installieren falls fehlend; (d) Repo nach `${EINK_INSTALL_DIR:-$HOME/E-INK-Picture}` klonen (`--depth 1 --branch main`) bzw. bei existierendem `.git` in den Update-Pfad wechseln; (e) `bash setup.sh --headless` mit durchgereichten Flags (`--update`, `--allow-preview-only`, `--dry-run`) aufrufen. Alle mutierenden Schritte laufen über einen `run()`-Wrapper, der unter `--dry-run` nur druckt.
   *Beweis:* `bash -n install.sh`; shellcheck clean; Pipe-Modus-Probe `cat install.sh | bash -s -- --dry-run` (mit Test-Overrides, s. Gate L1) druckt den Plan, mutiert nichts, Exit 0; Negativprobe unsupported Arch → Exit ≠ 0.

2. **AC2 — Headless-Modus in setup.sh:** Flag `--headless` (Alias `--yes`); zusätzlich schaltet setup.sh automatisch auf headless, wenn stdin kein TTY ist (`[ ! -t 0 ]` — Fakt 2). Im Headless-Pfad existiert **kein** `read -p` mehr; die beiden heutigen Prompts (SPI Z. 72, systemd Z. 224) werden durch die dokumentierten Defaults ersetzt (SPI = ja, systemd = ja). Der interaktive Modus bleibt für direkte Terminal-Aufrufe erhalten und nutzt dieselben Funktionen.
   *Beweis:* `grep -n 'read -p' setup.sh` zeigt nur noch Aufrufe hinter einem expliziten Interaktiv-Guard; Dry-Run headless (Gate L1) enthält „enable SPI“ und „install systemd services“ im Plan; `test_flag_parsing` in `scripts/test-setup.sh` grün.

3. **AC3 — Pinning (Waveshare + Python):**
   - setup.sh definiert am Kopf `WAVESHARE_EPD_REPO="https://github.com/waveshareteam/e-Paper.git"` und `WAVESHARE_EPD_PIN="500fa7c6f57b786102cccb866682f8cc43e08996"` (Fakt 4).
   - Bezug SD-/RAM-schonend statt pip-git (Fakt 3/11): temporäres Verzeichnis (`mktemp -d`), `git init` + `git fetch --depth 1 --filter=blob:none origin $WAVESHARE_EPD_PIN` (Fallback ohne `--filter` bei altem git), sparse-checkout auf `RaspberryPi_JetsonNano/python`, `checkout FETCH_HEAD`, dann `pip install --no-cache-dir <tmp>/RaspberryPi_JetsonNano/python`, Tmp löschen.
   - Idempotenz-Marker: installierter Pin wird in `$INSTALL_DIR/venv/.waveshare_pin` abgelegt; stimmt Marker und gelingt der Import-Check (AC4), wird der Fetch übersprungen.
   - In setup.sh existiert kein `pip install "git+` mehr.
   - `client/requirements.txt`: `requests>=2.31.0,<3` und `Pillow>=10.0.0,<13` (Untergrenzen unverändert, Fakt 9); setup.sh verwendet **exakt dieselben Constraint-Strings**.
   *Beweis:* `grep -c 'git+https' setup.sh` = 0; `grep WAVESHARE_EPD_PIN setup.sh` zeigt den Hash; Dry-Run-Plan enthält den Hash; Constraint-Gleichheit per grep in requirements.txt und setup.sh.

4. **AC4 — Kein stiller Preview-only-Modus mehr:**
   - Pi-Erkennung als Funktion (`is_raspberry_pi`), Quelle `/proc/device-tree/model` (Match „Raspberry Pi“), testbar via injizierbarem Model-String.
   - Auf Pi: Schlägt die Treiber-Kette fehl — GPIO-Basispakete, Waveshare-Install **oder** der finale Import-Check `"$INSTALL_DIR/venv/bin/python3" -c "from waveshare_epd import ${EINK_DISPLAY_DRIVER:-epd7in3e}"` (nur Import, keine Hardware-Initialisierung) — bricht setup.sh mit Exit ≠ 0 und einer Fehlermeldung ab, die das fehlgeschlagene Glied benennt. Ausnahme: Flag `--allow-preview-only` → deutliche WARN-Zeile + Weiterlauf.
   - Auf Nicht-Pi-Linux: automatisch Preview-only mit expliziter Notiz in Ausgabe und Summary (kein Fehler).
   - GPIO-Zoo (Z. 171–177): Pakete werden einzeln installiert, Fehler je Paket **sichtbar** (warn mit Paketname statt `2>/dev/null` über den Block); entscheidend für Fail/Pass ist der Import-Check.
   - apt-Härtung (Fakt 5): eine definierte Liste essentieller Pakete (mindestens `python3 python3-pip python3-venv git curl wget libopenjp2-7-dev libjpeg-dev zlib1g-dev`) muss erfolgreich installiert werden, sonst Exit ≠ 0 — das heutige `|| true` entfällt; optionale Pakete (Fonts, libgpiod, logrotate) dürfen mit sichtbarem warn fehlen.
   *Beweis:* `test_driver_fail_policy` und `test_pi_detection` in test-setup.sh (Entscheidungsfunktion mit injiziertem Pi-Status/Fehlerfall: Pi+Fehler → fail, Pi+Fehler+allow-flag → pass mit warn, Nicht-Pi+Fehler → pass); `grep -n '|| true' setup.sh` trifft keinen apt-Pfad essentieller Pakete mehr; Gate-L5-Review.

5. **AC5 — SPI headless aktivieren:** Wenn kein `/dev/spidev*` existiert: mit `raspi-config` → `sudo raspi-config nonint do_spi 0`; ohne `raspi-config` (Debian-13-/Nicht-RPi-OS-Fall) → idempotenter Eintrag `dtparam=spi=on` in `/boot/firmware/config.txt` (falls vorhanden, Bookworm+) sonst `/boot/config.txt`; existiert der Eintrag (auch auskommentiert-aktiv), keine Doppelung. In beiden Fällen: deutlicher Reboot-Hinweis in der Summary (SPI wird erst nach Reboot aktiv; der Client startet dank `Restart=always` danach von selbst). Kein Prompt im Headless-Modus.
   *Beweis:* `test_spi_config_fallback` in test-setup.sh (Funktion gegen Temp-Datei: fügt genau einmal hinzu, zweiter Lauf ändert nichts, wählt firmware-Pfad wenn vorhanden); Dry-Run-Plan zeigt den gewählten Mechanismus.

6. **AC6 — Idempotenz + `--update`:** Zwei komplette Läufe hintereinander = kein Schaden: venv wird wiederverwendet, `.env` nie überschrieben (Bestand Z. 196–202 bleibt), Waveshare-Skip via Marker (AC3), logrotate/Units werden deterministisch neu gerendert (Überschreiben identischen Inhalts ist ok). `--update` (von install.sh automatisch gesetzt, wenn das Zielverzeichnis schon ein Checkout ist): `git pull --ff-only` (bei Divergenz/lokalen Änderungen: Abbruch mit klarer Meldung, kein reset --hard), Server-Rebuild, pip-Sync (nur bei geändertem Pin/Constraints), Units neu rendern + `systemctl daemon-reload` + Service-Restart.
   *Beweis:* Dry-Run `--update` druckt exakt diese Schrittfolge; `test_flag_parsing` deckt `--update` ab; Doppellauf-Beweis auf Hardware ist Gate L3 (Offene Punkte), die Guard-Logik (Marker, .env-Existenz) ist in test-setup.sh unit-getestet.

7. **AC7 — systemd: Single Source + Restart-Härtung:** setup.sh rendert die Templates aus `systemd/` (sed über `%USER%`/`%INSTALL_DIR%`) nach `/etc/systemd/system/` — die Heredoc-Duplikate (Z. 230–273) entfallen ersatzlos (Fakt 6). Die Templates erhalten `Restart=always` und `StartLimitIntervalSec=0` (nie aufgeben — Backoff bleibt über `RestartSec=5`/`10` erhalten); Installation endet mit `sudo systemctl enable --now eink-server.service eink-client.service`.
   *Beweis:* `grep -c 'tee /etc/systemd' setup.sh` zeigt nur noch das Template-Rendering (kein Unit-Heredoc: `grep -c '\[Unit\]' setup.sh` = 0); `grep -n 'Restart=always' systemd/*.service` trifft beide; `test_render_unit_template` in test-setup.sh beweist, dass nach dem Rendern kein `%`-Platzhalter übrig ist; Dry-Run-Plan enthält `enable --now`.

8. **AC8 — Docker-Konflikt-Preflight:** Vor jeder Mutation prüft setup.sh (Preflight, auch von install.sh-Pfad durchlaufen): (a) läuft ein Container, dessen Name oder Image auf das Projekt matcht (Muster `eink|e-ink-picture`), → Abbruch mit Handlungsanweisung („`docker compose down` im alten Checkout; `data/` sichern/migrieren“); (b) ist Port `${PORT:-5000}` bereits belegt (und gehört nicht zu einem via `--update` neu zu startenden eigenen Service), → Abbruch mit Nennung des Ports. Fehlt Docker komplett, ist das kein Fehler (Fakt 8).
   *Beweis:* `test_docker_preflight` in test-setup.sh — Preflight-Funktion mit injizierter `docker ps`-Ausgabe (Match → fail, kein Match → pass, „docker fehlt“ → pass); Negativprobe im Gate L1.

9. **AC9 — Dry-Run + Mac-Testbarkeit:**
   - `--dry-run` (install.sh und setup.sh): alle mutierenden Kommandos (apt, wget/tar, pip, git-fetch, mkdir außerhalb Tmp, tee, systemctl, raspi-config) laufen über `run()` und werden als `[DRY-RUN] <cmd>` gedruckt statt ausgeführt; Exit 0 wenn der Plan vollständig ist.
   - Detektions-Overrides `EINK_TEST_OS` / `EINK_TEST_ARCH` / `EINK_TEST_PI_MODEL` werden **nur** im Dry-Run-/Source-Modus honoriert — damit ist der volle Plan auf dem Mac erzeugbar (z. B. `EINK_TEST_OS=Linux EINK_TEST_ARCH=aarch64`).
   - `scripts/test-setup.sh` (NEU): sourced setup.sh über einen Source-Guard (main läuft beim Sourcen nicht, z. B. `[[ "${BASH_SOURCE[0]}" == "$0" ]] && main "$@"`), enthält mindestens: `test_arch_mapping` (alle 4 Architekturen + unsupported → Fehler), `test_pi_detection`, `test_flag_parsing` (`--headless`/`--yes`/`--update`/`--allow-preview-only`/`--dry-run`), `test_docker_preflight`, `test_spi_config_fallback`, `test_driver_fail_policy`, `test_render_unit_template`. Ausgabe „N/N tests passed“, Exit ≠ 0 bei Failure.
   - Alle drei Skripte sind bash-3.2-kompatibel (Fakt 12) und shellcheck-clean (gezielte `# shellcheck disable=`-Ausnahmen nur mit Begründungskommentar).
   *Beweis:* Gate-L1-Kommandos unten, alle Exit 0; Negativprobe: absichtlich kaputte Assertion → test-setup.sh Exit ≠ 0.

10. **AC10 — Doku:** INSTALL.md beginnt mit dem Einzeiler (`curl -fsSL https://raw.githubusercontent.com/Kilian-Schwarz/E-INK-Picture/main/install.sh | bash`), dokumentiert die Flags (`--headless`/`--yes`, `--allow-preview-only`, `--update` inkl. `curl … | bash -s -- --update`, `--dry-run`) und `EINK_INSTALL_DIR`; der bisherige manuelle Weg (git clone + ./setup.sh) bleibt als Alternative dokumentiert. README.md-Installations-Abschnitt nennt den Einzeiler und verweist auf INSTALL.md.
    *Beweis:* `grep -c 'raw.githubusercontent.com/Kilian-Schwarz/E-INK-Picture/main/install.sh' INSTALL.md README.md` ≥ 1 je Datei; alle Flags in INSTALL.md greppbar.

## Non-Goals

- **Kein Setup-Wizard im Browser** (E2.3) — der Installer endet mit URL-Hinweis auf `/designer`, mehr nicht.
- **Keine Self-Update-API** (E2.4) — Update nur via erneutem Einzeiler bzw. `setup.sh --update`.
- **Keine Release-Assets/Binary-Downloads** (E6.2) — es wird immer lokal gebaut; wenn E6.2 existiert, kann install.sh später auf Binaries umgestellt werden (bewusst als Folge-Task).
- **Keine Änderungen am Docker-Pfad** — `docker-compose*.yml`, `scripts/setup-local.sh`, `scripts/setup-cloud-client.sh` bleiben unangetastet (auch deren ungepinnter Waveshare-Install: Legacy, separater Task).
- **Keine Änderungen an `eink.sh`** — bleibt als manuelle Start/Stop-Alternative; systemd ist der neue Default-Betriebsmodus.
- **Keine Migration von Docker-Daten** — der Preflight bricht bei Docker-Kollision nur ab und benennt die Schritte; die eigentliche Migration des Test-Pi ist ein manuell begleiteter Vorgang (Offene Punkte).
- **Keine Uninstall-Routine** — dokumentierter manueller Rückbau genügt (Risiken/Rollback).

## Verifikation

### Gate L1 — Nicht-interaktive Verifikation (Mac)

```sh
cd "/Users/ksch/Documents/005 - Hobby/Github/E-INK-Picture"

# Syntax
bash -n install.sh setup.sh scripts/test-setup.sh

# shellcheck (brew install shellcheck erlaubt, falls nicht vorhanden)
shellcheck install.sh setup.sh scripts/test-setup.sh

# Unit-artige Funktionstests (bash 3.2-tauglich)
bash scripts/test-setup.sh

# Dry-Run: voller Plan wie auf einem Pi Zero 2 W (aarch64), frische Installation
EINK_TEST_OS=Linux EINK_TEST_ARCH=aarch64 EINK_TEST_PI_MODEL="Raspberry Pi Zero 2 W" \
  bash setup.sh --headless --dry-run

# Dry-Run: Update-Pfad
EINK_TEST_OS=Linux EINK_TEST_ARCH=aarch64 EINK_TEST_PI_MODEL="Raspberry Pi Zero 2 W" \
  bash setup.sh --headless --update --dry-run

# Pipe-Modus (curl|bash-Simulation)
cat install.sh | EINK_TEST_OS=Linux EINK_TEST_ARCH=aarch64 bash -s -- --dry-run
```

Erwartung: alles Exit 0; die Dry-Run-Pläne enthalten den Waveshare-Pin-Hash, „enable SPI“, „systemctl enable --now“ und (im Update-Fall) pull/rebuild/restart; keinerlei Mutation auf dem Mac (kein sudo-Aufruf ausgeführt).

**Negativproben (nicht committen):**
- `EINK_TEST_ARCH=mips bash setup.sh --headless --dry-run` → Exit ≠ 0, Meldung nennt die Architektur.
- Injizierte `docker ps`-Ausgabe mit `eink-picture-server` → Preflight-Test schlägt Abbruch vor (test_docker_preflight rot, wenn man den Abbruch entfernt).
- `--allow-preview-only` temporär aus der Fail-Entscheidung entfernen → `test_driver_fail_policy` muss rot werden (beweist, dass der Test die Policy pinnt).

### Gate L3 — End-to-End auf dem Pi (OFFENER PUNKT, nicht Teil dieses Tasks)

Der echte Beweis „frisches Raspberry Pi OS → Einzeiler → Panel zeigt Design“ ist Gate L3 / E2.5 und **blockiert auf Kilians Umstellung des Test-Pi**: Der Pi läuft im Docker-Modus (PROGRESS.md Z. 98); vorher `data/` sichern, `docker compose down`, Umstellung abstimmen. Dann:
1. `curl -fsSL https://raw.githubusercontent.com/Kilian-Schwarz/E-INK-Picture/main/install.sh | bash` (erst nach Merge auf main möglich — vorher Raw-URL des Feature-Branches verwenden),
2. Erwartung: Exit 0, beide Services `active (running)`, `/health` OK, Panel-Refresh via `POST /api/trigger_refresh` sichtbar,
3. zweiter Lauf des Einzeilers → Update-Pfad, Services laufen danach weiter,
4. Waveshare-Pin damit funktional bestätigt (Fakt 4) — bei Problemen: Pin-Bump als Einzel-Commit.

### Gate L5 — Review

Review gegen diese Spec, insbesondere:
- install.sh dupliziert keine setup.sh-Logik; `main "$@"` als letzte Zeile; kein read/TTY (AC1),
- kein `pip install git+…` mehr; Pin-Variable + Marker-Skip; Tmp-Clone wird gelöscht (AC3),
- Fail-Policy: Pi + Treiberfehler = Exit ≠ 0, `--allow-preview-only` als einzige Ausnahme; kein `2>/dev/null || warn` mehr um Treiber-/Kern-Installationen (AC4),
- systemd nur noch aus `systemd/`-Templates; `Restart=always` + `StartLimitIntervalSec=0`; `enable --now` (AC7),
- Docker-Preflight vor der ersten Mutation (AC8),
- Overrides `EINK_TEST_*` außerhalb Dry-Run/Source-Modus wirkungslos (AC9),
- bash-3.2-Kompatibilität, shellcheck-Disables nur mit Begründung (AC9).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Bestehende Docker-Installation auf demselben Pi (Port 5000 + SPI-Devices doppelt belegt) | Preflight AC8 bricht hart ab und benennt die Schritte; keine automatische Migration (Non-Goal). Test-Pi-Umstellung ist explizit Kilian-Aufgabe (Gate L3). |
| Debian 13 / Nicht-RPi-OS ohne `raspi-config` | SPI-Fallback via config.txt-Edit (AC5) mit Pfad-Erkennung `/boot/firmware` vs. `/boot`; Reboot-Hinweis + `Restart=always` überbrücken den Erststart. |
| Waveshare-Pin veraltet oder Upstream bricht API | Pin ist bewusster Trade-off: reproduzierbar > frisch. Bump = Einzel-Commit mit L3-Nachweis. Initialer Pin (Fakt 4) ist erst nach Gate L3 hardware-bestätigt. |
| `--filter=blob:none`/Fetch-by-SHA scheitert (altes git, Proxy) | Fallback-Kette in AC3: erst partial, dann plain `--depth 1`-Fetch; schlimmstenfalls größerer Download, nie HEAD statt Pin. |
| sudo verlangt Passwort im Pipe-Modus | Früher `sudo -n`-Check (AC1) mit klarer Meldung („einmal `sudo -v` ausführen oder Skript herunterladen und direkt starten“); auf Standard-RPi-OS ist sudo passwortlos. |
| Obere Pillow-/requests-Schranken vs. piwheels-Wheel-Verfügbarkeit auf armv6 | Bounded Ranges statt Exact-Pins (AC3) lassen piwheels Spielraum; kein Testgerät für armv6 — bekannter blinder Fleck (wie E6.1). |
| Drift zu E6.1-CI-Constraints (Workflow hartkodiert nur Untergrenzen) | Untergrenzen bleiben identisch (AC3); CI testet damit weiterhin gültige Versionen. Obergrenzen-Sync in CI ist ein Ein-Zeilen-Folge-Commit, kein Blocker. |
| Raw-URL des Einzeilers funktioniert erst nach Merge auf `main` | Für Branch-Tests: Raw-URL des Feature-Branches; Doku nennt nur die main-URL. |
| `curl \| bash`-Sicherheitsbedenken | Standard-Trade-off für diese Projektklasse; Doku behält den manuellen Weg (git clone + Review + ./setup.sh) als gleichwertige Alternative (AC10). |
| **Rollback** | install.sh ist additiv; setup.sh/-Units per `git revert` rückrollbar. Laufzeit-Rückbau: `sudo systemctl disable --now eink-server eink-client`, Unit-Dateien + `/etc/logrotate.d/eink-picture` löschen, `$HOME/E-INK-Picture` entfernen — keine Änderungen außerhalb dieser Pfade (plus SPI-Flag, das harmlos aktiv bleiben darf). |

## Offene Punkte

1. **Test-Pi-Umstellung Docker → nativ** (Voraussetzung Gate L3/E2.5): `data/` sichern, `docker compose down`, mit Kilian abstimmen — erst danach ist der End-to-End-Beweis des Einzeilers möglich (PROGRESS.md Z. 98).
2. **Hardware-Bestätigung des Waveshare-Pins** `500fa7c6f57b786102cccb866682f8cc43e08996` — Teil von Gate L3; bis dahin gilt der Pin als plausibel (enthält beide Treiber), aber unbewiesen.
3. **Kein armv6-Testgerät** — Installer-Pfad für Pi Zero (v1) ist nur per Dry-Run/CI-Cross-Build abgesichert (gleicher blinder Fleck wie E6.1).
