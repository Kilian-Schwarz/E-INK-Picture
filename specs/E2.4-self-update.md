# E2.4: Self-Update mit Rollback — /api/update, Update-Skript, Health-Check-Beweis

**Status:** Draft
**Abhängigkeiten:** E6.2 (HART: `main.version` + Version-Feld in `/health` + mindestens ein existierendes GitHub-Release als Update-Ziel), E5.1 (Auth/Guard, gemergt), E2.1 (setup.sh-Infrastruktur inkl. Source-Guard und test-setup.sh-Muster, gemergt)
**Blockiert:** E2.5 / Gate L3 (vollständiger Lebenszyklus-Beweis: Install → Update → Rollback auf Hardware)
**Zuständig:** go-backend (Server-Endpoints + Go-Tests: `server/**`), pi-client (`scripts/update.sh` NEU, `scripts/test-update.sh` NEU, `setup.sh`, `systemd/eink-updater.service` NEU, `scripts/test-setup.sh`), frontend **klein** (Settings-Modal-Abschnitt in `server/templates/designer.html` + `server/static/js/designer.js`), docs (INSTALL.md, CHANGELOG.md)

## Ziel

Aus dem Designer heraus lässt sich das Gerät auf ein neueres Release-Tag aktualisieren — konservativ: der Server ersetzt **nie selbst** Binaries, sondern stößt ein detachtes Update-Skript an, das Backup, Checkout, Build, Service-Restart und einen Health-Check-Beweis (`/health`-Version == Ziel-Tag) ausführt und bei jedem Fehlschlag automatisch auf den alten Stand zurückrollt. Das Gerät hat keine Tastatur — **der Rollback-Pfad ist der eigentliche Task**, das Update nur der Anlass.

### Verifizierte Fakten (Bestandsaufnahme)

1. **Update heute, nativer Pfad:** `setup.sh --update` = `git pull --ff-only` (Z. 273, bricht bei Divergenz sauber ab) + Rebuild (Z. 424–440) + `systemctl restart` beider Units (Z. 732–734) — synchron, interaktiv angestoßen, kein Rollback. **Docker-Pfad:** `docker compose up --build` — vollständig außerhalb der Server-Kontrolle.
2. **`git pull --ff-only` bricht auf Detached-HEAD:** Ein Tag-Checkout (E6.2-Install oder dieses Update) ist detached — der Update-Fluss muss `git fetch --tags` + `git checkout --detach <tag>` sein, nie pull. Rollback = Checkout der zuvor aufgezeichneten Ref (Symbolic-Ref, falls auf Branch, sonst SHA).
3. **Auth-Realität (E5.1):** Der Guard ist deny-by-default und läuft vor dem Router — neue `/api/update/*`-Routen sind automatisch session-pflichtig (nicht in `publicRoutes`/`clientRoutes`, middleware/auth.go Z. 36–56), und cookie-authentifizierte mutierende Requests bekommen automatisch den Origin-/CSRF-Check. Der Client-Token öffnet exakt vier andere Endpoints — er darf **kein** Update auslösen.
4. **cgroup-Falle (zentral):** Beide Units laufen mit systemd-Default `KillMode=control-group`. Ein per `nohup`/`&` vom Server gestartetes Update-Skript lebt im cgroup von `eink-server.service` — beim `systemctl restart eink-server` (den das Skript selbst auslöst!) würde systemd den Updater mitkillen: Update stirbt mitten im Lauf, Gerät potenziell halb aktualisiert. → Der Updater MUSS in einer eigenen Unit laufen (`eink-updater.service`); `nohup` und vom Server gestartetes `systemd-run` scheiden aus.
5. **Rechte-Realität:** Beide Units laufen als `User=%USER%` (unprivilegiert); Binary (`server/eink-server`), Checkout und venv sind user-owned. `systemctl start/restart` von System-Units erfordert root → sudo. Auf Standard-RPi-OS hat der Erstnutzer NOPASSWD-sudo, garantiert ist das nicht → expliziter sudoers-Baustein (Sicherheitsbewertung unten).
6. **Version-Beweis existiert (E6.2):** `/health` liefert `{"status":"ok","version":…}` unauthentifiziert (Guard-Allowlist) — das Update-Skript kann ohne Session beweisen, dass die **Ziel**-Version antwortet (`version == tag`), nicht nur, dass irgendein Server 200 liefert. Voraussetzung: auch der On-Pi-Build stempelt (AC8), sonst meldet ein Quell-Build `dev` und der Beweis degradiert.
7. **Docker-Erkennung:** Docker legt `/.dockerenv` in jedem Container an (auch bei `FROM scratch`); `DEPLOYMENT_MODE` taugt NICHT (auch nativ „local“). Compose-Healthcheck (`cmd/healthcheck`) prüft nur Status 200 — Antwortänderungen sind sicher.
8. **Client läuft aus dem Checkout:** `ExecStart=%INSTALL_DIR%/venv/bin/python3 %INSTALL_DIR%/client/client.py` — der Tag-Checkout aktualisiert den Client-Code mit; wirksam erst nach Client-Restart. Der laufende Go-Server ist davon unberührt (go:embed).
9. **Panel-Timing:** Panel-Write ~19,5 s, Zyklus ~22,7 s (PROGRESS-Baseline) — ein Client-Restart kann mitten in einen Panel-Write fallen (Risiken).
10. **GitHub-API:** `GET /repos/Kilian-Schwarz/E-INK-Picture/releases/latest` unauthentifiziert = 60 Requests/h/IP → Cache + Offline-Toleranz sind Pflicht, kein Token verfügbar/nötig.
11. **Plattenplatz:** `go build` auf dem Pi braucht Build-Cache-Headroom, `git fetch` zusätzliche Objekte; ein voller Datenträger mitten im Update ist ein Bricking-Vektor → Preflight vor jeder Mutation.
12. **Test-Pi läuft im Docker-Modus** (PROGRESS-Baseline) — der Hardware-Beweis (Gate L3) ist bis zur nativen Umstellung blockiert (wie E2.1).

### Sicherheitsbewertung sudo/systemctl (bindend)

- **Verworfen:** (a) NOPASSWD auf `systemd-run` — kann beliebige Kommandos als root starten, äquivalent zu Voll-root. (b) `eink-updater.service` als root, ExecStart auf ein user-owned Skript — Privilege-Escalation user→root durch Editieren des Skripts. (c) Server ruft selbst `sudo systemctl restart` im Prozess — stirbt am eigenen Restart und kombiniert Webserver-Prozess mit sudo-Rechten.
- **Gewählt:** `/etc/sudoers.d/eink-picture` (0440, vor Installation `visudo -c`-validiert), Inhalt exakt zwei Kommandozeilen ohne Wildcards für den Service-User: `systemctl start eink-updater.service` und `systemctl restart eink-server.service eink-client.service` (absolute systemctl-Pfade).
- **Bewertung:** Faktisch keine Privilegien-Ausweitung. Beide Ziel-Units laufen als derselbe User und ihre ExecStart-Artefakte (Binary, client.py, update.sh) sind user-owned — wer User-Zugriff hat, kontrolliert heute schon, was diese Services ausführen (Binary tauschen + Prozess beenden genügt, `Restart=always` startet es). Die Regel „legalisiert“ nur das saubere Neustarten der eigenen Services. Kein root-Code-Pfad entsteht: `eink-updater.service` läuft ebenfalls als `User=%USER%`. Restrisiko: Restart-DoS durch den User — den DoS kann der User heute schon (kill). Grenze bewusst gezogen: Änderungen an Unit-Dateien oder sudoers selbst bräuchten root und sind vom Self-Update **ausgeschlossen** (Non-Goal) — dafür bleibt `setup.sh --update` per SSH der Weg.

### Bindende Richtungsentscheidungen

- Ablauf Apply: Server validiert → schreibt `data/update-request.json` (`{"tag":"vX.Y.Z"}`) → `sudo -n systemctl start eink-updater.service` → antwortet sofort **202**. Alles Weitere (inkl. Fehlschlag) läuft im Updater und landet in `data/update-status.json` — der Server verspricht nichts, was er nicht kontrolliert.
- Der Updater arbeitet fetch+checkout-basiert (Fakt 2), niemals `reset --hard`, bricht bei dirty Worktree ab (setup.sh-Philosophie).
- Health-Beweis = `version == tag`, nicht nur HTTP 200 (Fakt 6) — unterscheidet „neues Binary gesund“ von „altes Binary wieder hochgekommen“.
- Kein systemd/sudo verfügbar (z. B. eink.sh-Handbetrieb) → Apply antwortet mit klarem Fehler und Verweis auf `setup.sh --update`; es gibt keinen nohup-Fallback (Fakt 4).

## Akzeptanzkriterien

1. **AC1 — `GET /api/update/check`:** Antwort `{running_version, latest_version, update_available, checked_at, check_error?}`. Latest via GitHub-Releases-API (Repo fest, API-Base im Service injizierbar — Tests nutzen `httptest`, kein neues Env). Cache TTL 1 h, `?force=true` umgeht ihn. **Offline-tolerant:** Netz-/API-Fehler (Timeout, 403 Rate-Limit, 404 kein Release) → HTTP 200 mit `check_error` + letztem Cache-Stand, niemals 5xx wegen Netz. Vergleich: Tags `vX.Y.Z` numerisch (semver-artig, Prerelease-Suffix = älter als Release); `running_version` nicht tag-förmig (`dev`, git-describe-Hash) → `update_available: null` + Hinweis statt Ratespiel.
   *Beweis:* Go-Tests mit gemocktem GitHub-API: verfügbar / aktuell / Netzfehler / 403 / Cache-Hit (zweiter Call ohne API-Request) / force.

2. **AC2 — `POST /api/update/apply`:** Body optional `{"tag":"vX.Y.Z"}` (strikte Validierung `^v[0-9]+\.[0-9]+\.[0-9]+(-[0-9A-Za-z.]+)?$` — der Tag landet in einer Datei, die ein Shell-Skript liest); ohne Body gilt das letzte Check-Ergebnis (kein vorheriger erfolgreicher Check → 409 mit Hinweis). Ablauf: Request-Datei schreiben → injizierten Trigger ausführen (prod: `sudo -n systemctl start eink-updater.service`) → **202**. `409`, wenn laut Status-Datei bereits ein Update läuft (Stale-Regel: `running` älter als 30 min gilt als abgestürzt und wird mit Warnung überschrieben). Trigger-Fehlschlag (kein sudo/systemd) → 500 mit Meldung inkl. `setup.sh --update`-Verweis.
   *Beweis:* Go-Tests mit Fake-Trigger (aufgezeichnetes Kommando in `t.TempDir()`): genau ein Aufruf assertiert, Request-Datei-Inhalt assertiert, 202; Parallel-Apply → 409; ungültiger Tag → 400; **kein echtes systemctl in Tests**.

3. **AC3 — `GET /api/update/status`:** Serviert `data/update-status.json` (Schema: `state` ∈ `idle|running|success|failed_rolled_back|failed`, `from_version`, `to_version`, `started_at`, `finished_at`, `message`); fehlende Datei → `idle`. Die Datei ist der Marker für die UI („update failed, rolled back“) und überlebt Server-Restarts per Design (liegt in `data/`).
   *Beweis:* Go-Tests: idle ohne Datei, jede State-Variante aus Fixture-Datei.

4. **AC4 — Docker-Modus ehrlich:** Erkennung über Existenz von `/.dockerenv` (Pfad im Service injizierbar für Tests). Apply → **501** mit Klartext („Self-update is not available in Docker mode — update via `docker compose pull/build` on the host“); Check und Status bleiben funktional (reine Anzeige). UI zeigt bei 501 den Hinweis statt des Buttons.
   *Beweis:* Go-Tests mit injiziertem Fake-`/.dockerenv`; Meldungstext greppbar.

5. **AC5 — Auth-Pflicht:** Alle drei Endpoints ohne Session → 401 (deny-by-default, Fakt 3 — Test pinnt das trotzdem explizit); `X-Client-Token` öffnet sie NICHT; Apply mit Session aber Cross-Origin → 403 (CSRF-Check greift). Keine Aufnahme in `publicRoutes`/`clientRoutes`.
   *Beweis:* Guard-/Handler-Tests: 401-Matrix, Client-Token-Negativprobe, Origin-Negativprobe auf Apply.

6. **AC6 — `scripts/update.sh` + `systemd/eink-updater.service`:** Unit: Type=oneshot, `User=%USER%`, ExecStart=`%INSTALL_DIR%/scripts/update.sh`, kein Restart=. Skript (Funktionen sourcebar via Source-Guard, bash-3.2-kompatibel, shellcheck-clean, nutzt setup.sh-Funktionen per Sourcen statt Duplikation):
   1. Request-Datei lesen + Tag revalidieren (Regex wie AC2); Status → `running`,
   2. Preflight: kein `/.dockerenv`, Worktree clean (sonst Abbruch, kein `reset --hard`), freier Platz ≥ 500 MB auf dem Install-Dateisystem (`df -Pm`),
   3. Ist-Stand sichern: `git symbolic-ref -q HEAD || git rev-parse HEAD` als Rollback-Ref; `cp server/eink-server server/eink-server.bak`,
   4. `git fetch --tags origin` + Existenz des Tags prüfen → `git checkout --detach <tag>`,
   5. idempotenter Dependency-Sync (marker-guarded setup.sh-Funktionen: core-Python, Waveshare-Pin),
   6. Build nach Temp-Datei mit `-ldflags "-s -w -X main.version=<tag>"`, dann **atomares `mv`** aufs Binary,
   7. `sudo -n systemctl restart eink-server.service eink-client.service`,
   8. Health-Beweis: bis zu 30 Versuche × 2 s gegen `http://localhost:${PORT:-5000}/health`; Erfolg NUR wenn HTTP 200 **und** `version == <tag>` (Fakt 6),
   9. Erfolg → Status `success` (from/to-Version), Request-Datei löschen, `.bak` bleibt als manuelle Reserve liegen,
   10. **Rollback bei jedem Fehlschlag ab Schritt 4:** Binary aus `.bak` zurück (mv), Checkout der Rollback-Ref (Branch wird als Branch wiederhergestellt), Services erneut restarten, Best-Effort-Health-Check des Alt-Stands, Status `failed_rolled_back` mit dem fehlgeschlagenen Schritt in `message`. Schlägt selbst der Rollback fehl → Status `failed` (SSH-Fall, ehrlich benannt).
   *Beweis:* AC7-Bash-Tests; shellcheck; `bash -n`; Review gegen die Schrittfolge.

7. **AC7 — Bash-Tests `scripts/test-update.sh`** (Muster test-setup.sh: sourct update.sh, injizierte Kommandos/Antworten, „N/N tests passed“, Exit ≠ 0 bei Failure). Health-Check gegen einen **Fake-Health-Server** (lokaler `python3 http.server`-Ableger oder injizierte curl-Funktion mit Antwort-Skript). Mindest-Szenarien: (a) neue Version antwortet mit Ziel-Tag → success; (b) Health liefert 200 mit **alter** Version → Rollback (beweist die Version-Assertion); (c) Health antwortet gar nicht → Rollback; (d) Build schlägt fehl → Rollback **ohne** Restart eines kaputten Binaries (altes Binary wurde nie ersetzt); (e) Disk-Preflight schlägt fehl → Abbruch vor jeder Mutation; (f) dirty Worktree → Abbruch; (g) Status-Datei enthält nach jedem Szenario den korrekten Endzustand. **Negativprobe:** Rollback-Aufruf temporär entfernen → (b)/(c) müssen rot werden (Tests pinnen den Rollback, nicht den Happy Path).
   *Beweis:* Lauf auf dem Mac (bash 3.2) grün; Negativprobe protokolliert.

8. **AC8 — Installer-Bausteine (setup.sh):** rendert + installiert `eink-updater.service` (in `SYSTEMD_UNITS`-Mechanik integriert, aber ohne `enable --now` — oneshot wird nur via Start getriggert), schreibt `/etc/sudoers.d/eink-picture` (0440; exakt die zwei Kommandos aus der Sicherheitsbewertung; vor dem Aktivieren `visudo -c -f` gegen die Temp-Datei, bei Fehler Abbruch statt kaputtem sudoers), beides idempotent + im Dry-Run-Plan sichtbar. `build_server` stempelt ab jetzt `-X main.version=$(git describe --tags --always)` (Fakt 6 — sonst ist der Health-Beweis auf Quell-Builds wertlos).
   *Beweis:* `scripts/test-setup.sh` erweitert (Unit-Rendering ohne `%`-Reste, sudoers-Inhalt exakt, Idempotenz); Dry-Run-Plan enthält Unit, sudoers und Stempel.

9. **AC9 — Frontend minimal (klein halten):** Im Settings-Modal ein „System“-Abschnitt: laufende Version, Button „Check for updates“ (→ AC1), bei `update_available` Versionsanzeige + „Install update“ (→ AC2, mit Bestätigungsdialog inkl. Rollback-Hinweis), danach Status-Polling (AC3, alle 5 s bis Endzustand): `success` → Erfolgsmeldung mit neuer Version, `failed_rolled_back` → deutliche Warnbox („Update failed — previous version was restored“), 501 → Docker-Hinweistext. Kein Auto-Check beim Laden (Rate-Limit, Fakt 10), keine neue JS-Datei nötig.
   *Beweis:* Manuelle L2-Probe gegen Dev-Server mit gemocktem Backend-Zustand; kein JS-Fehler in der Konsole.

10. **AC10 — Doku + Scope:** INSTALL.md-Abschnitt „Self-Update“ (Ablauf, Rollback-Garantie, Docker-Ausnahme, Sicherheitsmodell sudoers inkl. der Bewertung von oben, manueller Fallback `setup.sh --update`); CHANGELOG-Eintrag. Diff berührt nur die in „Betroffene Dateien“ gelisteten Pfade; keine neuen Env-Variablen (`.env.example` unverändert — bewusst).
    *Beweis:* grep der Doku-Schlüsselwörter; `git diff --stat`.

## Non-Goals

- **Kein Auto-Update/Scheduler** — Updates sind immer eine explizite Nutzeraktion; ein Update-Hinweis-Badge o. Ä. wäre ein Folge-Task.
- **Kein Downgrade-UI** — die API akzeptiert jeden validen Tag (Notfall-Weg per curl), die UI bietet nur das neueste Release an.
- **Keine Unit-/sudoers-Migration durch Self-Update** — Änderungen an `systemd/`-Templates oder sudoers erfordern root und bleiben `setup.sh --update` per SSH vorbehalten (Sicherheitsbewertung); der Updater fasst `/etc` nie an.
- **Kein Docker-Self-Update** — 501 + Hinweis ist die ganze Docker-Story (AC4); Compose-Update bleibt Host-Aufgabe.
- **Kein Binary-Download von Release-Assets** — der Updater baut aus dem Tag-Checkout (wie setup.sh); Umstellung auf Release-Binaries ist derselbe Folge-Task wie bei install.sh (E6.2 Non-Goal).
- **Kein eink.sh-(PID-File)-Support** — systemd ist der Betriebsmodus; ohne systemd/sudo antwortet Apply mit klarem Fehler (Richtungsentscheidung).
- **Keine GPG-/Checksum-Verifikation des Checkouts** — HTTPS + Tag-Ref genügt v1 (Quelle ist das eigene Repo).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| **Gerät wird durch Update unbrauchbar (Kern-Risiko)** | Der gesamte AC6/AC7-Aufbau ist die Mitigation: Binary-Backup + Ref-Restore + Version-bewiesener Health-Check + Status-Marker. Doppelter Boden: schlägt der Build fehl, wurde das alte Binary nie ersetzt. Rest-Worst-Case (Rollback selbst scheitert, Status `failed`): SSH + `setup.sh --update` — dokumentiert, nicht wegzuspecen. |
| Update-Restart trifft laufenden Panel-Write (~19,5 s, Fakt 9) | client.py hat keinen SIGTERM-Handler — SIGTERM mitten im SPI-Write kann ein halb beschriebenes Panel hinterlassen; der nächste Refresh heilt es. Akzeptiert für v1 (Update ist eine seltene, bewusste Aktion); graceful Shutdown des Clients (Write zu Ende, dann exit) als benannter Folge-Task, nicht hier hineingezogen. |
| Plattenplatz läuft beim Build voll (Fakt 11) | Preflight ≥ 500 MB vor jeder Mutation (AC6.2); Schwellwert wird in Gate L3 gegen die Realität kalibriert. Go-Build-Cache wächst über Updates — Hinweis in INSTALL.md (`go clean -cache` als Wartungsgriff). |
| GitHub-Rate-Limit (60/h/IP) oder offline | Cache 1 h + `check_error`-Feld statt 5xx (AC1); kein Auto-Check beim UI-Laden (AC9). Update-Anwendung selbst braucht nur git-Zugriff, nicht die API. |
| Stromausfall mitten im Update | Reihenfolge minimiert das Fenster: Binary-Swap atomar (mv), Restart zuletzt. Nach Reboot startet `Restart=always` den Stand auf der Platte; Status bleibt `running` → Stale-Regel (AC2) gibt den Weg frei, `.bak` + Rollback-Ref liegen noch da. Version-Skew Server-Binary vs. Client-Checkout ist möglich, heilt mit erneutem Apply. |
| sudoers-Tippfehler sperrt sudo systemweit | `visudo -c` gegen die Datei VOR Installation, Abbruch bei Fehler (AC8); Datei in `/etc/sudoers.d/` (defekte Include-Datei wird von modernem sudo ignoriert, sperrt aber nie die Haupt-sudoers). |
| Updater aktualisiert sich selbst | Die laufende update.sh-Instanz ist immer die **alte** Version (Skript ist beim Checkout bereits in Ausführung/im Speicher der Shell — bash liest nach; deshalb: Skript-Konvention `main "$@"` am Ende + Gesamtlogik in Funktionen, damit Nachladen während des Laufs nichts Halbes ausführt). Änderungen an update.sh wirken ab dem nächsten Update. |
| Dauerzustand Detached-HEAD nach Tag-Update | Bewusst: Das Gerät steht ab dann auf Tags, nicht auf `main`; `setup.sh --update` (pull-basiert) meldet das sauber als Fehler (Fakt 1/2) und INSTALL.md erklärt beide Welten. |
| 501-Erkennung schlägt in exotischen Runtimes fehl (Podman: kein `/.dockerenv`) | Akzeptiert: Podman ist kein unterstützter Deployment-Modus; schlimmstenfalls scheitert Apply am fehlenden systemd/sudo mit der klaren 500-Meldung (Richtungsentscheidung) — kein stiller Schaden. |
| **Rollback (des Features selbst)** | Additiv: Routen + Skript + Unit + sudoers-Baustein per `git revert` entfernbar; Laufzeit-Rückbau: `sudo rm /etc/sudoers.d/eink-picture /etc/systemd/system/eink-updater.service && sudo systemctl daemon-reload`. |
