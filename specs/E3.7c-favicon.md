# E3.7c — Serve /favicon.ico (kill 404/401 console noise)

## Problem
No `/favicon.ico` route exists. The deny-by-default Guard (`middleware/auth.go`) plus the
`/static/`-only static serving (`main.go:248-250`) mean a browser's implicit
`GET /favicon.ico` probe returns **401** for anonymous users (`auth.go:116`) and **404**
when logged in (no mux handler). This is console/log noise on every page load.

## Acceptance criteria
- **AC1 — 200 for everyone.** `GET /favicon.ico` returns **200** with a valid icon and a
  correct `Content-Type`, for BOTH anonymous (no session, password set) and authenticated
  requests. No 401, no 404, no redirect to `/login`. Requires adding the route to the
  public allowlist (`publicRoutes` at `auth.go:36-43`) or extending a public bypass.
- **AC2 — Real asset.** Add a small icon asset under `server/static/` (covered by the
  existing `//go:embed static/*`, `main.go:29`). Simple and on-brand — the e-ink
  panel / 6-color-palette motif is a good fit. Keep it small (a few KB). A PNG is
  acceptable (serve with `image/png`); an `.ico` is also fine.
- **AC3 — Router consistency.** Register the route in `main.go` consistent with the
  existing static pattern (`main.go:245-250`): either a dedicated handler serving the
  embedded file, or a redirect to `/static/favicon.ico` (the `/static/` target is already
  publicly reachable via the guard bypass at `auth.go:86-89`).
- **AC4 — Templates link it.** Add `<link rel="icon">` to the designer page
  (`designer.html`) and the login page so browsers use it directly. The `/favicon.ico`
  route must still work (browsers probe it regardless of the link).
- **AC5 — Tests.** `gofmt -l` empty, `go vet` clean. A handler test asserts 200 +
  content-type for `/favicon.ico` both anonymous (no session) and with a valid session;
  extend the guard/e2e allowlist test so the public status is locked in.

## Non-goals
- No PWA manifest, no multiple icon sizes / apple-touch-icon — one favicon suffices.
