# E5.1: Authentifizierung für das LAN-erreichbare Gerät

**Status:** Draft
**Abhängigkeiten:** keine harten. E2.3 (Setup-Wizard) konsumiert später `POST /api/auth/setup` — die Endpoint-Semantik hier ist dafür bindend. SSRF-Härtung der nutzergesteuerten URLs ist ein SEPARATER Task (Non-Goal hier, s. u.).
**Koordination:** Wer `main.go` (Routing) oder `middleware/` parallel anfasst: Merge-Reihenfolge abstimmen — dieser Task baut die Middleware-Kette um. E2.3 darf `POST /api/auth/setup` nicht umbenennen/ändern, ohne diese Spec zu aktualisieren.
**Zuständig:** go-backend (Server: middleware, auth-Paket, handlers, config, main, compose, env, e2e). **Pflicht-Review zusätzlich durch security-reviewer** (Gate L5). Zwei abgegrenzte Sub-Tasks:
- **Sub-Task pi-client** (`client/client.py`, `client/config.py`, `client/test_client.py`): `X-Client-Token`-Header auf allen vier Server-Calls.
- **Sub-Task frontend-designer** (`server/templates/login.html` NEU, `server/templates/designer.html`, `server/static/js/auth.js` NEU): Login-Seite, 401-Interceptor, „Kein Passwort gesetzt"-Banner, Logout-Button.

## Ziel

Der Server ist im LAN erreichbar (Port 5000, HTTP) und hat heute **null Authentifizierung: alle 55 Routen-Registrierungen offen** (54 `mux.HandleFunc` + 1 `mux.Handle` für `/static/`, main.go Z. 104–196), darunter destruktive Endpoints (`POST /delete_design`, `DELETE /api/media/images/{filename}`, `POST /update_settings`, Uploads). Jeder im LAN (Gast-WLAN, kompromittiertes IoT-Gerät) kann Designs löschen, Bilder hochladen, Settings ändern und den privaten Preview (ggf. Kalenderdaten) abrufen.

Nach diesem Task gilt:

1. **EIN Admin-Passwort** (kein Multi-User), bcrypt-gehasht in `data/auth.json` (0600) persistiert — NICHT im Repo, NICHT in `settings.json` (die geht per `GET /settings` an den Client raus).
2. **Session-Cookie-Auth** für Browser (HttpOnly, SameSite=Lax), Login-Seite + Logout, In-Memory-Session-Store (Neustart = Re-Login, akzeptiert).
3. **Separates Client-Token** (`EINK_CLIENT_TOKEN`, Header `X-Client-Token`) für den Python-E-Ink-Client — der pollt headless und kann keinen Cookie-Flow.
4. **Deny-by-default**: Die Guard-Middleware schützt alles, was nicht explizit auf der Public-Allowlist steht — neue Routen sind automatisch geschützt.
5. **Kein Lockout von Bestandsinstallationen**: Solange kein Passwort gesetzt ist, bleibt Auth deaktiviert (laute Warnung + UI-Banner), Verhalten wie heute.
6. **CORS-Wildcard fällt**: mit Credentials ist `Access-Control-Allow-Origin: *` verboten; local-Mode wird strikt same-origin (keine CORS-Header), cloud-Mode nur mit expliziter Origin-Liste.

### Verifizierte Fakten zum Ist-Zustand

1. **55 Routen-Registrierungen, null Auth:** main.go Z. 104–196; einzige Middleware ist `Logging(CORS(mux))` (Z. 199–200). Kein Handler prüft irgendeine Identität. (Frühere Discovery nannte 49 — heute nachgezählt: 54 `HandleFunc` + 1 `Handle`.)
2. **CORS ist im local-Mode hart `*`** (cors.go Z. 8–11: `origin := "*"`, nur bei `deploymentMode != "local"` UND nicht-leerer Liste wird sie benutzt) — und der Config-Default für `CORS_ALLOWED_ORIGINS` ist ebenfalls `*` (config.go Z. 37). Preflight erlaubt nur `GET, POST` und `Content-Type` (Z. 13–14) — PUT/PATCH/DELETE der neuen APIs wären cross-origin heute schon kaputt; real nutzt das Frontend ausschließlich same-origin-Fetches.
3. **Der Python-Client nutzt VIER Endpoints, nicht drei** (client.py): `GET /settings` (Z. 52, Display-Konfig: Treiber, Farben!), `GET /preview` (Z. 68), `GET /api/refresh_status` (Z. 161), `POST /api/client_heartbeat` (Z. 173). Er sendet heute keinerlei Auth-Header (grep über `client/`: kein `Authorization`/`X-Client`). **Konsequenz: `GET /settings` MUSS per Client-Token erreichbar bleiben** — die Manager-Annahme „drei Endpoints" war unvollständig.
4. **`Config.SecretKey` ist toter Code:** definiert (config.go Z. 13), geladen mit Default `dev-secret` (Z. 35), im gesamten Go-Code nie referenziert (grep). Wird ersatzlos entfernt — Server-seitige Random-Session-Tokens brauchen kein Signing-Secret.
5. **`GET /settings` leakt heute keine Secrets:** `GetSettingsResponse` (services/settings.go Z. 112–125) baut ein explizites Struct (display_type, display, refresh_interval, render_quality, dither_algorithm, calibration). Der bcrypt-Hash darf trotzdem NIE in `models.Settings` landen — `SaveSettings` marshalt das ganze Struct nach `settings.json`, und die geht per Client-Token raus. Deshalb separate Datei `data/auth.json`.
6. **Frontend: 39 `fetch(`-Stellen in 7 JS-Dateien** (design-dashboard 12, designer 10, storage 8, media-library 6, properties-panel/toolbar/widgets je 1), davon 23 mutierend (POST/PUT/PATCH/DELETE). Alle same-origin, kein Framework, kein zentraler HTTP-Wrapper. `fetch` sendet same-origin Cookies per Default (`credentials: 'same-origin'`) — **für den Session-Flow muss KEINE der 39 Stellen angefasst werden**; nur 401-Behandlung fehlt (Architektur-Richtung 6).
7. **Mutierende Server-Routen: 22 Registrierungen** (4× design-legacy-POST, upload/delete_image, 7× `/api/designs`-Mutationen, 4× `/api/media`-Mutationen, `POST /api/preview_live`, `/refresh-display`, `/update_settings`, `/api/trigger_refresh`, `/api/client_heartbeat`) — der CSRF-Schutz muss generisch greifen, nicht pro Route.
8. **Deploy-Pfade und Env:** Docker: `docker-compose.yml` reicht Env explizit an beide Container (`SECRET_KEY`-Zeile existiert dort, Z. „environment"); Nativ: `setup.sh` kopiert `.env.example` → `.env` (Z. 196 ff.), `systemd/eink-{server,client}.service` laden BEIDE dieselbe `.env` via `EnvironmentFile` — ein generiertes `EINK_CLIENT_TOKEN` in `.env` erreicht nativ automatisch beide Prozesse.
9. **Bestehende Handler-Tests umgehen die Middleware** (z. B. settings_test.go: `h.GetSettings(w, req)` direkt) — die Guard-Middleware bricht sie nicht. Der Docker-Healthcheck (`/healthcheck`-Binary, cmd/healthcheck) trifft `/health` — muss öffentlich bleiben, sonst geht der Container auf unhealthy.
10. **e2e-Skript ist komplett unauthentifiziert** (scripts/test-e2e.sh: 20+ `curl -sf` ohne Header/Cookies) — läuft nach diesem Task unverändert nur noch in Phase A (kein Passwort gesetzt).
11. **TLS existiert nicht:** `server/certs/` und `client/certs/` sind leer, Server lauscht plain HTTP (`srv.ListenAndServe`, main.go Z. 218). Cookie-`Secure` ist im LAN-Default also nicht setzbar — dokumentierte Realität, s. Risiken.

### Bindende Architektur-Richtung (Manager-Vorgaben + entschiedene Fragen)

1. **Frage 1 — öffentliche Routen & Client-Token: entschieden wie in der Klassifikation oben.** Abweichung von der Manager-Annahme: es sind VIER Client-Endpoints inkl. `GET /settings` (Fakt 3). Header ist `X-Client-Token: <hex>` (kein `Authorization: Bearer` — kein Schema-Parsing, keine Verwechslung mit künftigen Auth-Schemata). Vergleich per `crypto/subtle.ConstantTimeCompare`. Token kommt aus `EINK_CLIENT_TOKEN` (.env, von den Setup-Skripten via `openssl rand -hex 32` generiert; Docker-Compose reicht es an BEIDE Container, nativ liest die gemeinsame `.env` beide Units — Fakt 8). Ist Auth aktiv, aber `EINK_CLIENT_TOKEN` leer → die vier Endpoints verlangen Session, der Client ist blind: `slog.Error` beim Start („client endpoints require session; e-ink client will fail — set EINK_CLIENT_TOKEN") — bewusst Fail-Closed, kein stiller Bypass. Die Client-Anpassung (Header senden, wenn Token gesetzt; sonst wie heute ohne Header) ist Sub-Task pi-client.
2. **Frage 2 — Session-Store: In-Memory, stdlib only.** `map[string]session{expiry}` + `sync.Mutex`, Token 32 Byte `crypto/rand` base64url (43 Zeichen), TTL **7 Tage sliding** (Zugriff verlängert), Kapazität **32** Sessions (älteste fliegt — Single-Admin-Gerät, bounded Memory), Janitor-Ticker 10 min. Neustart = Re-Login: akzeptiert und dokumentiert. Kein Redis, keine Persistenz, kein Cookie-Signing (Token ist opak und server-seitig — deshalb fällt `SECRET_KEY`, Fakt 4).
3. **Frage 3 — bcrypt via `golang.org/x/crypto/bcrypt`, Cost `bcrypt.DefaultCost` (10). Neue Dependency, begründet:** (a) jede ernsthafte Alternative (scrypt, pbkdf2) liegt AUCH in `x/crypto` — die Modul-Frage ist also identisch, und `golang.org/x/*` ist bereits Vertrauensanker des Projekts (`x/image`, Go-Team-gepflegt); (b) bcrypt ist misuse-resistent: Salt + Cost + Algorithmus stecken im Hash-String, `CompareHashAndPassword` ist eingebaut konstantzeit — scrypt/pbkdf2 erfordern handgerollte Salt-Persistenz, Parameter-Encoding und Vergleich (mehr Code = mehr Review-Fläche beim Security-Thema); (c) Cost 10 auf dem Pi Zero 2 W (Cortex-A53) kostet grob 0,5–1,5 s pro Login — für einen Login-only-Pfad akzeptabel und ein natürlicher Brute-Force-Dämpfer; Gate L3 misst nach, > 3 s → Cost 9 im selben MR (dokumentieren).
4. **Frage 4 — Upgrade-Pfad: konservativ, kein Lockout.** Auth ist NUR aktiv, wenn ein Passwort-Hash existiert (`data/auth.json`). Kein Hash: Verhalten exakt wie heute, plus (a) `slog.Warn` beim Start UND stündlich („authentication disabled — no admin password set, anyone on this network has full access; set one via the web UI or EINK_ADMIN_PASSWORD"), (b) UI-Banner im Designer (via `GET /api/auth/status`), (c) `POST /api/auth/setup` offen (first-come-first-served im LAN — Restrisiko der Übergangsphase, dokumentiert; E2.3-Wizard macht das später zum geführten Pflichtschritt). Bootstrap-Reihenfolge beim Start: `data/auth.json` existiert → benutzen, `EINK_ADMIN_PASSWORD` wird ignoriert + `slog.Warn` („EINK_ADMIN_PASSWORD ignored, password already set — delete data/auth.json to reset"); kein Hash + `EINK_ADMIN_PASSWORD` gesetzt → hashen, persistieren, Auth aktiv (Empfehlung in .env.example: Var danach wieder leeren, Hash ist ja persistiert). **Recovery = `data/auth.json` löschen + Neustart** → zurück in den No-Auth-Zustand (physischer/SSH-Zugriff aufs Gerät nötig — angemessen fürs Produkt). Kein Zwangs-Setup, kein Passwort-Reset-Flow. CHANGELOG dokumentiert das explizit (AC9).
5. **Frage 5 — CORS:** local-Mode: **gar keine CORS-Header** (Designer wird vom selben Server ausgeliefert, same-origin braucht kein CORS; ist `CORS_ALLOWED_ORIGINS` gesetzt, wird sie mit `slog.Warn` ignoriert). cloud-Mode: `CORS_ALLOWED_ORIGINS` = kommagetrennte explizite Origins (Schema+Host+Port); matcht der Request-`Origin` exakt → Echo dieses Origins + `Access-Control-Allow-Credentials: true` + `Vary: Origin`; `*` ist mit Credentials verboten → Startup-`slog.Warn` + behandelt wie unkonfiguriert (same-origin only). Var behält den Namen `CORS_ALLOWED_ORIGINS` (Bestands-.envs), aber der Default in config.go wird `""` statt `*`. Preflight: `Access-Control-Allow-Methods: GET, POST, PUT, PATCH, DELETE, OPTIONS`, `-Headers: Content-Type, X-Client-Token`; `OPTIONS` wird VOR der Guard beantwortet (Preflights tragen keine Cookies). Breaking Change für Cloud-Setups mit `*` → CHANGELOG (AC9).
6. **CSRF — Entscheidung: SameSite=Lax + serverseitiger Origin-Check, KEIN Double-Submit-Token.** Begründung: Lax verhindert Cookie-Mitgabe bei cross-site POST/PUT/PATCH/DELETE bereits auf Browser-Ebene; der Origin-Check ist die zweite, browserunabhängige Schicht: für jede mutierende (non-GET/HEAD/OPTIONS), per **Cookie** authentifizierte Anfrage gilt — `Origin`-Header vorhanden → muss exakt dem erwarteten Origin entsprechen (local: `Host`-basiert; cloud: Host oder konfigurierte Liste), sonst **403**; kein `Origin`, aber `Sec-Fetch-Site` vorhanden → nur `same-origin`/`none` passieren; beides absent (sehr alte Clients) → durchlassen, Lax trägt allein. Client-Token-Requests sind exempt (kein Cookie → kein CSRF-Vektor). Warum kein Double-Submit: das hieße Token-Plumbing in 23 mutierende Vanilla-JS-Fetch-Stellen (Fakt 6/7) ohne zentralen Wrapper — maximale Diff-Fläche für null Sicherheitsgewinn gegenüber Lax+Origin bei einem same-origin-Frontend. Der Origin-Check lebt in der Guard-Middleware → gilt automatisch für alle 22 mutierenden Routen (Fakt 7).
7. **Frontend-401-Handling (Sub-Task frontend-designer):** NEUE Datei `static/js/auth.js`, als erstes Script in `designer.html` geladen; wrappt `window.fetch` global: Response-Status 401 (außer auf `/api/auth/*`) → `window.location = '/login'`. Damit bleiben alle 39 bestehenden Fetch-Stellen (Fakt 6) UNANGETASTET — kein Refactoring auf einen apiFetch-Wrapper in diesem Task (Follow-up-Kandidat). auth.js rendert zusätzlich das Banner (Status-Endpoint) und hängt den Logout-Click an (`POST /api/auth/logout` → Redirect `/login`). Login-Seite: eigenes Template `login.html`, ein Passwortfeld, Fetch auf `/api/auth/login`, bei 200 Redirect fest auf `/designer` (kein `?next=` — keine Open-Redirect-Fläche), Fehlertext bei 401/429; Optik über die vorhandenen `designer.css`-Tokens.
8. **Session-Cookie:** Name `eink_session`, `Path=/`, `HttpOnly`, `SameSite=Lax`, `Max-Age` = TTL; `Secure` nur wenn `r.TLS != nil` ODER `EINK_COOKIE_SECURE=true` (Betrieb hinter TLS-Proxy). LAN-HTTP-Realität: ohne TLS (Fakt 11) ist das Cookie auf dem Draht mitlesbar — dokumentiert (README/INSTALL Sicherheitsabschnitt), TLS-Terminierung ist Non-Goal. Login erzeugt IMMER ein frisches Token (keine Session-Fixation), Logout löscht server-seitig + setzt `Max-Age=0`.
9. **Rate-Limit (Login + Setup):** Fixed Window in-memory: pro Client-IP (aus `r.RemoteAddr`, Host-Teil) max. **5 Versuche pro 60 s** — der 6. Versuch im Fenster bekommt **429** mit `Retry-After`, unabhängig von der Passwort-Korrektheit. Erfolgreicher Login leert den Zähler der IP. Hinter Reverse-Proxy (cloud) zählt die Proxy-IP — bekannte Einschränkung, dokumentiert; `X-Forwarded-For` wird bewusst NICHT geparst (spoofbar ohne Trust-Konzept, Non-Goal). Fehlversuche werden mit IP geloggt (`slog.Warn`), NIE mit Passwort- oder Token-Werten (security.md-Regel).
10. **Middleware-Kette (main.go):** `Logging(CORS(Guard(mux)))`. Die Guard prüft in dieser Reihenfolge: (1) `OPTIONS` → durch (CORS hat geantwortet), (2) Public-Allowlist, (3) `X-Client-Token` für die vier Client-Routen, (4) Session-Cookie (Lookup + Sliding-Renew), (5) Origin-Check falls Cookie-Auth + mutierende Methode. Genau EINE Middleware für Auth+CSRF — eine Entscheidungsstelle, ein Test-Target, keine Reihenfolge-Fallen.

## Akzeptanzkriterien

1. **AC1 — Deny-by-default & Klassifikation (httptest):** Mit gesetztem Passwort, tabellengetriebener Test über den KOMPLETT verdrahteten Handler-Stack (Router + Middleware wie in main.go, dafür Router-Konstruktion in testbare Funktion extrahieren): (a) je eine Route aus jeder Session-Gruppe (z. B. `GET /designs`, `POST /update_settings`, `DELETE /api/media/images/x.png`, `GET /api/widgets/system`) ohne Session → **401** mit JSON-Body `{"message":…}`; (b) `GET /designer` ohne Session → **302** mit `Location: /login`; (c) dieselben Routen mit gültigem Session-Cookie → Status wie heute (200/404-Fachlogik); (d) ein NICHT registrierter Pfad (`GET /api/does_not_exist`) ohne Session → 401, nicht 404 (Beweis deny-by-default: Guard vor Router); (e) `GET /health`, `GET /login`, `GET /api/auth/status`, `GET /static/js/designer.js` ohne Session → 200.
2. **AC2 — Client-Token-Endpoints:** Mit gesetztem Passwort: alle VIER Routen (`GET /settings`, `GET /preview`, `GET /api/refresh_status`, `POST /api/client_heartbeat`) mit korrektem `X-Client-Token` → 200; ohne Token und ohne Session → 401; mit falschem Token → 401; mit Session statt Token → 200 (Designer nutzt `/settings` und `/preview` selbst, Fakt 6). Eine Session-only-Route (`POST /update_settings`) mit gültigem Client-Token, ohne Session → **401** (Token ist KEIN Generalschlüssel). Token-Vergleich nachweislich `subtle.ConstantTimeCompare` (Review-Punkt L5).
3. **AC3 — bcrypt-Roundtrip & Cookie-Attribute:** `POST /api/auth/setup` mit Passwort → 200; `data/auth.json` existiert mit Mode 0600 und `$2a$`/`$2b$`-Hash; Klartext taucht in keiner Datei und keinem Log auf (Test greppt Log-Buffer). `POST /api/auth/login` mit korrektem Passwort → 200 + `Set-Cookie: eink_session=…` mit `HttpOnly`, `SameSite=Lax`, `Path=/`, ohne `Secure` (kein TLS im Test); mit `EINK_COOKIE_SECURE=true` → mit `Secure`. Falsches Passwort → 401. Zweiter `setup`-Call nach gesetztem Passwort → 403 und Hash unverändert.
4. **AC4 — Rate-Limit:** 5 Login-Versuche (falsches Passwort) von einer IP innerhalb 60 s → alle 401; der 6. Versuch → **429** mit `Retry-After`, AUCH mit korrektem Passwort; eine andere Remote-IP (httptest `req.RemoteAddr` setzen) → weiterhin 401/200; nach Fensterablauf (injizierte Clock) → wieder möglich. Gleicher Test für `/api/auth/setup`.
5. **AC5 — CSRF-Negativtest:** `POST /update_settings` mit GÜLTIGEM Session-Cookie, aber `Origin: http://evil.example` → **403** und `settings.json` unverändert; mit `Origin` = eigener Host → 200; ohne `Origin` mit `Sec-Fetch-Site: cross-site` → 403; ohne beide Header → 200 (Alt-Client-Pfad, Lax trägt). Client-Token-Request (`POST /api/client_heartbeat` + Token, ohne Cookie) mit fremdem Origin → 200 (exempt, kein Cookie).
6. **AC6 — Session-Lifecycle:** Sessions expiren: mit injizierter Clock TTL überschreiten → nächster Request 401 und Eintrag aus dem Store entfernt; Zugriff innerhalb TTL verlängert (sliding, Clock vor + nach Renew prüfen). `POST /api/auth/logout` → Folgerequest mit altem Cookie 401, `Set-Cookie` mit `Max-Age=0`. Store-Kapazität: 33. Login verdrängt die älteste Session (deren Cookie → 401). Store-Tests laufen unter `-race` (paralleler Zugriff Login/Guard/Janitor).
7. **AC7 — Upgrade-Pfad ohne Lockout:** OHNE Passwort (kein `data/auth.json`, kein `EINK_ADMIN_PASSWORD`): dieselbe AC1-Routentabelle liefert überall die heutigen Statuscodes (kein 401/302 irgendwo), `GET /api/auth/status` → `{"password_set":false,…}`, Startup loggt die Warnung (Log-Buffer-Assert). MIT `EINK_ADMIN_PASSWORD=x` und ohne auth.json: nach Start existiert der Hash, Auth aktiv, Login mit `x` funktioniert. Mit BEIDEM (auth.json + Env): Env wird ignoriert, Warnung geloggt, altes Passwort gilt weiter.
8. **AC8 — CORS:** local-Mode: Response enthält KEINE `Access-Control-*`-Header (auch nicht auf OPTIONS-Preflight-Antworten hinausgehend nötig; Preflight antwortet 204 ohne Allow-Origin). cloud-Mode + `CORS_ALLOWED_ORIGINS=http://app.example`: Request mit `Origin: http://app.example` → Echo + `Access-Control-Allow-Credentials: true` + `Vary: Origin`; anderer Origin → keine CORS-Header; Konfiguration `*` → Startup-Warnung, Verhalten wie unkonfiguriert. In KEINER Konstellation `Allow-Origin: *` zusammen mit `Allow-Credentials`.
9. **AC9 — Suite, e2e & Doku:** `cd server && go test ./... -count=1` grün (Auth-/Guard-/Store-Tests inkl. `-race`), `gofmt -l .` leer, `go vet ./...` sauber; Golden-Suite unberührt grün. `scripts/test-e2e.sh` erweitert und grün: Phase A (frisches `DATA_DIR`, kein Passwort) = bestehende Checks unverändert; Phase B: setup → geschützte Route ohne Cookie 401 → Login (`curl -c` Cookie-Jar) → dieselbe Route 200 → Client-Endpoint mit `X-Client-Token` 200 / ohne 401 → Logout → 401. `.env.example` (+`EINK_ADMIN_PASSWORD`, `EINK_CLIENT_TOKEN`, `EINK_COOKIE_SECURE`, −`SECRET_KEY`), `docker-compose.yml` (Token an beide Container, `SECRET_KEY` raus), Setup-Skripte (Token-Generierung + .env-Upgrade) angepasst; `CHANGELOG.md` (Keep a Changelog) dokumentiert: Auth-Feature, Upgrade-Pfad („bestehende Installationen bleiben offen, bis ein Passwort gesetzt wird"), Recovery via auth.json-Löschung, CORS-`*`-Breaking-Change (cloud), `SECRET_KEY`-Entfernung; README/INSTALL: Sicherheitsabschnitt (LAN-HTTP, kein TLS, Cookie-Realität).
10. **AC10 — Sub-Task pi-client:** `client/config.py` liest `EINK_CLIENT_TOKEN` (Default `""`); alle vier Requests senden `X-Client-Token` wenn Token nicht leer, sonst exakt heutiges Verhalten (kein Header); `client/test_client.py` deckt beide Fälle ab (Header gesetzt/abwesend, z. B. via requests-mock/Monkeypatch). Client gegen Server ohne Auth = heutiges Verhalten (Kompatibilität in beide Richtungen).

## Non-Goals

- **Kein Multi-User, keine Rollen** — ein Admin-Passwort, Punkt (Produktmodell-Vorgabe).
- **Keine TLS-Terminierung** — Server bleibt plain HTTP; `Secure`-Cookie nur via `EINK_COOKIE_SECURE` hinter eigenem Proxy. LAN-Sniffing-Realität wird dokumentiert, nicht gelöst.
- **Kein OAuth/OIDC, keine Passwort-Reset-Mail** — Recovery ist auth.json-Löschung mit Gerätezugriff.
- **Keine SSRF-Fixes** — nutzergesteuerte URL-Fetches (Widgets, `/refresh-display`-Proxy) sind ein separater Task; hier wird nur der Zugriff auf diese Endpoints authentifiziert.
- **Kein Refactoring der 39 Fetch-Stellen** auf einen apiFetch-Wrapper — der globale Interceptor (Architektur-Richtung 7) reicht; Wrapper ist Follow-up-Kandidat.
- **Kein persistenter Session-Store, kein Redis** — Neustart = Re-Login (Architektur-Richtung 2).
- **Kein `X-Forwarded-For`-Trust-Konzept** — Rate-Limit zählt `RemoteAddr` (Architektur-Richtung 9).
- **Kein Setup-Wizard-UI** — nur der `POST /api/auth/setup`-Endpoint + Banner; der geführte Wizard ist E2.3.
- **Kein Audit-Log/Brute-Force-Persistenz** — In-Memory-Limiter genügt dem Bedrohungsmodell (LAN-Gerät).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| **Lockout** (Passwort vergessen) | Recovery = `data/auth.json` löschen + Neustart (Gerätezugriff nötig) → zurück in No-Auth-Zustand mit Banner; in README/INSTALL + CHANGELOG dokumentiert (AC9). Kein Zwangs-Setup beim Upgrade (AC7). |
| First-come-Setup im offenen Zustand: jeder im LAN kann das erste Passwort setzen | Übergangsphase bewusst akzeptiert (heute kann jeder im LAN ALLES) — lautes Log + Banner drängen zum sofortigen Setzen; E2.3 macht Setup zum geführten Erstschritt. Setup-Endpoint ist rate-limitiert und nach dem ersten Erfolg tot (403). |
| Cookie über LAN-HTTP mitlesbar (kein TLS, Fakt 11) | Bedrohungsmodell-Entscheidung: schützt gegen zufällige/neugierige LAN-Teilnehmer und CSRF, nicht gegen aktive MITM. Dokumentiert; `EINK_COOKIE_SECURE` für Proxy-Setups; TLS ist Non-Goal. |
| Frontend-Fetches nach Session-Expiry: 39 Stellen ohne Fehlerbehandlung (Fakt 6) | Globaler `fetch`-Interceptor in `auth.js` (eine Datei, kein Touch an den 39 Stellen); Ausnahme `/api/auth/*` verhindert Redirect-Loop auf der Login-Seite. Sub-Task frontend-designer. |
| E-Ink-Client bricht beim Upgrade (Token fehlt in alter .env) | Setup-Skripte hängen `EINK_CLIENT_TOKEN` an bestehende .env an (Fakt 8); solange kein Passwort gesetzt ist, laufen Client-Endpoints ohnehin offen weiter (AC7) — der Bruch kann erst NACH bewusstem Passwort-Setzen auftreten und ist dann per Startup-`slog.Error` diagnostizierbar (Architektur-Richtung 1). Docker: Compose reicht Token an beide Container. |
| Rate-Limit hinter Reverse-Proxy bestraft alle Nutzer kollektiv (eine Proxy-IP) | Single-Admin-Gerät: kollektiv = eine Person; XFF-Parsing ohne Trust-Konzept wäre spoofbar → bewusst Non-Goal, dokumentiert. |
| bcrypt Cost 10 zu langsam auf Pi Zero 2 W | Gate L3 misst; Fallback Cost 9 (immer noch 2^9 Runden) im selben MR. Betrifft nur Login/Setup, nie den Render-/Poll-Pfad. |
| CORS-Umstellung bricht Cloud-Setups mit `*` | Breaking Change, gewollt (Credentials + `*` ist unhaltbar); CHANGELOG + INSTALL beschreiben die nötige explizite Origin-Liste (AC9). local-Mode ist unbetroffen (same-origin). |
| In-Memory-Sessions weg bei jedem Deploy/Neustart | Akzeptiert (Architektur-Richtung 2): Re-Login-Kosten minimal, dafür null Persistenz-/Verschlüsselungs-Komplexität. |
| Guard-Middleware als Single Point of Failure | Genau deshalb EINE Middleware mit Tabellen-Tests (AC1–AC6) statt verstreuter Checks; deny-by-default macht Fehler „zu strikt" (401, sichtbar) statt „zu offen" (leise). |
| **Rollback** | Ein Server-MR (auth-Paket, middleware, handlers, config, main, Tests) + Compose/Env/Skripte/Doku + zwei kleine Sub-MRs (client, frontend). `git revert` stellt den offenen Zustand wieder her; `data/auth.json` bleibt als tote Datei liegen (harmlos, wird ignoriert), `EINK_CLIENT_TOKEN` in .env ist additiv. Keine persistenten Formatänderungen an settings.json/Designs. |
