# E6.2: Release-Workflow — Tag → Binaries + install.sh als Release-Assets

**Status:** Draft
**Abhängigkeiten:** E6.1 (CI-Pipeline, gemergt: Cross-Build-Matrix als Vorlage); Mini-Anteil go-backend (version-Variable, siehe AC1) muss VOR dem Workflow-Teil gemergt sein
**Blockiert:** E2.4 (Self-Update — konsumiert Release-Tags, `main.version` und das Version-Feld in `/health`); mittelbar jede Binary-Distribution an Nutzer
**Zuständig:** test-engineer — Schreibrecht NUR `.github/**` für den Workflow-Teil. **Ausnahme AC1 (Mini-Anteil):** `server/main.go` + `server/internal/handlers/health.go` (+ Tests) gehören go-backend und werden als separater, vorgelagerter Commit umgesetzt — nicht vom test-engineer.

## Ziel

Ein Push eines Tags `v*` erzeugt automatisch ein GitHub-Release mit: den vier Cross-Build-Binaries (`server-linux-<arch>-<version>`, version-gestempelt via ldflags), `install.sh` als Asset (Frischinstallation exakt auf dem Tag-Stand via `EINK_REPO_BRANCH=<tag>`), `sha256sums.txt` über alle Assets und Release-Notes aus dem passenden CHANGELOG-Abschnitt. Der Release-Prozess (CHANGELOG-Umbenennung, Taggen) ist als Konvention dokumentiert.

## Kontext

### Betroffene Dateien

| Datei | Rolle |
|---|---|
| `.github/workflows/release.yml` | NEU — Haupt-Deliverable (test-engineer) |
| `.github/workflows/ci.yml` | Nur lesen — Cross-Build-Matrix als Vorlage; wird bewusst NICHT zu `workflow_call` refactored (Fakt 2) |
| `server/main.go` | Mini-Anteil go-backend (AC1): `var version = "dev"` + Startup-Log |
| `server/internal/handlers/health.go` | Mini-Anteil go-backend (AC1): Version im `/health`-JSON (heute statisch `{"status":"ok"}`, Z. 5–8) |
| `install.sh` | Nur lesen — wird unverändert als Release-Asset angehängt; `EINK_REPO_BRANCH` (Z. 23) akzeptiert Tags (Fakt 5) |
| `CHANGELOG.md` | Nur lesen — Keep-a-Changelog-Format mit `## [Unreleased]`-Block (Z. 8); Quelle der Release-Notes; die Umbenennung beim Taggen ist Prozess (AC6), keine Änderung in diesem Task |
| `setup.sh` | Nur lesen — `build_server` nutzt `-ldflags="-s -w"` (Z. 437); `--update` = `git pull --ff-only` (Z. 273) → bricht auf Tag-Checkout (Fakt 5) |

### Verifizierte Fakten

1. **Keine Tags, keine Releases:** `git tag -l` ist leer; Remote ist `github.com/Kilian-Schwarz/E-INK-Picture`. Dieser Workflow erzeugt das erste Release überhaupt — keine Namens-/Konventionskollisionen.
2. **Cross-Build-Matrix existiert in ci.yml** (Z. 70–117): 4 Targets (`arm64`, `arm/GOARM=6`, `arm/GOARM=7`, `amd64`), `GOOS=linux`, `CGO_ENABLED=0`, `fail-fast: false`, Artefaktnamen `server-linux-<arch>`. Wiederverwendung via `workflow_call` würde einen Umbau der frisch bewiesenen ci.yml erfordern (E6.1 Gate L2 grün auf 865dcb2) — **Entscheidung: Matrix wird in release.yml dupliziert** (mit Verweis-Kommentar auf ci.yml als Quelle; Drift-Risiko bewusst getragen, siehe Risiken), plus release-spezifische Abweichungen: Version im Artefaktnamen, ldflags-Stempel.
3. **Keine version-Variable vorhanden:** `grep -rn 'var version' server/` trifft nichts; `main.go` loggt keinen Versionsstand, `health.go` antwortet statisch. `-ldflags "-X main.version=…"` auf ein nicht existierendes Symbol ist ein stiller No-Op — **der go-Anteil (AC1) ist harte Voraussetzung für den Stempel**, nicht optional.
4. **Stempel-Verifikation ohne Ausführung:** `go version -m <binary>` zeigt die Build-Settings inkl. der `-ldflags`-Zeile auch bei `-s -w`-Binaries (Buildinfo überlebt das Stripping) — funktioniert auf dem amd64-Runner für alle vier Architekturen. ARM-Binaries müssen für den Beweis nie ausgeführt werden.
5. **install.sh ist tag-fähig für Frischinstallationen:** `EINK_REPO_BRANCH` fließt in `git clone --depth 1 --branch` (install.sh Z. 23/127), und git akzeptiert dort auch Tags → `curl -fsSL <asset-url>/install.sh | EINK_REPO_BRANCH=v0.9.0 bash` installiert exakt den Tag-Stand. **Limitation:** Der Checkout ist dann Detached-HEAD; `setup.sh --update` (`git pull --ff-only`) schlägt darauf fehl. Update aus einem Tag-Checkout heraus ist E2.4 (Update-Skript arbeitet fetch+checkout-basiert) — hier nur ehrlich dokumentiert.
6. **CHANGELOG-Extraktion ist mechanisch möglich:** Abschnitte beginnen mit `## [x.y.z]`-Überschriften; der Abschnitt eines Tags `vX.Y.Z` ist der Text zwischen `## [X.Y.Z]` und der nächsten `## [`-Zeile (awk-Einzeiler). Konvention: Tag trägt das `v`-Präfix, die Überschrift nicht.
7. **Version-Feld in /health ist additiv sicher:** `server/cmd/healthcheck` prüft nur Status 200 (kein Body-Parse), `client/client.py` ruft `/health` nie auf, der Compose-Healthcheck nutzt das healthcheck-Binary. Kein Konsument parst das JSON strikt.
8. **GITHUB_TOKEN braucht `contents: write`** zum Erstellen von Releases; Repo-Default-Permissions können restriktiver sein → explizites `permissions:`-Feld im Workflow.
9. **`gh` CLI ist auf ubuntu-latest vorinstalliert** und kann Releases mit Assets + Notes-Datei erstellen (`gh release create --verify-tag --notes-file … <files>`). Gegenüber `softprops/action-gh-release` entfällt eine Dritt-Action (Supply-Chain) komplett — **Entscheidung: `gh release create`** (Begründung siehe AC5; softprops@v2 Major-gepinnt bleibt als gleichwertige Alternative dokumentiert, falls gh-Eigenheiten auftauchen).

## Akzeptanzkriterien

1. **AC1 — Version-Variable + /health (Mini-Anteil go-backend, vorgelagerter Commit):** `server/main.go` erhält `var version = "dev"` (Package-Level, überschreibbar via `-ldflags "-X main.version=…"`). Der Startup-Log (`"server starting"`) bekommt ein `version`-Attribut. `/health` liefert `{"status":"ok","version":"<version>"}` — dafür wird `handlers.HealthCheck` zu einem parametrisierten Konstruktor (z. B. `handlers.HealthCheck(version string) http.HandlerFunc`), main reicht `version` durch; bestehende Tests werden angepasst, ein neuer Test prüft das Feld. **Bewertung Version pre-auth sichtbar:** `/health` ist public (Guard-Allowlist) — die Versionsnummer ist damit unauthentifiziert lesbar. Akzeptiert: LAN-Gerät, und E2.4 braucht genau das — das Update-Skript muss nach dem Restart OHNE Session beweisen können, dass die Ziel-Version läuft (`version == tag` statt nur HTTP 200). Klein halten: keine weiteren Felder, kein eigener /version-Endpoint.
   *Beweis:* `go test ./... -count=1` grün; `curl /health` (Dev-Lauf) zeigt `"version":"dev"`; Testbuild mit `-ldflags "-X main.version=v0.0.0-test"` zeigt `v0.0.0-test`.

2. **AC2 — Workflow-Grundgerüst:** Neue Datei `.github/workflows/release.yml`, `name: release`, Trigger ausschließlich `push` auf Tags `v*` (`tags: ["v*"]` — kein Branch-, kein PR-Trigger). Explizit `permissions: contents: write` (Fakt 8). Kein `concurrency`-Cancel (Releases sind selten und dürfen nie halb abgebrochen werden).
   *Beweis:* Datei im Diff; `actionlint` bzw. YAML-Parse (Gate L1).

3. **AC3 — Gate-Job vor dem Build:** Ein Job `gate` läuft zuerst und spiegelt die ci.yml-Prüfungen: gofmt (identische Zeile wie ci.yml), `go vet ./...`, `go test ./... -count=1` sowie die Python-Schritte (3.11, nur `requests`+`Pillow`, `py_compile` + `unittest`). Build- und Release-Jobs hängen via `needs: gate` daran. Grund: ein Tag kann auf einen Commit zeigen, der nie durch CI lief — es darf kein Release aus ungetestetem Stand entstehen. Kommentar im Workflow verweist auf ci.yml als Quelle (Drift-Hinweis).
   *Beweis:* `needs:`-Kette im YAML; Gate-Kommandos identisch zu ci.yml (Diff-Vergleich im Review).

4. **AC4 — Build-Matrix mit Versionsstempel:** Job `build` (needs: gate) mit derselben 4-Target-Matrix wie ci.yml (Fakt 2: `arm64`, `armv6`, `armv7`, `amd64`; `CGO_ENABLED=0`, `fail-fast: false`), aber:
   - Artefaktname `server-linux-<arch>-${{ github.ref_name }}` (z. B. `server-linux-armv6-v0.9.0`),
   - Build mit `go build -ldflags "-s -w -X main.version=${{ github.ref_name }}"` (`-s -w` wie setup.sh, Stempel neu),
   - Verifikation pro Binary: `test -s` UND `go version -m <binary>` enthält `main.version=<tag>` (Fakt 4 — beweist den Stempel ohne Ausführung, auch für ARM),
   - Upload via `actions/upload-artifact` (Major-gepinnt).
   *Beweis:* Lokale Nachstellung eines Matrix-Eintrags inkl. `go version -m`-Grep (Gate L1).

5. **AC5 — Release-Job (Assets + Checksums + Notes):** Job `release` (needs: build):
   - lädt alle vier Binaries via `actions/download-artifact`,
   - erzeugt `sha256sums.txt` im Standard-`sha256sum`-Format über die vier Binaries **und** `install.sh` (aus dem Checkout des Tag-Commits),
   - extrahiert die Release-Notes aus CHANGELOG.md: Abschnitt `## [X.Y.Z]` passend zum Tag `vX.Y.Z` (Fakt 6). **Ist der Abschnitt leer oder fehlt er, schlägt der Workflow fehl** — laut und absichtlich: das erzwingt den Release-Prozess aus AC6, kein stiller Fallback auf Auto-Notes,
   - erstellt das Release mit `gh release create "$TAG" --verify-tag --title "$TAG" --notes-file <notes> <4 Binaries> install.sh sha256sums.txt` (`GH_TOKEN: ${{ github.token }}`; Entscheidung Fakt 9 — keine Dritt-Action; falls im ersten echten Lauf gh-Eigenheiten blocken, ist `softprops/action-gh-release@v2` die dokumentierte, Major-gepinnte Alternative),
   - markiert als Prerelease, wenn der Tag einen Bindestrich enthält (`v1.0.0-rc.1`-Stil), sonst reguläres Release.
   Ergebnis: genau 6 Assets (4 Binaries, `install.sh`, `sha256sums.txt`), Notes = CHANGELOG-Abschnitt.
   *Beweis:* Extraktions-Skript lokal gegen ein simuliertes CHANGELOG mit `[0.9.0]`-Abschnitt (positiv) und ohne (negativ → Exit ≠ 0) — Gate L1; Asset-Zählung ist Gate L2.

6. **AC6 — Dokumentierter Release-Prozess (Konvention, im Workflow-Kopfkommentar + dieser Spec):**
   1. **Manager/docs-writer** benennt im CHANGELOG `## [Unreleased]` in `## [X.Y.Z] - YYYY-MM-DD` um und legt einen frischen, leeren `## [Unreleased]`-Block darüber an; Commit `chore(release): prepare vX.Y.Z` auf `main`.
   2. **Manager** wartet CI-grün auf diesem Commit ab und taggt: `git tag vX.Y.Z && git push origin vX.Y.Z`. Nur der Manager taggt.
   3. Der Workflow übernimmt den Rest; der Manager verifiziert die Release-Seite (Gate L2).
   **Erster Tag: `v0.9.0`** als Release-Candidate-Konvention — `v1.0.0` bleibt für die erfüllte Definition of Done reserviert. Das Auslösen ist expliziter Manager-Folgeschritt, nicht Teil dieses Tasks.
   *Beweis:* Kopfkommentar in release.yml enthält die drei Schritte; Spec-Abschnitt vorhanden.

7. **AC7 — Scope-Hygiene:** Der test-engineer-Diff berührt ausschließlich `.github/**`. Der AC1-Diff (go-backend) berührt ausschließlich `server/main.go`, `server/internal/handlers/health.go` und zugehörige Tests. Keine Secrets im Workflow (GITHUB_TOKEN genügt, Fakt 8/9).
   *Beweis:* `git diff --stat` je Commit.

## Non-Goals

- **Kein Binary-Install-Pfad in install.sh** — der Installer bleibt quellbasiert (clone + Build); die Umstellung auf Release-Binary-Download ist ein bewusster Folge-Task (dann inkl. Checksum-Verifikation clientseitig).
- **Kein Self-Update** — Endpoint, Update-Skript, Rollback sind E2.4 (konsumiert die Artefakte dieses Tasks).
- **Keine Docker-Images/GHCR** — Registry-Publishing ist ein eigener Task; die Cross-Binaries decken die native Distribution ab.
- **Keine Signierung** (GPG/cosign/Attestations) — `sha256sums.txt` genügt für v1; Signierung wäre Schlüsselverwaltungs-Overhead ohne aktuellen Bedrohungsanlass.
- **Keine CHANGELOG-Automatisierung** — die Unreleased-Umbenennung bleibt bewusst ein manueller, gereviewter Schritt (AC6); kein Release-Bot, kein Auto-Tagging.
- **Kein workflow_call-Refactoring von ci.yml** — bewusste Duplikation der Matrix (Fakt 2).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Der erste echte Lauf ist der Beweis (Runner lokal nicht ausführbar) | Wie E6.1: Gate L1 beweist Kommandos, nicht den Workflow. `v0.9.0` ist bewusst als RC-Konvention gewählt — ein Fehlschlag kostet Tag-Löschung + Fixup, kein Nutzer hängt an v0.9.0. |
| Matrix-Drift zwischen ci.yml und release.yml (neues Target nur in einer Datei) | Bewusster Trade-off (Fakt 2); Verweis-Kommentare in beiden Dateien; Targets ändern sich selten (Deployment-Realität). |
| Tag auf kaputtem/ungetestetem Commit | Gate-Job (AC3) bricht das Release ab; Prozess (AC6) verlangt zusätzlich CI-grün vor dem Taggen. |
| CHANGELOG-Umbenennung vergessen → Release ohne Notes | Extraktion schlägt hart fehl (AC5) — das Release entsteht gar nicht erst; Fehlermeldung nennt den erwarteten Abschnitt. |
| `gh release create`-Eigenheiten (z. B. Notes-Encoding, Asset-Limits) | Dokumentierte Alternative softprops@v2 (Major-Pin); Wechsel ist ein Ein-Datei-Fixup. |
| Versionsstempel ≠ tatsächlicher Code-Stand (Tag nachträglich verschoben) | `--verify-tag` + Checkout via Tag-Ref im Workflow; Tags werden per Konvention nie verschoben (AC6: nur Manager taggt). |
| armv6-Binary grün gebaut, auf echtem Pi Zero defekt | Gleicher blinder Fleck wie E6.1 (kein Testgerät); `go version -m` beweist nur den Stempel, nicht Lauffähigkeit. Dokumentiert, kein Blocker. |
| Version-Feld in /health verrät Patch-Stand pre-auth | Bewertet in AC1: akzeptiert (LAN-Gerät, E2.4-Rollback-Beweis braucht es unauthentifiziert). Wer das Gerät im LAN erreicht, sieht heute schon Login-Seite + Verhalten. |
| **Rollback** | Workflow: `git revert` der release.yml. Fehlgeschlagenes/falsches Release: Release + Tag auf GitHub löschen (`gh release delete v0.9.0 --yes && git push origin :v0.9.0`), Ursache fixen, neu taggen. AC1-Anteil: additiv, revertbar. |

## Offene Punkte

1. **Erster Tag `v0.9.0`** — Manager-Folgeschritt nach Merge (Gate L2, inkl. CHANGELOG-Prepare-Commit durch Manager/docs-writer).
2. **Umstellung install.sh auf Release-Binaries** (Non-Goal hier) — eigener Folge-Task, sobald v0.9.0 existiert und E2.4 die Update-Semantik für Tag-Checkouts geklärt hat.
3. **Branch-Protection/Tag-Schutz** (nur Manager darf `v*` pushen) — GitHub-Repo-Einstellung, keine Datei in diesem Repo.
