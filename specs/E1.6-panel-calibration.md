# E1.6: Panel-Kalibrierung — Quantisierung gegen die reale Spectra-6-Palette + wählbare Dithering-Algorithmen

**Status:** Draft
**Abhängigkeiten:** E1.1 (gemergt) — Golden-Harness-Konventionen (Font-Pinning, `-update`-Disziplin, deterministische Designs) gelten unverändert. E1.2 (Last-Sent-Artifact) wird in Gate L3 zur Kontrolle genutzt.
**Koordination:** E1.5 (Element-Rotation) erzeugt parallel Golden-Churn an denselben Dateien — Merge-Reihenfolge abstimmen (s. Risiken).
**Zuständig:** render-quality — Go-Server (models, services, handlers) + Tests + `testdata/`. Kein Frontend-, kein Client-Anteil.

## Ziel

Die Palette-Quantisierung in `PreviewService.Render` dithert nicht mehr gegen die idealen Treiber-Primärfarben (#FF0000 etc.), sondern gegen eine **perzeptuelle Panel-Palette** (realistische RGB-Erscheinung der 6 Spectra-6-Farben) und mappt das Ergebnis index-erhaltend zurück auf die reinen Treiber-Codes. Zusätzlich: konfigurierbare **Vorkompensation** (Gamma/Sättigung/Kontrast als Preset pro Display-Profil) und **wählbarer Dithering-Algorithmus** (`floyd_steinberg` | `atkinson`). Das Ausgabe-PNG enthält weiterhin AUSSCHLIESSLICH die Treiber-Farben — Client, Treiber und die Golden-/Palette-Exaktheits-Mechanik bleiben strukturell unverändert.

**Warum:** Das reale Spectra-6-Panel (epd7in3e) zeigt deutlich entsättigte, dunklere Farben als die Ideal-Primärfarben, gegen die heute gedithert wird. Floyd-Steinberg verteilt Quantisierungsfehler relativ zur *angenommenen* Farbe der Palette-Einträge — nimmt der Algorithmus Gelb als #FFFF00 an, obwohl das Panel ~#EFDE44 zeigt, gewichtet er Gelb-Anteile systematisch falsch (Gelb wirkt auf dem Panel dunkler/grünlicher, Fotos kippen im Farbeindruck). Dithern gegen die reale Erscheinung + Ausgabe der Treiber-Codes korrigiert das, ohne irgendetwas am Übertragungsformat zu ändern.

### Verifizierte Fakten zum Ist-Zustand

1. **Quantisierung heute:** `quantizeToPalette(finalImg, displayCfg.Colors)` baut eine `color.Palette` direkt aus den Ideal-Hexwerten aus `models/display.go` und dithert per `draw.FloydSteinberg.Draw` (stdlib). Für epd7in3e sind das `#000000, #FFFFFF, #FF0000, #FFFF00, #00FF00, #0000FF` — genau die Codes, die der Waveshare-Treiber `epd7in3e` als Palette-Eingabe erwartet. Die Ausgabe-Codes sind also FIX; nur die *Distanzrechnung beim Dithern* darf realistischere Werte nutzen.
2. **stdlib-Semantik:** `draw.FloydSteinberg` arbeitet strikt zeilenweise links→rechts (keine Serpentine), Nearest-Match über quadratische Distanz (`color.Palette.Index`), Fehlergewichte 7/16, 3/16, 5/16, 1/16. Sie schreibt in ein `*image.Paletted` — **die Indizes referenzieren die Palette**, d. h. ein nachträglicher Austausch von `paletted.Palette` ändert die Farbzuordnung aller Pixel index-erhaltend, ohne einen einzigen Pixel neu zu berechnen. Das ist der Mechanismus für Stufe 2 der Quantisierung.
3. **PNG-Encoding:** `png.Encode` eines `*image.Paletted` schreibt die PLTE aus `paletted.Palette` — nach dem Palette-Swap enthält das PNG ausschließlich Treiber-Farben. `TestPaletteExactness` (prüft jeden dekodierten Pixel gegen die Ideal-Palette) bleibt daher unverändert gültig.
4. **Settings-Muster (`render_quality`):** typisierter String in `models/settings.go`, `omitempty` im persistierten `Settings`, immer gesetzt in `SettingsResponse`, Default-Normalisierung bei leerem Wert in `SettingsService.GetSettings`, Getter für Services, Whitelist-Validierung mit 400 in `handlers/settings.go` (leer = unverändert lassen). Die neuen Felder folgen exakt diesem Muster.
5. **Golden-Harness:** `setupGoldenServices` schreibt settings.json als String (golden_test.go Z. 59) — die neuen Felder MÜSSEN dort explizit gepinnt werden, sonst hängen Golden Files an Default-Änderungen statt an expliziter Konfiguration. `goldenDesigns` derzeit `{"basic", "gradient"}` (+ `"rotation"` nach E1.5).
6. **Renderfluss:** Element-Loop → supersampled RGBA → CatmullRom-Downscale auf 800×480 → Quantisierung. Vorkompensation setzt auf dem **downgescalten** `finalImg` an (384.000 Pixel, ein Durchlauf), nicht auf dem supersampled Canvas — billiger und äquivalent, da Precomp pro Pixel unabhängig ist.
7. **API-Wege für das Testbild (verifiziert in `main.go`/`handlers/design.go`):** `POST /api/designs` akzeptiert `{name, canvas, elements}` (das testdata-JSON-Format ist eine Obermenge davon), `POST /api/designs/{id}/activate`, `POST /api/trigger_refresh`, `POST /update_settings`. Damit ist der dokumentierte Panel-Workflow (AC7) ohne neuen Code möglich.

### Bindende Architektur-Richtung (Manager)

1. **Zweistufige Quantisierung:** Fehlerdiffusion rechnet gegen die perzeptuelle Panel-Palette; danach index-erhaltender Palette-Swap `paletted.Palette = driverPalette` (Kontext Punkt 2). Das Ausgabe-PNG enthält ausschließlich die 6 Treiber-Farben. Golden Files der epd7in3e-Designs DÜRFEN sich dadurch ändern — bewusst, im selben Commit, mit Sichtprüfung.
2. **Vorkompensation:** pro Display-Profil ein Preset `{Gamma, Saturation, Contrast float64}` (Identität = `{1, 1, 1}`), angewandt auf `finalImg` VOR der Fehlerdiffusion, nur bei `calibration: "default"`. Definition (deterministisch, integer-basiert):
   - Gamma + Kontrast als kombinierte 256-Einträge-LUT pro Kanal: `lut[v] = clamp(round((255·(v/255)^Gamma − 128)·Contrast + 128), 0, 255)` — die LUT wird EINMAL pro Render gebaut (kein `math.Pow` pro Pixel).
   - Sättigung danach kanal-mischend in Ganzzahlarithmetik: `luma = (299·r + 587·g + 114·b) / 1000`; `out_c = clamp(luma + (c − luma)·sat1000/1000, 0, 255)` mit `sat1000 = int32(round(Saturation·1000))`.
   - Identitäts-Preset ⇒ der komplette Pass wird übersprungen (kein Pixel angefasst) — Voraussetzung für die Byte-Identitäts-ACs.
   - Preset epd7in3e: `{Gamma: 1.0, Saturation: 1.15, Contrast: 1.0}` (konservativ, initial guess, tunable — moderater Sättigungs-Boost ist gängige Praxis für Spectra-6-Fotorendering). Preset waveshare_7in5_v2: Identität; dessen `PanelPalette` == Treiber-Palette (`#000000, #FFFFFF`).
3. **Atkinson-Dithering (Eigenimplementierung, stdlib hat nur FS):**
   - Scanreihenfolge: zeilenweise, links→rechts, oben→unten, **keine Serpentine** — Begründung: Determinismus und direkte Vergleichbarkeit mit der stdlib-FS-Referenz, die ebenfalls strikt links→rechts arbeitet; Serpentine wäre im Panel-A/B-Vergleich eine zweite unkontrollierte Variable.
   - Pro Pixel: `value_c = clamp(src_c + acc_c, 0, 255)`; Nearest-Match = minimale quadratische RGB-Distanz gegen die aktive Palette, bei Gleichstand gewinnt der **niedrigste Palette-Index**; `err_c = value_c − palette[idx]_c` (int32).
   - Diffusion: `err_c / 8` (Go-Ganzzahldivision, Truncation Richtung 0) auf genau 6 Nachbarn: `(x+1,y)`, `(x+2,y)`, `(x−1,y+1)`, `(x,y+1)`, `(x+1,y+1)`, `(x,y+2)`. Außerhalb der Bildgrenzen: verwerfen. Die verbleibenden 2/8 des Fehlers werden bewusst NICHT verteilt (Atkinson-Charakteristik: höherer lokaler Kontrast, hellere Lichter — auf E-Ink oft ruhigeres Bild).
   - Implementierung: rollierender `int32`-Fehlerpuffer über 3 Zeilen (y, y+1, y+2), ≈ 800·3·3·4 B ≈ 29 KB; keine Float-Operation in der inneren Schleife; Ausgabe direkt als `*image.Paletted`.
4. **Konfiguration (Muster `render_quality`):**
   - `models.DitherAlgorithm`: `"floyd_steinberg"` (Default) | `"atkinson"`; Settings-Feld `dither_algorithm,omitempty`.
   - `models.CalibrationMode`: `"default"` (Default) | `"off"`; Settings-Feld `calibration,omitempty`.
   - Semantik: `"default"` = Profil-Preset aktiv (perzeptuelle Palette + Vorkompensation des Display-Profils); `"off"` = Ideal-Palette, keine Vorkompensation — der Fluchtweg zum heutigen Verhalten.
   - **Abwärtskompatibilität (bindende Entscheidung):** fehlende/leere Felder = NEUE Defaults, d. h. Kalibrierung ist für epd7in3e ab diesem Release AKTIV. Der CHANGELOG dokumentiert das sichtbar; wer das alte Bild will, setzt `{"calibration":"off"}` via `POST /update_settings`.
   - `GET /settings` (`SettingsResponse`) liefert beide Felder immer aufgelöst; `POST /update_settings` validiert per Whitelist (leer = unverändert, unbekannter Wert = 400) — exakt das `render_quality`-Muster.
5. **Defensive Invariante:** `GetCalibrationProfile(t)` liefert für unbekannte Display-Typen oder bei `len(PanelPalette) != len(DisplayConfig.Colors)` das Identitäts-Profil (Treiber-Palette, Precomp-Identität) und loggt eine Warnung — die Pipeline darf niemals mit einer Palette anderer Länge dithern (Index-Mapping wäre korrupt).
6. **`raw=true` (Debug-Preview) bleibt unverändert:** weder Vorkompensation noch Quantisierung — der Designer-Raw-Modus zeigt weiterhin die reine Komposition.

## Akzeptanzkriterien

1. **AC1 — Settings-Felder Ende-zu-Ende:** `dither_algorithm` und `calibration` existieren in `Settings`/`SettingsResponse`; `GET /settings` liefert bei fehlender settings.json bzw. fehlenden Feldern `"floyd_steinberg"`/`"default"`; `POST /update_settings` persistiert gültige Werte, lässt nicht gesendete Felder unverändert und beantwortet unbekannte Werte (z. B. `"dither_algorithm":"bayer"`) mit 400 ohne Persistenz-Nebenwirkung.
   *Beweis:* neue Subtests in `handlers/settings_test.go` (Muster `TestUpdateSettings_RefreshInterval`) und `services/settings_test.go` (Defaults + Save/Load-Roundtrip) grün.

2. **AC2 — `"off"` + `floyd_steinberg` ist das heutige Verhalten, byte-exakt:** Ein neuer Test `TestCalibrationOffMatchesLegacy` quantisiert ein synthetisches 800×480-Gradientenbild (Muster `writeGradientPhoto`) für beide Display-Profile einmal über die neue Pipeline (`calibration:"off"`, FS) und einmal über die wörtliche Legacy-Referenz (Palette aus `DisplayConfig.Colors` + `draw.FloydSteinberg.Draw`, als Inline-Snippet im Test, NICHT über die neue Pipeline) — Pixel- und Palette-identisch.
   *Beweis:* Test grün; als Negativprobe (manuell, nicht committen) muss er rot werden, wenn `"off"` fälschlich die Panel-Palette nutzt.

3. **AC3 — Treiber-Palette-Exaktheit über die gesamte Matrix:** `TestPaletteExactness` wird erweitert: (a) bestehende Mechanik (FS × `default`, 3 Qualitäten × beide Profile) läuft neu über die Designs {`gradient`, `calibration`}; (b) neue Kombinationen {`atkinson`×`default`, `floyd_steinberg`×`off`, `atkinson`×`off`} × `high` × beide Profile × `gradient`. Jeder unique RGB-Wert jedes Outputs ist Mitglied der **Ideal-Treiber-Palette** (die perzeptuellen Werte dürfen NIE ins Output leaken). Zusätzlich asserted der Test, dass das dekodierte PNG `*image.Paletted` ist und dessen Palette in Reihenfolge und Länge exakt `DisplayConfig.Colors` entspricht.
   *Beweis:* erweiterte Subtests grün; Fail listet fremde RGB-Werte mit Pixelanzahl (bestehendes Muster).

4. **AC4 — Kalibrierung wirkt genau dort, wo sie soll:** Für das `gradient`-Design bei `render_quality: high` gilt: (a) epd7in3e: Render mit `calibration:"default"` ≠ Render mit `"off"` (Bytes); (b) waveshare_7in5_v2: `"default"` == `"off"` byte-identisch (Identitäts-Profil); (c) alle `*__waveshare_7in5_v2.png` Golden Files sind im Commit unverändert (`git diff --stat`).
   *Beweis:* neuer Test `TestCalibrationAffectsOnlyCalibratedProfiles` grün; Golden-Suite für 7in5_v2 ohne `-update` grün.

5. **AC5 — Atkinson korrekt und messbar verschieden:** (a) Ein tabellengetriebener Unit-Test prüft `ditherAtkinson` auf einem kleinen synthetischen Input (z. B. 6×4 Pixel `#808080` gegen die S/W-Palette) gegen eine **von Hand nach der Definition in „Architektur-Richtung" Punkt 3 vorgerechnete** Index-Matrix (im Test hart kodiert, nicht aus der Implementierung generiert) — jede Änderung an Gewichten, Nachbarn, Scan-Reihenfolge oder Tie-Break macht ihn rot. (b) Für das `gradient`-Design (beide Profile, `high`, `calibration:"default"`) gilt: Atkinson-Output ≠ FS-Output (Bytes), beide palette-exakt (via AC3). (c) Determinismus: `TestRenderDeterminism`-Mechanik (warme + kalte Instanz, byte-identisch) läuft zusätzlich einmal mit `atkinson` auf `gradient`.
   *Beweis:* `TestAtkinsonDiffusionMatrix`, `TestDitherAlgorithmsProduceDifferentOutput`, Determinismus-Subtest grün.

## Non-Goals

- **Keine UI-Änderungen** — der Settings-UI-Toggle für `dither_algorithm`/`calibration` ist E3; bis dahin Konfiguration via `POST /update_settings` (curl).
- **Keine ICC-Profile / kein Farbmanagement** — die perzeptuelle Palette ist ein pragmatischer RGB-Satz, keine kolorimetrische Kalibrierung (Lab/ΔE bleibt außen vor).
- **Keine Per-Element-Kalibrierung** — die Pipeline wirkt global auf das fertige Komposit; Element-Properties bleiben unangetastet.
- **Kein Waveshare-Treiber-Fork, keine Client-Änderung** — das Übertragungsformat (PNG mit exakt den 6 Treiber-Farben) ist unverändert.
- **Keine Serpentinen-Scans, keine weiteren Algorithmen** (Bayer/Ordered, Jarvis, Stucki, Sierra, Blue-Noise …) — nur `floyd_steinberg` und `atkinson`; die Architektur (Algorithmus hinter typisierter Settings-Konstante) lässt spätere Erweiterung zu.
- **Keine simulierten Zusatzfarben** (13-Farben-Checkerboard-Tricks à la myembeddedstuff) — Palette bleibt 6 Einträge.
- **Keine Kalibrierwerte für waveshare_7in5_v2** — S/W-Profil bleibt Identität (bindend); wer dort später kalibrieren will, trägt nur Werte in `CalibrationProfiles` ein.
- **Keine automatische Messschleife** — es gibt keine Kamera am Test-Pi; optisches Tuning ist explizit manueller Scope (Gate L3).

### Gate L3 — Hardware (Sichtprüfung durch Kilian, nachgelagert)

Es gibt keine Kamera am Test-Pi — dieser Schritt ist manuell und entscheidet über das Tuning der Startwerte (nicht über den Merge; die maschinellen Gates L1/L5 sind dafür maßgeblich).

```sh
SERVER=http://<pi>:<port>

# 1. Kalibrier-Testbild importieren (einmalig) und aktivieren
curl -s -X POST $SERVER/api/designs -H 'Content-Type: application/json' \
  --data-binary @server/internal/services/testdata/designs/calibration.json
# → "id" aus der Antwort verwenden:
curl -s -X POST $SERVER/api/designs/<id>/activate
curl -s -X POST $SERVER/api/trigger_refresh

# 2. A/B-Vergleiche: Settings umschalten, jeweils neu refreshen
curl -s -X POST $SERVER/update_settings -H 'Content-Type: application/json' -d '{"calibration":"off"}'
curl -s -X POST $SERVER/api/trigger_refresh
curl -s -X POST $SERVER/update_settings -H 'Content-Type: application/json' -d '{"calibration":"default"}'
curl -s -X POST $SERVER/update_settings -H 'Content-Type: application/json' -d '{"dither_algorithm":"atkinson"}'
curl -s -X POST $SERVER/api/trigger_refresh
```

Bewertungskriterien am Panel: Grauverlauf neutral (kein Farbstich in den Stufen), Hauttöne plausibel (nicht ziegelrot/grünlich), Gelb-Felder leuchtend statt oliv, Textmatrix überall lesbar. Kontrolle, was tatsächlich gesendet wurde: `/tmp/eink_last_sent.png` (E1.2) vom Pi ziehen. Ergebnis des Vergleichs (welches Preset/welcher Algorithmus gewinnt, ggf. angepasste `CalibrationProfiles`-Werte) fließt als Folge-Commit mit Golden-`-update` ein.

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Startwerte der Panel-Palette treffen Kilians Panel nicht exakt (Panel-Streuung, Community-Werte statt eigener Messung) | Erwartbar und eingeplant: Werte sind als „initial guess, tunable" markiert, leben als Konstanten in `models/calibration.go` und sind per Golden-`-update`-Commit änderbar; Kalibrier-Testbild + A/B-Workflow (Gate L3) existieren genau dafür. Schlimmster Fall: `{"calibration":"off"}` stellt das heutige Bild sofort wieder her. |
| Golden-Churn: alle epd7in3e-Goldens ändern sich in diesem Task | Bewusst (Manager-Vorgabe), im selben Commit, mit Sichtprüfung im MR. 7in5_v2-Goldens sind byte-fixiert (AC4c) und dienen als eingebaute Kontrolle gegen versehentliche Pipeline-Regressionen. |
| Kollision mit E1.5 (beide Tasks fassen `preview.go`-Renderende bzw. Golden Files an) | Merge-Reihenfolge abstimmen; wer zweiter ist, rebased und regeneriert die epd7in3e-Goldens (`rotation__waveshare_7in3_e.png` fällt dann mit unter AC6). Vor Beginn `git pull --rebase origin main`. |
| Fehlerdiffusion gegen Palette mit Weiß < #FFFFFF erzeugt Rauschen in Weißflächen | Durch die Weißpunkt-adaptierte Default-Palette (Weiß `#E8E8E8`) begrenzt: reines Quellen-Weiß clampt in der Diffusion und bleibt stabil auf dem Weiß-Index; das Kalibrier-Testbild enthält große Weißflächen, sodass Restrauschen in der Golden-Sichtprüfung und am Panel sofort auffiele. |
| RAM/Performance auf dem Pi (Precomp-Pass + eigener Ditherer) | LUT statt `math.Pow`, Ganzzahl-Sättigung, 29-KB-Fehlerpuffer, alles auf dem 800×480-Bild statt dem supersampled Canvas; Budget „Quantisierung max. ~2×" wird per Benchmark belegt (AC8). Server-Budget ~10 MB bleibt unberührt. |
| Nutzer wundert sich über verändertes Panel-Bild nach Update (Default AN) | CHANGELOG-Eintrag (AC9) + dokumentierter Fluchtweg `{"calibration":"off"}`; Settings-API meldet den aktiven Zustand in `GET /settings`. |
| Atkinson verwirft 25 % des Fehlers — auf stark texturierten Fotos kann das Detail in Schatten/Lichtern kosten | Bekannte Algorithmus-Eigenschaft, kein Bug; FS bleibt Default, Atkinson ist Opt-in. A/B am Panel entscheidet den Geschmack (Gate L3). |
| **Rollback** | Ein Server-Commit (models + services + handlers + testdata), keine persistenten Formatänderungen (neue settings.json-Felder werden von altem Code ignoriert), keine Client-/API-Bruchstellen — `git revert <commit>` stellt den Vorzustand inkl. alter Goldens her. |
