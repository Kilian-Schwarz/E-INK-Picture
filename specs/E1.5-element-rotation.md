**Status:** Draft
**Abhängigkeiten:** E1.1 (gemergt) — Golden-File-Harness wird um ein Rotations-Design erweitert; deren Konventionen (Font-Pinning, `-update`-Disziplin, deterministische Elemente) gelten unverändert

**Zuständig:** render-quality — Go-Renderer + Tests + `testdata/`. **KEIN Frontend-Anteil nötig:** Verifiziert, das Frontend verwirft Rotation NICHT (s. Kontext) — der im Task-Briefing vorbehaltene storage.js-Fix mit Zweit-Agent frontend-designer entfällt.

## Ziel

Der Go-Renderer (`PreviewService.Render`) wertet `Element.Rotation` aus und rendert rotierte Elemente exakt mit der Semantik des Fabric.js-Designers (Winkel in Grad, im Uhrzeigersinn, um den Top-Left-Anker des Elements). Heute liest der Renderer `Rotation` nirgends — rotierte Elemente erscheinen auf dem Panel (und in der server-gerenderten Live-Preview des Designers) unrotiert: WYSIWYG-Bruch.

### Verifizierte Rotations-Semantik des Frontends (bindender Kontext)

Fabric.js 5.3.1 (CDN-Pin in `server/templates/designer.html:403`).

1. **Rotation wird gespeichert:** `storage.js` `canvasToDesignJSON()` Z. 94: `rotation: Math.round(obj.angle || 0)` — Fabric-`angle` in **Grad**, auf ganze Zahl gerundet (das Modell ist trotzdem `float64`; der Renderer muss beliebige Floats akzeptieren, auch negative — die Properties-Panel-Eingabe `parseFloat` erlaubt z. B. `-45`).
2. **Rotation wird beim Laden restauriert:** `storage.js` Z. 171: `obj.set('angle', elem.rotation)`. Properties-Panel (`properties-panel.js` Z. 36/49/95) bindet das Feld `prop-rotation` direkt an `obj.angle`. **Das Frontend verwirft Rotation also NICHT** — der Roundtrip Designer→JSON→Designer ist intakt; nur der Go-Renderer ignoriert das Feld (grep `Rotation` über `server/`: einzige Treffer sind die Modell-Definition `models/design.go:120` und die V1-Migration `services/design.go:213`, die `Rotation: 0` setzt).
3. **Ursprung/Pivot:** `element-factory.js` setzt auf keinem Top-Level-Objekt `originX`/`originY` (Textbox, Rect, Image, Widget-Group) → Fabric-Default `originX: 'left'`, `originY: 'top'`. Damit gilt für den persistierten Zustand: **(x, y) ist die Canvas-Position des unrotierten Top-Left-Ankers; das Element rotiert um `rotation` Grad im Uhrzeigersinn um genau diesen Punkt (x, y).** Fabrics `centeredRotation: true` betrifft nur das interaktive Drehen am Handle (Fabric rechnet dabei `left/top` so um, dass visuell um das Zentrum gedreht wird) — der gespeicherte Zustand `(left, top, angle)` ist immer vollständig durch die Top-Left-Anker-Regel beschrieben.
4. **width/height sind unrotiert:** `getScaledWidth()`/`getScaledHeight()` (storage.js Z. 79–80) liefern die skalierten Objektmaße, NICHT die rotierte Bounding-Box.
5. **Live-Previews des Designers sind server-gerendert und zeigen den Bug:** Der Preview-Modal (`toolbar.js` Z. 134–154, `POST /api/preview_live?raw=true`) und die Mini-LivePreview (`designer.js` Z. 3 ff.) gehen beide durch `PreviewService.Render` → rotierte Elemente erscheinen dort heute unrotiert. Nur die Fabric-Canvas selbst zeigt Rotation korrekt. E1.5 repariert damit automatisch auch beide Server-Previews.

### Verifizierte Fakten zum Renderpfad

6. **Renderfluss:** Supersampled RGBA-Canvas (`scale` = 2.0/1.5/1.0 je `render_quality`, Z. 96–101) → Element-Loop zeichnet direkt in `img *image.RGBA` (alle Typen via `renderTextV`/`renderImageElement`/`renderShapeElement` mit `(img, x, y, w, h)`) → CatmullRom-Downscale (Z. 222–228) → `quantizeToPalette` mit Floyd-Steinberg (Z. 230–236). Eine Rotation, die in den supersampled Canvas komponiert wird, liegt damit **automatisch VOR der finalen Quantisierung** — Anti-Aliasing der rotierten Kanten wird mitgedithert (bindende Manager-Vorgabe).
7. **Koordinaten-Skalierung:** `x := int(elem.X * scale)` usw. (Z. 124–127) — der Rotations-Anker muss dieselben int-truncierten, skalierten Werte nutzen wie der Direktpfad, sonst weicht Rotation 0° von quasi-0° ab.
8. **Culling ist rotations-blind:** Z. 129 `if x+w < 0 || x > renderW || y+h < 0 || y > renderH { continue }` prüft die unrotierte Box. Ein Element, dessen unrotierte Box off-canvas liegt, dessen rotierter Körper aber sichtbar wäre, würde fälschlich übersprungen — muss auf die rotierte AABB umgestellt werden (AC6).
9. **`golang.org/x/image/draw` ist bereits Dependency** (`xdraw`-Import Z. 23); `Kernel.Transform` + `f64.Aff3` (`golang.org/x/image/math/f64`) sind ohne `go.mod`-Änderung verfügbar.

### Bindende Implementierungs-Richtung (Manager)

- **Rotation ≠ 0 (mod 360):** Element in einen Offscreen-RGBA-Puffer der Größe w×h (skaliert) rendern — Zeichenursprung (0, 0), Widgets mit ihrem px/py-Padding innerhalb des Puffers —, dann per affiner Transformation in den supersampled Canvas kompositen: `xdraw.CatmullRom.Transform(img, aff, offscreen, offscreen.Bounds(), xdraw.Over, nil)`. Der Puffer ist außerhalb gezeichneter Pixel transparent; `Over` komponiert korrekt.
- **Matrix (legt die Fabric-Semantik exakt fest):** θ = Rotation·π/180; in Bildkoordinaten (y nach unten) ist „im Uhrzeigersinn" die Standard-Rotationsmatrix. Offscreen-Punkt (u, v) → Canvas:
  `dstX = x + u·cosθ − v·sinθ`, `dstY = y + u·sinθ + v·cosθ`
  also `f64.Aff3{cosθ, −sinθ, float64(x), sinθ, cosθ, float64(y)}` mit x, y aus Kontext Punkt 7.
- **Exakte 90°-Vielfache:** Für `mod(rotation, 360) ∈ {90, 180, 270}` (nach Normalisierung negativer Winkel) die Koeffizienten exakt aus {−1, 0, 1} setzen statt `math.Sin/Cos` (dort ist z. B. `cos(π/2) ≈ 6.1e-17` ≠ 0) — sonst ist die Achsenparallelität von AC4 numerisch nicht haltbar.
- **Rotation == 0 (mod 360):** exakt der bestehende Direktpfad, KEIN Offscreen, KEIN Transform — Byte-Identität ist AC1. Offscreen-Puffer werden **nur bei Rotation ≠ 0 alloziert** (Pi-RAM).
- **Task-Schnitt:** Der Typ-Switch (Z. 177–219) wird so refaktoriert, dass jeder Element-Typ in ein übergebenes Ziel-`*image.RGBA` an einen übergebenen Ursprung zeichnet; der Rotations-Wrapper ist damit typ-agnostisch und gilt automatisch auch für alle `widget_*`-Typen (Golden-Abdeckung aber nur über die deterministischen Typen, s. Non-Goals).

## Akzeptanzkriterien

1. **AC1 — Rotation 0 ist byte-identisch zu vorher:** `TestGoldenRender` für `basic` und `gradient` bleibt **ohne** `-update` grün; die 4 bestehenden Golden-PNGs sind im Commit unverändert.
   *Beweis:* `go test ./internal/services -run TestGoldenRender -count=1` grün; `git diff --stat` zeigt keine Änderung unter `testdata/golden/basic__*` / `testdata/golden/gradient__*`.

2. **AC2 — Neues deterministisches Rotations-Design + Golden Files:** `testdata/designs/rotation.json` (Canvas 800×480, `models.DesignV2`) enthält mindestens: ein Text-Element mit `rotation: 30` (`fontFamily: "testfont.ttf"`, E1.1-Font-Pinning), ein Shape mit `rotation: 45` (Fill + Stroke), ein Image-Element mit `rotation: 90` (`gradient_photo.png` aus dem E1.1-Harness) — dazu mindestens ein unrotiertes Referenz-Element im selben Design. `"rotation"` wird in `goldenDesigns` aufgenommen; die 2 neuen Golden-PNGs (`rotation__waveshare_7in5_v2.png`, `rotation__waveshare_7in3_e.png`, `render_quality: high`) werden per `-update` erzeugt, gesichtet und **im selben Commit** eingecheckt.
   *Beweis:* `TestGoldenRender` läuft die neuen Subtests grün; Dateien im Commit.

3. **AC3 — Palette-Exaktheit gilt weiter (rotierte Kanten werden mitgedithert):** `TestPaletteExactness` wird von fix `"gradient"` auf die Design-Liste {`gradient`, `rotation`} ausgeweitet — alle 3 `render_quality`-Stufen × beide Display-Profile. Jeder unique RGB-Wert des Outputs ist Mitglied der Panel-Palette; das beweist, dass das Anti-Aliasing der rotierten Kanten vor der Quantisierung liegt und kein Nachquantisierungs-Artefakt entsteht.
   *Beweis:* erweiterte Subtests grün; Fail listet fremde RGB-Werte mit Pixelanzahl (bestehendes Muster).

4. **AC4 — 90°/180°/270° bleiben exakt achsenparallel (Eckpunkt-Assertion):** Neuer Test rendert je ein Design mit einem einzelnen schwarzen, achsenparallelen Rechteck (weißer Hintergrund, `render_quality: fast` ⇒ scale 1.0, pixel-exakt, kein Supersampling-Blur) mit Anker (x, y) = (400, 200), w = 200, h = 100 und `rotation ∈ {0, 90, 180, 270}`. Erwartete belegte Rechtecke (aus der Matrix in „Implementierungs-Richtung", Halboffen [min, max)):
   | rotation | x-Bereich | y-Bereich |
   |---|---|---|
   | 0 | [400, 600) | [200, 300) |
   | 90 | [300, 400) | [200, 400) |
   | 180 | [200, 400) | [100, 200) |
   | 270 | [400, 500) | [0, 200) |
   Assertion: **jeder** Pixel innerhalb des erwarteten Rechtecks ist schwarz, **jeder** Pixel außerhalb ist weiß — Toleranz 0 (möglich durch die exakten 90°-Koeffizienten und den integer-alignten Transform).
   *Beweis:* neuer Test (z. B. `TestRotationRightAnglesAxisParallel`) grün; mit `math.Sin/Cos` statt exakter Koeffizienten muss er rot werden (Negativprobe, s. Verifikation).

5. **AC5 — Determinismus deckt Rotation ab:** `TestRenderDeterminism` iteriert `goldenDesigns` und erfasst das Rotations-Design damit automatisch (warme/kalte Instanz, byte-identisch).
   *Beweis:* Subtests `rotation__*` grün.

6. **AC6 — Culling nutzt die rotierte AABB:** Ein Element, dessen unrotierte Box vollständig off-canvas liegt, dessen rotierter Körper aber in den Canvas ragt, wird gerendert. Testfall (scale 1.0): Shape mit x = 50, y = −60, w = 200, h = 40, `rotation: 90`, Fill `#000000` — unrotiert gilt `y+h = −20 < 0` (heutiges Culling würde überspringen), rotiert belegt es [10, 50) × [0, 140). Assertion: schwarze Pixel in diesem Bereich vorhanden. Gegenprobe: ein auch nach Rotation vollständig off-canvas liegendes Element liefert ein Rendering, das byte-identisch zum leeren Canvas ist (Skip-Pfad funktioniert weiter).
   *Beweis:* neuer Test (z. B. `TestRotatedElementCulling`) grün.

7. **AC7 — Suite + Statik:** `go test ./... -count=1` (in `server/`) komplett grün und unter dem 60-s-Budget aus E1.1; `gofmt -l .` leer; `go vet ./...` sauber. Der Diff berührt außer `preview.go` keinen Produktivcode (insbesondere kein `models/`, kein `go.mod`).
   *Beweis:* Kommandos unter „Verifikation"; `git diff --stat`.

## Non-Goals

- **Keine Frontend-Änderungen.** Explizites Discovery-Ergebnis: `storage.js` speichert und restauriert `rotation` bereits korrekt (Kontext Punkte 1–3) — der im Auftrag vorbehaltene storage.js-Fix samt Zweit-Agent frontend-designer ist NICHT nötig und entfällt.
- **Keine Golden Files für rotierte `widget_*`-Typen** — Live-Daten-Widgets sind nicht deterministisch (E1.1-Konvention). Der Rotations-Codepfad ist typ-agnostisch und gilt für Widgets mit; abgedeckt wird er über text/shape/image.
- **Kein rotations-bewusster Text-Umbruch:** Text wird wie im Designer in der unrotierten w×h-Box layoutet/geclippt und dann als Ganzes rotiert (Fabric rotiert den clipPath mit) — kein Re-Layout entlang der Rotation.
- **Keine Interaktion mit Conditions/propertyChanges** — Rotation wird gelesen wie x/y/w/h; bedingte Rotations-Änderungen sind kein Teil dieses Tasks.
- **Keine V1-Design-Rotation** — die V1-Migration setzt `Rotation: 0` (design.go Z. 213), das bleibt so.
- Keine zusätzlichen Golden Files für `medium`/`fast` (AC3 deckt diese Stufen über die Palette-Assertion ab — E1.1-Konvention).
- Keine Performance-Optimierung über „Offscreen nur bei Rotation ≠ 0" hinaus (kein Puffer-Pooling etc.).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Text-Bounding-Box vs. Rotation (Clipping): Glyphen, die die w×h-Box überragen, werden vor der Rotation abgeschnitten | Entspricht dem Designer-Verhalten (Fabric-clipPath rotiert mit dem Objekt); die bekannte, rotations-unabhängige Layout-Differenz Renderer↔Fabric (Zeilenumbruch, Kerning) wird weder besser noch schlechter. Golden Files frieren den Ist-Stand ein. |
| Performance/RAM auf dem Pi: Offscreen-Puffer pro rotiertem Element, bei scale 2.0 bis zu 1600×960×4 ≈ 6 MB für ein vollflächiges Element, plus Transform-Kosten | Allokation NUR bei Rotation ≠ 0 (bindend); typische rotierte Elemente sind klein; Puffer ist nach dem Kompositen sofort freigebbar (GC). Server-Budget ~10 MB bleibt im Normalfall unberührt. Bei Bedarf `render_quality: fast/medium` als Betreiber-Stellschraube. |
| Float-Staub bei 90°-Vielfachen bricht Achsenparallelität | Durch exakte Koeffizienten {−1, 0, 1} ausgeschlossen; AC4 erzwingt es mit Toleranz 0. |
| Anti-Aliasing der rotierten Kanten erzeugt Zwischentöne | Gewollt — sie entstehen VOR der Quantisierung und werden von Floyd-Steinberg mitgedithert; AC3 beweist die Palette-Exaktheit für alle Qualitätsstufen. |
| CatmullRom-`Transform`-Bytes ändern sich mit `golang.org/x/image`-Bump | Bekanntes E1.1-Risiko, gleiche Mitigation: Golden-Suite wird rot, `-update` im selben Commit wie der Bump, mit Sichtprüfung. |
| Kollision mit parallelen Renderer-Tasks (Golden Files sind global für `preview.go`) | AC1 (Byte-Identität bei Rotation 0) minimiert die Angriffsfläche: Bestands-Goldens bleiben unverändert, Merge-Konflikte beschränken sich auf `goldenDesigns`/neue Dateien. Vor Beginn `git pull --rebase origin main`. |
| Negative/übergroße Winkel aus API-Clients (Frontend rundet, API validiert nicht) | Normalisierung `mod 360` (+360 bei negativem Rest) im Renderer; von AC4 (270 ≡ −90) implizit mitgetestet, Review-Punkt in L5. |
| **Rollback** | Ein Server-Commit (preview.go + Tests + 2 neue testdata-PNGs), keine persistenten Formate, keine API-Änderungen — `git revert <commit>` stellt den Vorzustand her; Bestands-Goldens waren nie angefasst. |
