# E5.2: Panel-Schonung — Content-Skip & Nacht-/Sleep-Modus (kombiniert E5.2 + E5.3)

**Status:** Draft
**Abhängigkeiten:** E1.1 (gemergt) — der byte-exakte Render-Determinismus (`TestRenderDeterminism`, Golden-Suite) ist die Geschäftsgrundlage des Content-Skips: identischer Input ⇒ identische PNG-Bytes. E1.2 (gemergt) — MockEPD-/Test-Muster im Client, Last-Sent-Artefakt-Semantik. E5.1 (Auth) — nur Kontext: die Client-Endpoints sind ggf. tokenpflichtig (`X-Client-Token`), ändert an diesem Task nichts.
**Zuständig:** GETEILT, zwei Phasen. **Phase 1: go-backend** (Settings, Fensterlogik, `reason`-Feld). **Phase 2: pi-client** (Content-Skip, Forced-Handling, 24h-Guard). Der Schnittstellen-Kontrakt (s. u.) ist so geschnitten, dass beide Phasen unabhängig mergebar sind: der Client behandelt ein fehlendes `reason`-Feld konservativ als „forced" (kein Skip) — alte Server + neuer Client verhalten sich exakt wie heute.
**Koordination:** Wer `services/settings.go`/`handlers/settings.go` parallel anfasst (z. B. Settings-UI-Arbeit): Merge-Reihenfolge abstimmen. `client/client.py` wird nach E1.2/E1.4-Mustern erweitert.

## Ziel

Das Panel wird nur noch angefasst, wenn es etwas Neues zu zeigen gibt — und nachts gar nicht:

1. **E5.3 Nachtfenster (Server):** Settings `sleep_start`/`sleep_end` (`"HH:MM"`, leer = aus). Im Fenster unterdrückt der Server Intervall-Refreshes (`should_refresh=false`); ein **manueller Trigger durchbricht das Fenster** (explizite Nutzeraktion gewinnt — bindende Entscheidung). Der Client braucht dafür KEINE Änderung.
2. **E5.2 Content-Skip (Client):** Der Client hasht die PNG-Bytes von `GET /preview` (SHA-256). Byte-identisch zum zuletzt aufs Panel geschriebenen Bild ⇒ Panel-Write wird übersprungen (kein `init`/`display`/`sleep` — Panel bleibt im Deep Sleep), Heartbeat wird trotzdem gesendet. **Messziel: statisches Design (ohne Uhr-Widget) ⇒ 0 Panel-Refreshes bei unverändertem Inhalt.**
3. **Panel-Care-Leitplanken:** 24h-Guard (spätestens nach `EINK_MAX_SKIP_HOURS` wird trotz identischem Inhalt geschrieben — Waveshare empfiehlt mind. 1 Refresh/24 h) und Kill-Switch `EINK_CONTENT_SKIP=false`.

**Was dieser Task bewusst NICHT baut:** `full_refresh_every_n` mit Partial-Modus für epd7in5_V2 — die Treiber-Recherche gibt das nicht sauber her (Befund 3), dokumentiertes Follow-up statt Bau auf Verdacht.

**Ehrliche Nutzenrechnung:** Der Pi läuft und pollt weiter (alle 30 s, vernachlässigbar); gespart werden **Panel-Zyklen** (HV-Stress, Ghosting-Aufbau, Lebensdauer) und der Refresh-Strom des Panels (~25 s Aktivphase beim epd7in3e pro Full-Refresh), nicht der Pi-Standby-Verbrauch. Nachtfenster 23–06 Uhr bei Default-Intervall 3600 s ≈ 7 vermiedene Refreshes/Nacht plus kein nächtliches Flackern; Content-Skip vermeidet bei statischen Designs ~24 Refreshes/Tag.

### Verifizierte Fakten zum Ist-Zustand

1. **Der Server entscheidet über Refreshes, der Client ist dumm:** `GetRefreshStatus` (services/settings.go Z. 206–252) liefert `should_refresh=true`, wenn (a) `last_refresh_trigger` neuer als `last_client_refresh` ist ODER (b) das Intervall seit `last_client_refresh` abgelaufen ist (Prüfreihenfolge: Trigger zuerst). Der Client pollt alle `EINK_POLL_INTERVAL` s (Default 30) `GET /api/refresh_status`, holt bei `true` das PNG von `GET /preview`, schreibt es aufs Panel und meldet `POST /api/client_heartbeat` (client.py `main`-Loop Z. 290–307). **Der natürliche Ort für das Nachtfenster ist damit der Server** — geprüft und festgelegt: eine Client-Lösung müsste Serverzeit/Zeitzone duplizieren und würde beim manuellen Trigger kollidieren.
2. **`refresh_status` rendert NICHTS:** `GetRefreshStatus` liest nur Timestamps aus `settings.json`. Gerendert wird ausschließlich bei `GET /preview` / `POST /api/preview_live` (handlers/preview.go), hinter der Render-Semaphore N=1 (E5.6), ~4,1 s pro Render auf dem Pi. **Konsequenz für das Hash-Design:** Ein server-seitiger Content-Hash müsste im Poll-Pfad rendern — alle 30 s ein 4-s-Render wäre absurd; ein Hash über Design-JSON + Widget-Daten wäre billiger, aber lückenhaft (Settings wie `render_quality`/`dither_algorithm`/`calibration` und Font-/Code-Stände fließen ebenfalls ins Bild ein) und damit false-negative-anfällig ⇒ eingefrorenes Panel. Ein Render-Cache wäre ein eigenes Projekt. **Entscheidung: Client-seitig hashen** — der Client lädt das PNG bei `should_refresh=true` ohnehin; SHA-256 über die Wire-Bytes kostet Mikrosekunden und hat per Konstruktion keine semantische Lücke.
3. **Treiber-Fähigkeiten am gepinnten Commit `500fa7c6` (waveshareteam/e-Paper, Pin aus setup.sh Z. 34), via Quelltext verifiziert:**
   - **epd7in3e (6-Farb):** NUR Full-Refresh — öffentliche Methoden `init`, `getbuffer`, `display`, `Clear`, `sleep` (+ Low-Level). Kein `display_Partial`, kein `init_fast`. Ein Full-Refresh dauert ~25 s. ⇒ „jeder 10. Refresh full, sonst schneller Modus" ist für dieses Panel **prinzipiell nicht umsetzbar**; E5.2 reduziert sich hier auf „unnötige Refreshes vermeiden" = Content-Skip.
   - **epd7in5_V2 (B/W):** hat `init_fast()`, `init_part()` und `display_Partial(Image, Xstart, Ystart, Xend, Yend)`. ABER: `display_Partial` schreibt nur RAM `0x13` (neues Bild) und diff-t gegen den alten RAM-Inhalt; `sleep()` macht POWER_OFF + DEEP_SLEEP (`0x07 0xA5`) + `module_exit()` — der Controller-RAM ist danach weg. Unser Client-Lebenszyklus ist aber `init → display → sleep` pro Refresh (client.py Z. 163–200), und Waveshare-Precautions verlangen genau das („screen must be set to sleep mode when not refreshing", sonst HV-Schaden). Partial-Refresh bräuchte also einen umgebauten Lebenszyklus (Basisbild-Rewrite nach jedem Aufwachen oder Wachhalten zwischen Partials) plus Ghosting-Management (Waveshare: Global-Refresh nach ~5 Partials, sonst „residual image … even damage the screen"). **Nicht sauber ohne Hardware-Iteration ⇒ Follow-up (s. Non-Goals), nicht Teil dieses Tasks.**
   - Weitere Waveshare-Precautions (Panel-Care-Grundlage): Refresh-Intervall mind. 180 s empfohlen; **mindestens 1 Refresh pro 24 h** — Begründung für den 24h-Guard.
4. **Render-Determinismus ist bewiesen:** Die Golden-Suite (E1.1) prüft byte-identische PNG-Ausgabe (`TestGoldenRender`, `TestRenderDeterminism`). Identischer Input (Design + Widget-Daten + Settings) ⇒ identische Bytes; Go's PNG-Encoder bettet keine Timestamps ein. Ein Byte-Hash-Vergleich hat damit keine falschen Skips: Inhalt geändert ⇒ Bytes geändert ⇒ Refresh. Umgekehrt lösen echte Datenänderungen (Wetter, Kalender) korrekt einen Refresh aus — das ist gewollt.
5. **Settings-Muster:** Partial-Update in `UpdateSettings` (handlers/settings.go) — Enums via `!= ""`-Guard, löschbare/nullbare Werte via Pointer (`RefreshInterval *int`, Z. 37). Neue String-Felder, die auf `""` (= aus) zurückgesetzt werden können, MÜSSEN dem Pointer-Muster folgen (`*string`: `nil` = nicht angefasst, `""` = löschen). `Settings` wird mit `omitempty` persistiert — alte `settings.json` ohne die neuen Felder bleiben gültig (= Fenster aus).
6. **Zeitbasis:** Server-Timestamps sind UTC/RFC3339 (`LastRefreshTrigger`, `LastClientRefresh`). Das Nachtfenster ist dagegen Wanduhrzeit: verglichen wird gegen die **lokale Serverzeit** (`time.Now()` → lokale TZ). Docker reicht `TZ` durch (docker-compose.yml Z. 24, Default `Europe/Berlin`); nativ gilt die System-TZ des Pi. `GetRefreshStatus` nutzt heute `time.Now`/`time.Since` direkt — **für die Fenster-Tests muss eine injizierbare Clock rein** (Feld `now func() time.Time` am `SettingsService`, Default `time.Now`; Tests setzen das Feld package-intern).
7. **Heartbeat-Body wird heute ignoriert:** `ClientHeartbeat` dekodiert `status`, verwendet es nicht (handlers/settings.go Z. 137–149). Der Client sendet `status: "refreshed"` (client.py Z. 224–236). Additiv nutzbar für Skip-Diagnose.
8. **Client-Startup schreibt immer:** Nach dem Start macht der Client ein unkonditionales Initial-Display (client.py Z. 277–284) — unabhängig von `refresh_status`. Bleibt unverändert (Non-Goal); Konsequenz: ein Client-Neustart in der Nacht schreibt das Panel einmal. Dokumentiert, akzeptiert.
9. **Test-Infrastruktur vorhanden:** Server-Tests nutzen `t.TempDir()`-basierte Services + httptest (settings_test.go beide Ebenen); Client-Tests nutzen `unittest` mit `MockEPD`/`RecordingEPD` und gepatchtem `client.config` (test_client.py). Die Fenster-Grenzfälle brauchen die injizierte Clock (Fakt 6), die Client-Zeit-Tests patchen `time.monotonic` im Client-Modul.

### Bindende Architektur-Entscheidungen

1. **Nachtfenster im Server, NUR im Intervall-Zweig.** In `GetRefreshStatus` wird der Intervall-Check übersprungen, wenn die lokale Serverzeit im Fenster liegt. Der Trigger-Zweig (manuell) bleibt unangetastet ⇒ **manueller Trigger durchbricht das Fenster** — wer nachts auf „Refresh" drückt, meint das so. Das ist zugleich der Fluchtweg gegen ein „eingefrorenes" Panel.
2. **Fenster-Semantik:** `sleep_start`/`sleep_end` als `"HH:MM"` (24 h). Minuten seit Mitternacht `m`, `s`, `e`: nicht-wrappend (`s < e`): unterdrückt wenn `s <= m < e`; wrappend über Mitternacht (`s > e`, z. B. 23:00–06:00): unterdrückt wenn `m >= s || m < e`. Halboffen `[start, end)`: um `sleep_end` ist der erste Refresh wieder erlaubt. `s == e` wird beim Schreiben abgelehnt (mehrdeutig); beide leer = aus.
3. **Erststart-Ausnahme:** Ist `last_client_refresh` leer (Client hat noch nie refresht), greift das Fenster NICHT — ein fabrikneues, leeres Panel die ganze Nacht leer zu lassen wäre falsch. Fail-open Richtung „Inhalt zeigen".
4. **Defensives Laden:** Ungültige Fensterwerte in `settings.json` (hand-editiert, z. B. `"25:99"` oder nur ein Feld gesetzt) ⇒ Fenster gilt als aus + `slog.Warn`. Fail-open Richtung „Refresh findet statt", niemals Richtung „Panel eingefroren".
5. **`reason`-Feld im `refresh_status`:** `"manual"` (Trigger-Zweig) | `"interval"` (Intervall-Zweig), `omitempty` — nur gesetzt wenn `should_refresh=true`. Präzedenz wie heutige Prüfreihenfolge: Trigger zuerst ⇒ bei beidem gilt `"manual"`.
6. **Content-Skip client-seitig, konservativ:** SHA-256 über die **rohen HTTP-Body-Bytes** von `GET /preview` (vor jedem Pillow-Decode — billigster und verlustfreiester Vergleichspunkt, Determinismus per Fakt 4). Skip NUR wenn ALLE Bedingungen gelten: `CONTENT_SKIP` aktiv ∧ `reason == "interval"` ∧ Hash identisch zum letzten **erfolgreich** aufs Panel geschriebenen Bild ∧ 24h-Guard nicht abgelaufen ∧ Hardware vorhanden (`epd is not None`; der No-Hardware-/Preview-Pfad skippt nie). Der Hash-Zustand lebt **nur im Prozess-Speicher** (kein Persistieren): Neustart ⇒ erster Frame wird immer geschrieben. Hash wird erst NACH erfolgreichem `display_image(...) == True` aktualisiert; ein Display-Fehler lässt ihn unverändert (nächster Zyklus schreibt erneut).
7. **Forced-Semantik & Versions-Skew:** `reason == "manual"` ⇒ Panel-Write auch bei identischem Hash (der Nutzer will einen Refresh, z. B. gegen Ghosting). **Fehlt das `reason`-Feld** (alter Server, Phase 1 noch nicht deployt) ⇒ wie `"manual"` behandeln = nie skippen = exakt heutiges Verhalten. Dadurch sind Phase 1 und 2 in beliebiger Reihenfolge deploybar.
8. **24h-Guard:** Liegt der letzte tatsächliche Panel-Write länger als `EINK_MAX_SKIP_HOURS` (Default 24, `0` = aus) zurück, wird trotz identischem Hash geschrieben (Waveshare-Empfehlung, Fakt 3; zugleich Sicherheitsnetz gegen jeden unvorhergesehenen Hash-Fehlschluss). Gemessen mit `time.monotonic()` (immun gegen Wanduhr-Sprünge). Der Guard erzeugt KEINE Refreshes im Nachtfenster — er wirkt nur am Skip-Entscheidungspunkt, d. h. wenn der Server ohnehin `should_refresh=true` geliefert hat.
9. **Heartbeat auf Skip:** Der Client heartbeatet auch beim Skip (`status: "skipped"` statt `"refreshed"`), sonst bliebe `should_refresh=true` und der Client würde das PNG bei jedem 30-s-Poll erneut laden. Server-Semantik bleibt gleich (beide Werte setzen `last_client_refresh`); der Wert wird nur geloggt. **Dokumentierte Bedeutungsverschiebung:** `last_client_refresh` misst ab jetzt „Content beim Client aktuell", nicht „Panel physisch beschrieben".
10. **Kill-Switch:** `EINK_CONTENT_SKIP=false` deaktiviert den Skip vollständig (Default: aktiv). Erste Verteidigungslinie, falls auf der Hardware irgendetwas Unerwartetes auftritt.

### Schnittstellen-Kontrakt (go-backend ↔ pi-client)

Phase 2 baut ausschließlich gegen diesen Kontrakt — keine weiteren Server-Annahmen.

**`GET /settings` und `POST /update_settings`** (Response-Erweiterung, Request-Erweiterung):

```jsonc
// POST /update_settings — Request (Ausschnitt; Felder optional, Pointer-Semantik)
{ "sleep_start": "23:00", "sleep_end": "06:00" }   // Fenster setzen (beide nötig)
{ "sleep_start": "", "sleep_end": "" }             // Fenster ausschalten
// Validierungsfehler => 400: Format != HH:MM, nur ein Feld resultierend gesetzt, start == end

// GET /settings — Response (Ausschnitt; Felder immer präsent, "" = aus)
{ "sleep_start": "23:00", "sleep_end": "06:00", "refresh_interval": 3600, ... }
```

**`GET /api/refresh_status`** (Response-Erweiterung):

```jsonc
{ "should_refresh": true,  "reason": "interval", "refresh_interval": 3600, ... } // Intervall abgelaufen
{ "should_refresh": true,  "reason": "manual",   ... }                           // manueller Trigger
{ "should_refresh": false, ... }                                                 // kein reason-Feld (omitempty)
```

**`POST /api/client_heartbeat`** (Body-Konvention, server-seitig semantikfrei):

```jsonc
{ "status": "refreshed", "timestamp": "..." }  // Panel wurde beschrieben
{ "status": "skipped",   "timestamp": "..." }  // Content identisch, Panel-Write übersprungen
```

## Akzeptanzkriterien

### Phase 1 — Server (go-backend)

1. **AC1 — Settings-Roundtrip + Validierung (httptest):** `POST /update_settings` mit `{"sleep_start":"23:00","sleep_end":"06:00"}` ⇒ 200, `GET /settings` liefert beide Felder; `{"sleep_start":"","sleep_end":""}` ⇒ Fenster aus (`""`/`""`). 400-Fälle: `"23:0"`, `"24:00"`, `"aa:bb"`, nur ein Feld resultierend gesetzt (auch: eines setzen, anderes leeren), `start == end`. Nicht mitgesendete Felder bleiben unverändert (Pointer-Semantik). Eine `settings.json` ohne die neuen Felder lädt fehlerfrei mit Fenster aus.
   *Beweis:* neue Tests in `handlers/settings_test.go`; Laden-Alt-Datei-Test in `services/settings_test.go`.

2. **AC2 — Fensterlogik mit injizierter Clock (Tabellen-Test):** `SettingsService` erhält `now func() time.Time` (Default `time.Now`); `GetRefreshStatus` nutzt ausschließlich `s.now()`. Tabellen-Test mit abgelaufenem Intervall (`last_client_refresh` alt) und Fenster 23:00–06:00: um 22:59 ⇒ `should_refresh=true`; um 23:00, 03:00, 05:59 ⇒ `false`; um 06:00 ⇒ `true` (halboffen `[start,end)`). Nicht-wrappendes Fenster 12:00–14:00: 11:59/14:00 ⇒ `true`, 12:00/13:59 ⇒ `false`. Fenster aus (`""`/`""`) ⇒ heutiges Verhalten. Erststart (`last_client_refresh == ""`) um 03:00 im Fenster ⇒ `true` (Entscheidung 3). Ungültige Werte auf Disk (`"25:99"`, nur ein Feld) ⇒ Fenster aus + Warnung (Entscheidung 4).

   *Beweis:* Kommandos unter „Verifikation".

### Phase 2 — Client (pi-client)

6. **AC6 — Skip-Pfad vollständig:** Zwei Poll-Zyklen mit byte-identischem PNG und `reason:"interval"` (gemockte `requests`): Zyklus 1 schreibt (MockEPD `init`+`display`+`sleep` gerufen, Heartbeat `status:"refreshed"`); Zyklus 2 skippt — **kein** `epd.init`, **kein** `epd.display`, **kein** `epd.sleep` (Panel bleibt im Deep Sleep), Heartbeat wird trotzdem gesendet mit `status:"skipped"`, das Last-Sent-Artefakt (E1.2) wird NICHT neu geschrieben (mtime/Inhalt unverändert), eine INFO-Logzeile enthält `skipping panel refresh (content unchanged)` (stabile Grep-Basis für HIL).
   *Beweis:* neuer Test in `test_client.py` mit `RecordingEPD` + gemocktem `_server_get`/`_server_post`.

7. **AC7 — Änderung erzwingt Write, Fehler friert nichts ein:** (a) Geänderte PNG-Bytes ⇒ `display` wird gerufen, Heartbeat `"refreshed"`. (b) Der Hash wird erst NACH `display_image(...) == True` übernommen: schlägt `epd.display` fehl (Exception im MockEPD), bleibt der alte Hash stehen und der NÄCHSTE Zyklus mit denselben Bytes schreibt erneut (kein Skip auf nie-gezeigten Inhalt). (c) Bei fehlgeschlagenem Display wird KEIN Heartbeat gesendet (heutiges Verhalten unverändert).
   *Beweis:* Tests `test_skip_hash_updated_only_after_success` u. ä.

8. **AC8 — Forced-Semantik & Versions-Skew:** `reason:"manual"` ⇒ Panel-Write trotz identischer Bytes. Antwort OHNE `reason`-Feld (alter Server) ⇒ Panel-Write trotz identischer Bytes (kein Skip, heutiges Verhalten).
   *Beweis:* zwei Tests mit identischen Bytes und variiertem `refresh_status`-Body.

9. **AC9 — 24h-Guard:** Mit gepatchtem `time.monotonic`: identische Bytes, `reason:"interval"`, letzter Panel-Write > `MAX_SKIP_HOURS` her ⇒ Write erfolgt; danach Guard-Uhr zurückgesetzt. `EINK_MAX_SKIP_HOURS=0` ⇒ Guard aus (Skip bleibt Skip).
   *Beweis:* Test mit `unittest.mock.patch` auf `client.time.monotonic`.

10. **AC10 — Kill-Switch, Neustart, No-Hardware:** (a) `CONTENT_SKIP=False` (gepatcht) ⇒ nie Skip, identische Bytes werden geschrieben. (b) Frischer Prozess-Zustand (Modul-Reset im Test) ⇒ erster Frame wird immer geschrieben (kein persistierter Hash). (c) `epd is None` (Preview-only) ⇒ Skip-Logik inaktiv, Verhalten wie heute.
    *Beweis:* drei Tests; (b) über explizites Zurücksetzen des Modul-Zustands im setUp.

11. **AC11 — Config-Hygiene:** `client/config.py` definiert `CONTENT_SKIP` (Default aktiv; nur der String `"false"`, case-insensitiv, deaktiviert) und `MAX_SKIP_HOURS` (int, Default 24) nach Modulebenen-Muster; `client/.env.example` UND Root-`.env.example` dokumentieren `EINK_CONTENT_SKIP` und `EINK_MAX_SKIP_HOURS` mit Kommentar (ohne echte Werte).
    *Beweis:* `grep -n EINK_CONTENT_SKIP client/.env.example .env.example` trifft beide; Config-Default-Tests analog `TestLastSentPathConfig`.

12. **AC12 — Bestand grün:** Alle vorhandenen Client-Tests laufen unverändert durch (insbesondere E1.2-Artefakt- und E1.4-Resize-Tests).
    *Beweis:* pytest-Lauf unter „Verifikation".

## Non-Goals

- **KEIN Partial-/Fast-Refresh für epd7in5_V2 (`full_refresh_every_n`)** — bewusst ausgeklammert (Fakt 3): `display_Partial` diff-t gegen Controller-RAM, den unser pflichtgemäßer Deep-Sleep pro Zyklus zerstört; Waveshare verlangt Global-Refresh nach ~5 Partials gegen Ghosting/Panelschaden. Ein Follow-up-Task bräuchte: umgebauten Client-Lebenszyklus (Basisbild-Rewrite nach Wake ODER Wachhalten zwischen Partials — gegen die Sleep-Precaution abzuwägen), Partial-Zähler mit erzwungenem Full-Refresh (`full_refresh_every_n`, Startwert ≤ 5), `init_fast()` als risikoärmere Zwischenstufe, und zwingend HIL-Iteration am echten Panel. Für epd7in3e ist Partial hardwareseitig ausgeschlossen — Content-Skip ist dort das Maximum.
- **Keine Settings-UI** — API-only; die UI-Erweiterung (Zeitfenster-Picker) ist ein Folge-Task (Muster: render_quality-Selector).
- **Kein Pi-Stromsparen / kein Poll-Stopp nachts** — der Client pollt unverändert alle 30 s; Ersparnis ist ehrlich: Panel-Zyklen und Panel-Refresh-Strom, nicht Pi-Verbrauch (Ziel s. o.).
- **Kein server-seitiger Content-Hash / Render-Cache** — bewertet und verworfen (Fakt 2): entweder Doppel-Render im Poll-Pfad oder lückenhafter semantischer Hash mit Frozen-Panel-Risiko.
- **Keine Änderung am Initial-Display beim Client-Start** (Fakt 8) und keine Persistenz des Hash über Neustarts (bewusst konservativ).
- **Kein Mindest-Intervall-Enforcement** (Waveshare empfiehlt ≥ 180 s; heutiger Server akzeptiert jedes `refresh_interval > 0`) — als Panel-Care-Hinweis in README dokumentieren, Validierung ist Follow-up.
- **Keine Änderung an Render-Pfad, Golden-Files, Frontend** (`server/static/` bleibt unberührt).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| **Uhr-/Minuten-Widgets machen den Content-Skip wirkungslos** (jede Minute andere Bytes) | Systembedingt korrekt — das Bild ändert sich ja wirklich. Ehrlich dokumentiert: für Uhr-Designs ist das **Nachtfenster der Haupthebel** (E5.3), nicht der Skip. Keine heuristische „Uhr-Ausnahme" bauen (wäre genau der gefährliche semantische Hash aus Fakt 2). |
| **Hash-Fehlschluss ⇒ eingefrorenes Panel** (Skip, obwohl Panel anders aussieht als angenommen) | Konservativ designt: Byte-Identität statt Semantik (kein false-negative per Konstruktion, Fakt 4); Hash nur nach ERFOLGREICHEM Write übernommen (AC7); in-memory ⇒ Neustart schreibt immer; manueller Trigger erzwingt Write (AC8); 24h-Guard als letztes Netz (AC9); Kill-Switch `EINK_CONTENT_SKIP=false` (AC10). Fünf unabhängige Verteidigungslinien. |
| Falsche/fehlende TZ (v. a. Docker) ⇒ Fenster wirkt zur falschen Uhrzeit | `TZ` wird in docker-compose durchgereicht (Default Europe/Berlin, Fakt 6); README-Hinweis beim Fenster-Setting. Symptom ist harmlos (Refreshes zur falschen Zeit unterdrückt), Trigger geht immer. DST-Kanten verschieben das Fenster um 1 h Wanduhrzeit — akzeptiert, dokumentiert. |
| Versions-Skew Server/Client (nur eine Phase deployt) | Kontrakt deckt beide Richtungen: alter Server + neuer Client ⇒ `reason` fehlt ⇒ nie Skip (AC8); neuer Server + alter Client ⇒ Client ignoriert `reason`, verhält sich wie heute; Fenster wirkt trotzdem (Server-only). |
| Bedeutungsverschiebung `last_client_refresh` (auch Skip zählt als „refresht") | Bewusste Entscheidung 9 (sonst Poll-Sturm auf `/preview`); im CHANGELOG + README dokumentiert; Diagnose bleibt über Heartbeat-`status`-Logging möglich. |
| Nachtfenster unterdrückt auch den allerersten Refresh nach Installation | Durch Erststart-Ausnahme (Entscheidung 3, AC2) ausgeschlossen. |
| Client lädt im Fenster weiter alle 30 s `refresh_status` | Bewusst: `refresh_status` ist billig (kein Render, Fakt 2); Poll-Stopp wäre Client-Komplexität ohne messbaren Nutzen (Non-Goal, ehrliche Nutzenrechnung s. Ziel). |
| Panel-Gesundheit bei sehr langen Skip-Strecken | 24h-Guard erzwingt Waveshare-konforme Mindest-Refreshrate (Fakt 3); Fenster + Guard zusammen können 24 h nicht überschreiten (Fenster < 24 h per Format, Guard misst echte Writes). |
| **Rollback** | Zwei unabhängige, additive Commits (Phase 1 Server, Phase 2 Client). `settings.json`-Felder sind `omitempty` und werden von altem Code ignoriert; `reason` ist additiv; Heartbeat-`status` war schon immer semantikfrei. `git revert` je Phase genügt; zurückbleibende `sleep_*`-Felder in settings.json sind für alten Code inert. |
