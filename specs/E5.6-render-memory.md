# E5.6: Render-Speicher & -Nebenläufigkeit auf 512-MB-Pi zähmen

**Status:** Draft
**Abhängigkeiten:** E1.1 (gemergt) — die Golden-/Palette-Suite ist das zentrale Sicherheitsnetz dieses Tasks: JEDE Maßnahme hier muss byte-identische Render-Ausgabe beweisen. E1.6 (gemergt) — Quantisierungs-Pipeline in `quantize.go` (Precomp-Pass wird hier angefasst).
**Koordination:** E4.4 (Widget-Fetch-Entkopplung) läuft separat — Widget-HTTP-Calls bleiben in diesem Task IM Render-Pfad (Risiko dokumentiert, s. u.). Wer `preview.go` parallel anfasst: Merge-Reihenfolge abstimmen.
**Zuständig:** go-backend — Ressourcen-/Infra-Thema (main, config, handlers, docker-compose, .env). Die render-quality-Dateien `services/preview.go`/`services/quantize.go` dürfen minimal berührt werden (Semaphore-Hook, Puffer-Diät) — keine Änderung an Render-Semantik oder Quantisierungs-Mathematik.

## Ziel

Der Server hält auf der 512-MB-Pi-Klasse ein planbares Speicherbudget ein: **gemessener Ausgangsbefund (HIL 2026-07-15, Pi 3, Docker): VmHWM 106,7 MB / RSS 98,2 MB** — das v1.0-Ziel ist Server **< 25 MB RSS (nativ, idle)**. Preview-Render dauert 4,1 s auf dem Pi (Baseline, darf sich nicht wesentlich verschlechtern).

Drei Hebel, alle ohne Änderung der Render-Ausgabe (Golden Files byte-identisch):

1. **Render-Semaphore** — global max. N gleichzeitige Renders (Default 1), weitere Requests warten mit Context-Abbruch statt parallel Puffer zu stapeln.
2. **GOMEMLIMIT** — Soft-Limit für die Go-Runtime, damit freier Heap nach Render-Bursts zügig ans OS zurückgeht statt bei GOGC=100 auf ~2× Live-Heap zu wachsen.
3. **Puffer-Diät wo billig** — nur No-Brainer: Precomp in-place statt Kopie, gepoolter Downscale-Scaler; kein Renderer-Umbau.

**Warum das die 98 MB erklärt:** Ein einziger High-Quality-Render alloziert ~35 MB (s. Puffer-Inventur), davon 24,6 MB allein der interne Temp-Puffer von `CatmullRom.Scale`. Bei GOGC=100 wächst das Heap-Ziel auf ~2× des Live-Peaks (~70 MB), unbegrenzt parallele Renders stapeln das, und der Scavenger gibt freie Pages nur zögerlich zurück → ~100 MB RSS nach Bursts, obwohl idle fast nichts lebt.

### Verifizierte Fakten zum Ist-Zustand

1. **Render-Entry-Points sind genau zwei:** `GET /preview` (auch der Abholweg des E-Ink-Clients) und `POST /api/preview_live` (Designer-Live-Preview), beide über `PreviewService.Render` (handlers/preview.go Z. 35/37/59). Es existiert **keinerlei Begrenzung paralleler Renders** — `net/http` startet pro Request eine Goroutine, weder `main.go` noch Handler noch Service drosseln. (Nebenbefund: `DesignHandler` hält ein ungenutztes `preview`-Feld — totes Feld, nicht Teil dieses Tasks.)
2. **`CatmullRom.Scale` alloziert pro Aufruf einen ungepoolten Temp-Puffer** (verifiziert in `golang.org/x/image@v0.36.0/draw/scale.go`): `Kernel.Scale` ruft `newScaler(dw, dh, sw, sh, usePool=false)`, dessen `Scale` `tmp := make([][4]float64, dw*sh)` frisch anlegt — **32 Bytes × dstBreite × srcHöhe**. `Kernel.NewScaler(...)` dagegen legt einen `sync.Pool` für exakt diesen Puffer an („optimized for scaling multiple times with the same fixed … width and height") und liefert byte-identische Ergebnisse (identischer `kernelScaler`-Codepfad, Puffer wird vollständig überschrieben).
3. **`precompensate` kopiert:** `quantize.go` Z. 84–85 legt per `image.NewRGBA(bounds)` + `draw.Draw` eine vollständige Kopie des Inputs an und mutiert dann `dst.Pix`. Beide Aufrufwege übergeben einen Puffer, der `Render` privat gehört (Downscale-`dst` bzw. bei `fast` der Canvas selbst) — in-place wäre in allen heutigen Pfaden safe, LUT+Sättigung sind pro Pixel unabhängig → byte-identisches Ergebnis.
4. **Go-Runtime-Verhalten:** GOGC=100 (Default) → Heap-Ziel ≈ 2× Live-Heap nach GC; freie Pages gehen nur über den Background-Scavenger zurück (träge). Die Runtime honoriert die native Env-Var `GOMEMLIMIT` seit Go 1.19 selbst; `debug.SetMemoryLimit` überschreibt sie — die Präzedenz-Logik in `main` muss die native Var respektieren (Architektur-Richtung Punkt 2). GOMEMLIMIT ist ein **Soft-Limit**: Überschreiten heißt aggressiver GC, kein OOM-Kill.
5. **Widget-Fetches laufen synchron IM Render-Pfad** (`fillWeatherContent`, `fillCalendarContent`, `fillNewsContent`, `fillCustomContent`) mit `defaultHTTPClient{Timeout: 10s}` (helpers.go Z. 16). Mit N=1 hält ein Render mit hängendem Feed die Semaphore bis zu >10 s — bekanntes Risiko, Entkopplung ist E4.4 (Non-Goal hier).
6. **HTTP-Timeouts:** `http.Server` hat `WriteTimeout: 30s` (main.go Z. 183) — schlägt es zu, wird die Verbindung geschlossen und `r.Context()` gecancelt. Warteschlangen-Wait MUSS deshalb auf `ctx.Done()` selektieren; ein zusätzlicher eigener Queue-Timeout ist nicht nötig.
7. **Docker-Memory-Limits wirkungslos auf dem Pi:** `docker-compose.yml` setzt `deploy.resources.limits.memory: 128M` (Server) / `64M` (Client), aber der HIL-Befund zeigt: cgroup-Memory-Controller auf dem Test-Pi nicht aktiv → Limits greifen nicht. Wird nur dokumentiert (INSTALL.md), nicht gefixt (Non-Goal).
8. **Golden-Harness (E1.1/E1.6) läuft ohne Netz und byte-exakt**; `settings.json` wird in `setupGoldenServicesVariant` gepinnt. `TestGoldenRender`, `TestPaletteExactness`, `TestRenderDeterminism` sind die Regressions-Wächter für dieses Task.
9. **Fonts sind gecached** (max. 10 Einträge, `fontCache`, preview.go Z. 39/54) — kein Handlungsbedarf, zählt zum Idle-Baseline-Heap.
10. **Nativ-Deploy:** `setup.sh` generiert `.env` aus `.env.example`; die systemd-Unit lädt sie per `EnvironmentFile` — neue Vars in `.env.example` reichen für beide Deploy-Wege.

### Bindende Architektur-Richtung (Manager)

1. **Render-Semaphore (global, konfigurierbar):**
   - `EINK_MAX_CONCURRENT_RENDERS` (int ≥ 1, Default **1**; ungültige Werte → Warnung loggen + Default). Env, NICHT settings.json — Infra-Parameter, kein User-Setting.
   - Implementierung als gepufferter Channel (`chan struct{}` Kapazität N) im `PreviewService`, plus `active atomic.Int32` (Inkrement nach Acquire, Dekrement bei Release) — dient dem Serialisierungs-Test (AC1) und kostet 2 Atomic-Ops pro Render.
   - `Render` bekommt einen `context.Context` als ersten Parameter (ebenso `RenderActive`/`RenderActiveRaw`); Handler reichen `r.Context()` durch, Tests `context.Background()`. Acquire: `select { case sem <- struct{}{}: … ; case <-ctx.Done(): return nil, ctx.Err() }`.
   - Handler antworten bei `ctx.Err()` mit **503** über das bestehende `jsonError`-Muster (Client weg oder WriteTimeout — Schreiben darf dann fehlschlagen, egal). Wartezeiten > 5 s per `slog.Warn` loggen (Diagnose für E4.4).
   - Die Semaphore umschließt den GESAMTEN `Render`-Body (inkl. Widget-Fetches — Fakt 5, Risiko dokumentiert). `/health`, Statics, Settings etc. bleiben ungegated.
   - Konstruktor-Kompatibilität: `NewPreviewService` behält Default N=1; `main` setzt den konfigurierten Wert (Setter oder zusätzlicher Parameter — Implementierungsdetail, aber bestehende Test-Aufrufer dürfen nicht flächig brechen).
2. **GOMEMLIMIT (Soft-Limit, Default 64 MiB):**
   - Präzedenz in `main`, VOR allem anderen: `EINK_GOMEMLIMIT` gesetzt → `debug.SetMemoryLimit(parse(v))`; sonst native `GOMEMLIMIT`-Env gesetzt → **nichts tun** (Runtime honoriert sie selbst, `SetMemoryLimit` würde sie überschreiben); sonst → `debug.SetMemoryLimit(64 MiB)`.
   - `EINK_GOMEMLIMIT`-Format: positive Ganzzahl mit optionalem Suffix `MiB` (z. B. `64MiB`, `48MiB`; nackte Zahl = Bytes); Sonderwerte `off`/`0` = kein Limit (heutiges Verhalten, Fluchtweg). Parser als pure Funktion (unit-testbar); angewandtes Limit beim Start per `slog.Info` loggen.
   - **Herleitung Default 64 MiB** (aus der Puffer-Inventur): Idle-Baseline ~5 MB + Live-Peak eines HQ-Renders ~32–35 MB ≈ **40 MB Live-Maximum im Normalbetrieb** (Semaphore=1 verhindert Stapelung). 64 MiB (67,1 MB) = ~1,6× Headroom → kein Dauer-GC im Normalfall, deckelt den Heap aber auf ~⅔ des gemessenen 98-MB-RSS. Die vom Manager genannten 48 MiB (50,3 MB) wären nur ~1,25× Headroom — GC liefe während jedes Render-Ausklangs nahezu durch (Thrash-Risiko); erst NACH bewährter Puffer-Diät (B2 senkt den Peak um ~24,6 MB) ist 48 MiB als Folge-Tuning sinnvoll — im CHANGELOG als Ausblick notieren.
   - **Risiko „zu niedrig" explizit dokumentieren:** Symptom ist stark steigende Renderzeit/CPU (GC-Anteil), kein Crash; Foto-Renders (Inventur #5/#6) dürfen das Soft-Limit transient reißen — die Runtime GC-t dann hart, killt aber nichts. Fluchtweg `EINK_GOMEMLIMIT=off`.
3. **Puffer-Diät — genau drei Kandidaten, bewertet:**
   - **B1 (MUSS) — `precompensate` in-place:** Bei `*image.RGBA`-Input direkt `src.Pix` mutieren (kein `NewRGBA`, kein `draw.Draw`); für andere Bildtypen bleibt der Kopier-Fallback. Ownership-Kontrakt als Doku-Kommentar an `quantizeForDisplay`: „darf src mutieren; Aufrufer übergeben einen render-privaten Puffer" (trifft auf beide heutigen Pfade zu, Fakt 3). Spart 1,54 MB + einen Kopier-Pass. Byte-identisch (Fakt 3).
   - **B2 (SOLL) — gepoolter Downscale-Scaler:** Downscale-Callsite (preview.go:207–208) nutzt einen per `CatmullRom.NewScaler(dw, dh, sw, sh)` erzeugten, wiederverwendeten Scaler statt `CatmullRom.Scale` — eliminiert die 24,58-MB-Allokation pro Render (Fakt 2; `sync.Pool` wird bei GC-Druck geleert → Idle-Ziel unbeeinträchtigt). Cache klein und gebunden halten: Key (dw, dh, sw, sh), max. ~3 Einträge (die drei Quality-Stufen des Standard-Canvas); abweichende Canvas-Maße (design-gesteuert!) fallen auf `CatmullRom.Scale` zurück — KEIN unbegrenzter Cache. NICHT auf `resizeImage` anwenden (Quellbild-Maße beliebig → Cache unbegrenzt; Non-Goal). Byte-identisch (Fakt 2).
   - **B3 (GEPRÜFT UND VERWORFEN) — PNG-Encode direkt in den ResponseWriter:** verlöre den `Content-Length`-Header und die saubere 500-Semantik bei Encode-Fehlern (Verhaltensänderung, vom Manager-Kriterium „ohne Verhaltensänderung" ausgeschlossen); der Puffer ist mit ≤ ~0,5 MB gegenüber #8 irrelevant. Entscheidung in dieser Spec dokumentiert, keine Code-Änderung.
4. **Coalescing/latest-wins für `/api/preview_live` — bewertet, hier Non-Goal:** Eine minimale Variante (globaler Single-Slot: ein wartender, noch nicht gestarteter Live-Render wird vom nächsten verdrängt, verdrängter Request bekommt z. B. 409) wäre ~30–50 Zeilen, erzwingt aber eine Frontend-Protokoll-Änderung (Designer muss den neuen Statuscode behandeln) und eigene Race-Tests. Für ein Single-User-Gerät mit Frontend-Debounce reicht die einfache FIFO-Semaphore: wartende Requests brechen ohnehin per `ctx.Done()` ab, wenn der Browser sie cancelt. → Non-Goal mit Follow-up-Notiz.
5. **Messbarkeit:**
   - `BenchmarkRenderHighQuality` (Design `basic`, HQ, floyd_steinberg, calibration default, Golden-Setup-Mechanik) mit `b.ReportAllocs()`; informativ zusätzlich Sub-Benchmark auf `gradient` (Foto-Pfad, bleibt hoch — Inventur #6).
   - `TestRenderAllocBudget`: 2 Warmup-Renders (Pool/Fontcache warm), dann `runtime.ReadMemStats`-`TotalAlloc`-Delta über genau einen Render → **< 20 MiB** (AC4; Herleitung s. dort). `TotalAlloc` ist monoton und GC-unabhängig — kein Flake durch GC-Timing.
   - Serialisierungs-Test end-to-end via `httptest` gegen den echten Handler-Stack (AC1).

## Akzeptanzkriterien

1. **AC1 — Semaphore serialisiert, nichts schlägt fehl:** Bei `EINK_MAX_CONCURRENT_RENDERS=1` (Default) feuern 4 parallele `GET /preview`-Requests via `httptest` gegen den verdrahteten Handler-Stack (deterministisches Design, Golden-Setup-Mechanik): alle 4 antworten 200 mit gültigem PNG, und das beobachtete Maximum von `active` (Atomic-Zähler im Gate) ist exakt 1. Der Test läuft unter `-race` sauber. Zusätzlich: ein Request, dessen Context bereits gecancelt ist, verlässt die Warteschlange ohne zu rendern und der Handler antwortet 503 (Unit-Test mit `context.WithCancel`).
   *Beweis:* neuer Test in `handlers/preview_test.go` (+ ggf. Service-Test) grün mit `-race -count=1`.

2. **AC2 — GOMEMLIMIT-Plumbing korrekt:** (a) Ohne jede Env-Var setzt `main` 64 MiB via `debug.SetMemoryLimit` und loggt das; (b) `EINK_GOMEMLIMIT=48MiB` überschreibt; (c) `EINK_GOMEMLIMIT=off` (oder `0`) setzt kein Limit; (d) ist die native `GOMEMLIMIT`-Env gesetzt und `EINK_GOMEMLIMIT` leer, ruft `main` `SetMemoryLimit` NICHT auf (native Var gewinnt, Fakt 4). Der Format-Parser (`64MiB` | Bytes | `off`/`0`; Fehler → Warnung + Default) ist eine pure Funktion mit Tabellen-Unit-Test inkl. Fehlerfällen.
   *Beweis:* Parser-Test grün; Präzedenz (a)–(d) als Unit-Test der Entscheidungsfunktion (Logik aus `main` herausgezogen, damit testbar); Startup-Log-Zeile im HIL-Lauf sichtbar.

3. **AC3 — Puffer-Diät umgesetzt wie spezifiziert:** (a) B1: `precompensate` alloziert für `*image.RGBA`-Input kein neues Bild — ein Unit-Test übergibt ein RGBA-Bild und asserted, dass das Ergebnis dasselbe Backing-Array nutzt (`&result.Pix[0] == &src.Pix[0]`); der Nicht-RGBA-Fallback kopiert weiterhin (Test mit z. B. `*image.NRGBA`). (b) B2: der Scaler-Cache ist auf ≤ 3 Einträge gebunden, abweichende Dimensionen nutzen den Fallback (Unit-Test mit ungewöhnlicher Canvas-Größe rendert korrekt). (c) B3 bleibt unangetastet: `Preview`/`PreviewLive` senden weiterhin `Content-Length`.
   *Beweis:* neue Unit-Tests grün; `git diff` zeigt keine Streaming-Änderung in den Handlern.

4. **AC4 — Alloc-Budget mit Zahl:** `TestRenderAllocBudget` misst das `TotalAlloc`-Delta eines High-Quality-Renders des `basic`-Designs (FS, calibration default, warmer Zustand): **< 20 MiB** (20 971 520 B). Herleitung: nach B1+B2 verbleiben Canvas 6,14 + Downscale-Ziel 1,54 + Paletted 0,41 + Text-Puffer (~3–6) + PNG/Kleinkram (~0,5) ≈ 12–15 MiB; 20 MiB gibt Marge für ungemessene Element-Puffer, fängt aber jede Reintroduktion der 24,6-MB-Temp-Allokation (#8) sicher (36+ MiB → rot). `BenchmarkRenderHighQuality` ist eingecheckt; die konkreten Vorher/Nachher-Zahlen (B/op, allocs/op — erwartet vorher ~36–40 MB/op) werden im MR dokumentiert.
   *Beweis:* Test grün; Negativprobe (manuell, nicht committen): B2 auf `CatmullRom.Scale` zurückdrehen → `TestRenderAllocBudget` muss rot werden.

5. **AC5 — Golden-/Palette-Suite byte-identisch grün:** `TestGoldenRender`, `TestPaletteExactness`, `TestRenderDeterminism` laufen OHNE `-update` vollständig grün; `git status --short server/internal/services/testdata/golden/` ist leer. Keinerlei Render-Verhaltensänderung — das ist das harte Gate für B1/B2.
   *Beweis:* Suite-Lauf unter „Verifikation"; leerer testdata-Diff im MR.

6. **AC6 — Config-Hygiene + Doku:** `.env.example` enthält `EINK_MAX_CONCURRENT_RENDERS` und `EINK_GOMEMLIMIT` mit erklärendem Kommentar (ohne echte Werte); `docker-compose.yml` reicht beide an den Server-Container durch (Default-Muster `${VAR:-default}`); `INSTALL.md` dokumentiert, dass die Docker-Memory-Limits auf Standard-Pi-OS ohne aktivierten cgroup-Memory-Controller wirkungslos sind (nur Doku, kein Fix); `CHANGELOG.md` (Keep a Changelog) beschreibt beide Vars, den 64-MiB-Default, den Fluchtweg `off` und den 48-MiB-Ausblick. `go test ./... -count=1` grün, `gofmt -l .` leer, `go vet ./...` sauber.
   *Beweis:* Kommandos unter „Verifikation"; `git diff --stat`.

## Non-Goals

- **Kein Coalescing/latest-wins für Preview-Live** — bewertet (Architektur-Richtung Punkt 4), bewusst verschoben: erfordert Frontend-Protokoll-Änderung; einfache FIFO-Queue + Context-Abbruch reicht für Single-User-Betrieb. Follow-up-Kandidat, falls die Designer-UX bei N=1 spürbar leidet.
- **Keine Widget-Fetch-Entkopplung** — die synchronen HTTP-Calls im Render-Pfad sind E4.4; hier wird nur das daraus folgende Semaphore-Risiko dokumentiert (Fakt 5, Risiken).
- **Kein Client-RAM** — Python-Client (~17 MB) bleibt unangetastet.
- **Keine Docker-Memory-Limits / kein cgroup-Fix** — cgroup-Memory auf dem Pi inaktiv (HIL-Befund); wird ausschließlich dokumentiert (AC6). Kernel-Cmdline-Änderungen am Pi sind nicht Teil des Repos.
- **Kein Streaming-/Tiled-Renderer-Umbau** — der Renderpfad (Supersampling → CatmullRom → Quantisierung) bleibt strukturell unverändert; ebenso kein Wechsel des Skalierungs-Kernels.
- **Kein Decode-Cache für Bild-Elemente und keine Optimierung von `resizeImage`** (Inventur #5/#6) — der Foto-Worst-Case (~150 MB transient bei 12-MP-Upload) ist dokumentiert, aber Folgearbeit (sinnvoller wäre serverseitiges Downscaling beim Upload).
- **Kein settings.json-Feld für Concurrency/Memlimit** — reine Env-Konfiguration (Infra, kein User-Setting).
- **Keine Frontend-Änderungen** (`server/static/` bleibt unberührt).

### Gate L3 — Hardware-in-the-Loop (Pi, nativ — Richtwerte, keine harten ACs)

Nativer Lauf (setup.sh-Deploy), da das v1.0-RSS-Ziel nativ definiert ist:

```sh
# Baseline nach Neustart
systemctl restart eink-server && sleep 5
PID=$(pgrep -f 'server' | head -1)
grep -E 'VmHWM|VmRSS' /proc/$PID/status

# Render-Burst: 10 sequenzielle HQ-Previews + 4 parallele
for i in $(seq 10); do curl -so /dev/null -w '%{time_total}s\n' http://localhost:5000/preview; done
for i in $(seq 4); do curl -so /dev/null http://localhost:5000/preview & done; wait
grep -E 'VmHWM|VmRSS' /proc/$PID/status

# Idle-Rückgabe beobachten
sleep 150 && grep -E 'VmHWM|VmRSS' /proc/$PID/status
```

Erwartungen (Richtwerte): **VmHWM ≤ ~60 MB** (vorher 106,7 MB), **VmRSS nach ≥ 2 min idle ≤ ~35 MB** (Weg Richtung v1.0-Ziel < 25 MB — das Ziel selbst hängt zusätzlich an Folgearbeiten, u. a. Foto-Pfad). **Renderzeit ≤ ~5 s** pro Preview (Baseline 4,1 s — deutlich mehr deutet auf GC-Thrash → `EINK_GOMEMLIMIT` erhöhen und Befund melden). Der 4-fach-Parallel-Burst darf weder Fehler noch OOM erzeugen. Ergebnisse im MR dokumentieren.

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| GOMEMLIMIT zu niedrig → GC-Thrash (Renderzeit/CPU steigen stark) | 64-MiB-Default mit Herleitung aus der Puffer-Inventur (~1,6× Live-Peak); Gate L3 misst Renderzeit gegen die 4,1-s-Baseline; Fluchtweg `EINK_GOMEMLIMIT=off`; 48 MiB erst als Folge-Tuning nach bewährter Diät. Soft-Limit ⇒ kein OOM-Kill, schlimmster Fall ist Langsamkeit. |
| N=1 + hängender Widget-Fetch (10-s-Client-Timeout, Fakt 5) blockiert die Queue | Bekannt und akzeptiert bis E4.4: `WriteTimeout` 30 s begrenzt jeden Request hart, wartende Requests brechen per Context ab (503 statt Hänger), Slow-Wait-Logging macht den Fall sichtbar. |
| Signaturänderung `Render(ctx, …)` zieht durch Handler + alle Tests | Rein mechanisch, compiler-erzwungen; keine Semantikänderung. Bestehende Test-Aufrufer nutzen `context.Background()`. |
| B2-Pool hält 24,6 MB zwischen Renders im Speicher | `sync.Pool` wird bei GC geleert (unter GOMEMLIMIT zügig, spätestens beim 2-min-Force-GC) → Idle-RSS-Ziel unbeeinträchtigt; Gate L3 prüft die Idle-Rückgabe explizit. |
| AC4-Budget (20 MiB) plattform-/versionsabhängig | `TotalAlloc`-Deltas sind deterministisch bis auf Map-/Slice-Growth-Rauschen (KB-Bereich); Marge 20 MiB vs. erwartete 12–15 MiB. Ein x/image- oder Go-Bump, der das Budget reißt, soll auffallen — dann Zahl bewusst im selben Commit anpassen (Golden-`-update`-Disziplin analog). |
| Fehler in B1/B2 verändert Pixel | Genau dafür ist AC5 das harte Gate: Golden-/Palette-/Determinismus-Suite byte-exakt, plus Pointer-/Fallback-Unit-Tests (AC3). Beide Maßnahmen sind nachweislich pfadidentisch (Fakten 2–3). |
| Docker-Messung ≠ Nativ-Messung | HIL-Ausgangsbefund war Docker; das v1.0-Ziel ist nativ definiert → Gate L3 misst nativ. Docker-cgroup-Limits auf dem Pi wirkungslos — dokumentiert (AC6), kein Fix hier. |
| Foto-Uploads sprengen jedes Budget transient (~150 MB, Inventur #6) | Bekannt, dokumentiert, Non-Goal; Semaphore verhindert Stapelung, GOMEMLIMIT drückt danach zurück; nachhaltiger Fix (Upload-Downscaling) als Folgearbeit notiert. |
| **Rollback** | Ein Server-Commit (main, config, services, handlers, Tests) + Compose/Env/Doku; Env-Vars sind additiv, keine persistenten Formatänderungen, keine API-/Client-Bruchstellen — `git revert <commit>` stellt den Vorzustand her. |
