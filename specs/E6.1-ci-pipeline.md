# E6.1: CI-Pipeline (GitHub Actions) — Format, Vet, Tests, Cross-Builds

**Status:** Draft
**Abhängigkeiten:** keine (E1.1-Golden-Harness ist bereits gemergt und wird von dieser Pipeline nur ausgeführt)
**Blockiert:** E6.2 (Release-Workflow — baut auf denselben Cross-Build-Schritten auf); mittelbar jede weitere Änderung, die ohne CI unbemerkt Format-/Test-Regressionen einschleusen könnte
**Zuständig:** test-engineer — Schreibrecht NUR `.github/**`; kein Produktivcode, keine Test-Dateien, keine Scripts

## Ziel

Jeder Push auf `main` und jeder Pull Request wird automatisch gegen Format (`gofmt`), Statik (`go vet`), die komplette Go-Test-Suite (inkl. Golden-/Palette-Harness), die Python-Client-Tests und Cross-Builds für alle vier Deployment-Targets geprüft. Insbesondere `armv6` (Pi Zero) und `armv7` haben **kein Testgerät** — der Cross-Build in CI ist der einzige Schutz davor, dass diese Targets unbemerkt brechen.

## Kontext

### Betroffene Dateien

| Datei | Rolle |

| `server/go.sum` | Nur lesen: existiert — `cache-dependency-path` für den Modul-Cache von `setup-go` |
| `server/internal/services/golden_test.go` | Nur lesen: `TestGoldenRender` (Z. 176), `TestPaletteExactness` (Z. 293) — laufen offline/deterministisch, siehe Fakten 3 |
| `client/test_client.py` | Nur lesen: `unittest`-Suite (781 Z.), Mock-EPD, kein pytest; `assertNoLogs` (Z. 562) → Python ≥ 3.10 |

### Verifizierte Fakten

1. **Repo-Realität:** Remote ist `github.com/Kilian-Schwarz/E-INK-Picture`, Trunk ist `main`, `.github/` existiert nicht — dieser Workflow ist die erste CI überhaupt. Keine Alt-Workflows, keine Namenskollisionen.
2. **Go-Version:** `server/go.mod` deklariert exakt `go 1.24.0`. `actions/setup-go` mit `go-version-file: server/go.mod` installiert eine dazu passende Toolchain; **`gofmt` muss aus genau dieser Toolchain kommen** (also nach `setup-go` einfach `gofmt` aus dem PATH), niemals aus einem separat installierten Go — Format-Regeln können sich zwischen Go-Minor-Versionen ändern, und ein Versions-Drift zwischen Dev-Maschine und CI erzeugt Phantom-Failures.
3. **Golden-Suite ist Runner-tauglich:** `golden_test.go` schreibt die eingebettete `goregular`-TTF (bereits Go-Dependency, kein Netzwerk) als `testfont.ttf` ins `t.TempDir()` und pinnt jedes Text-Element darauf — Systemfonts des Runners werden nie erreicht. Kein HTTP, kein `time.Now` im Assertion-Pfad. Designt für Byte-Identität auf jeder Maschine; der erste CI-Lauf auf `ubuntu-latest` ist allerdings der erste echte Beweis außerhalb von macOS/Docker (siehe Risiken).
4. **Python-Versions-Untergrenze:** `test_client.py` nutzt `unittest.TestCase.assertNoLogs` (Z. 562, seit Python 3.10) und `Image.Transpose` (Pillow ≥ 9.1). Der Ziel-Pi läuft Debian 13 mit Python ≥ 3.11. → CI pinnt **Python 3.11** (Deployment-Untergrenze, nicht „latest" — die CI soll das prüfen, was auf dem Pi läuft).
5. **Nicht `requirements.txt` installieren:** Die Datei enthält Hardware-Bindings (`RPi.GPIO`, `spidev`, `lgpio`, …), die auf `ubuntu-latest` teils nicht bauen und für die Tests irrelevant sind — `client.py` fängt fehlendes `waveshare_epd` per `ImportError` ab (Z. 44–46), die Tests mocken den Treiber vollständig. CI installiert nur `requests` und `Pillow` (mit denselben Constraints wie in `requirements.txt`: `requests>=2.31.0`, `Pillow>=10.0.0`).
6. **Cross-Build-Targets aus der Deployment-Realität:** `linux/arm64` (Pi 3 / Pi Zero 2 W, 64-bit-OS), `linux/arm GOARM=7` (32-bit-OS auf Pi 2+), `linux/arm GOARM=6` (Pi Zero / Pi 1, `armv6l` — vgl. Commit a585784), `linux/amd64` (Cloud-/Docker-Modus). Der Server ist pures Go (`net/http`, `golang.org/x/image`) — `CGO_ENABLED=0` liefert statische Binaries. **Das einzige Testgerät ist ein Pi 3 (aarch64); für `armv6`/`armv7` existiert keine Hardware — Build-Erfolg in CI ist deren gesamte Absicherung.**
7. **Test-Suite ist offline-fähig:** Alle Go-Tests (`handlers`, `services`) laufen ohne Netz (Weather-Tests mocken via `httptest`); die Python-Suite mockt `requests` vollständig. Kein Secret, kein Token, kein API-Key nötig — der Workflow braucht keinerlei Repository-Secrets.

## Akzeptanzkriterien

1. **AC1 — Workflow-Grundgerüst:** Es existiert genau eine neue Datei `.github/workflows/ci.yml` mit `name: ci` und Triggern `push` (nur Branch `main`) + `pull_request`. Eine `concurrency`-Gruppe pro Ref bricht überholte Läufe ab, aber **nur auf PRs** (`cancel-in-progress: ${{ github.event_name == 'pull_request' }}`) — Läufe auf `main` werden nie gecancelt (jeder Trunk-Commit behält sein vollständiges Ergebnis).
   *Beweis:* Datei im Diff; Validierung per `actionlint` falls lokal verfügbar, sonst mindestens ein YAML-Parse (siehe Gate L1).

2. **AC2 — Job `go` (ubuntu-latest):** `actions/checkout` (auf Major-Version gepinnt, z. B. `@v4`), `actions/setup-go` mit `go-version-file: server/go.mod` und aktiviertem Modul-Cache (`cache-dependency-path: server/go.sum`). Danach in `server/` (via `defaults.run.working-directory`):
   - `gofmt -l .` — Fail bei nicht-leerer Ausgabe, und die Ausgabe (die betroffenen Dateien) muss im Log sichtbar sein (z. B. `test -z "$(gofmt -l .)" || { gofmt -l .; exit 1; }`); `gofmt` stammt aus der setup-go-Toolchain (Fakt 2),
   - `go vet ./...`,
   - `go test ./... -count=1` — läuft die komplette Suite inkl. `TestGoldenRender` und `TestPaletteExactness`; kein Skip, kein `-short`.
   *Beweis:* Lokale Simulation der drei Kommandos grün (Gate L1); Negativprobe AC5.

3. **AC3 — Job `python` (ubuntu-latest):** `actions/setup-python` (Major-gepinnt) mit `python-version: "3.11"` (Fakt 4). Installiert werden ausschließlich `requests>=2.31.0` und `Pillow>=10.0.0` — explizit NICHT `requirements.txt` (Fakt 5; der Grund steht als Kommentar im Workflow). Danach in `client/`:
   - `python3 -m py_compile client.py config.py`,
   - `python3 -m unittest discover -v`.
   *Beweis:* Lokale Simulation mit Python 3.11 grün (Gate L1).

4. **AC4 — Job `cross-build` (ubuntu-latest, Matrix):** Vier Matrix-Einträge, alle mit `GOOS=linux` und `CGO_ENABLED=0`, nur `go build` (keine Tests, keine Emulation):
   | GOARCH | GOARM | Artefakt-Name |
   |---|---|---|
   | `arm64` | — | `server-linux-arm64` |
   | `arm` | `6` | `server-linux-armv6` |
   | `arm` | `7` | `server-linux-armv7` |
   | `amd64` | — | `server-linux-amd64` |
   Jeder Eintrag baut in `server/` (`go build -o <artefakt> .`), prüft dass das Binary existiert und nicht leer ist (`test -s`), und lädt es per `actions/upload-artifact` (Major-gepinnt) mit `retention-days: 7` hoch. `fail-fast: false`, damit ein kaputtes Target die anderen drei Ergebnisse nicht verschluckt. **Alle drei ARM-Builds müssen ein Binary erzeugen** — das ist der einzige Regressionsschutz für Pi Zero/armv7 (Fakt 6).
   *Beweis:* Lokale Simulation aller vier Matrix-Kommandos erzeugt vier nicht-leere Binaries mit korrekter Architektur (`file`-Check, Gate L1).

5. **AC5 — Negativprobe gofmt (lokal simuliert):** Die exakte gofmt-Kommandozeile aus AC2, lokal gegen einen temporär absichtlich falsch formatierten Stand ausgeführt (z. B. Kopie einer `.go`-Datei mit zerstörter Einrückung ins Package, danach rückstandslos entfernen), muss mit Exit ≠ 0 fehlschlagen und die Datei benennen. **Verifikations-Realität:** Der GitHub-Runner selbst kann lokal nicht laufen — alle Gate-L1-Nachweise simulieren nur die Workflow-Kommandos. Der echte Grün-Beweis (alle drei Jobs grün auf `ubuntu-latest`) entsteht erst nach dem Push und ist ein **expliziter Folgeschritt des Managers**, kein Bestandteil dieses Tasks.
   *Beweis:* Protokollierter lokaler Fail + anschließend sauberer Stand wieder grün; Hinweis auf den Folgeschritt im Task-Abschluss.

6. **AC6 — Scope-Hygiene:** Der Diff des Tasks berührt ausschließlich `.github/**`. Keine Secrets, keine Tokens im Workflow (es werden keine gebraucht, Fakt 7); keine `go.mod`-/`requirements.txt`-Änderung.
   *Beweis:* `git diff --stat` zeigt nur `.github/workflows/ci.yml`.

## Non-Goals

- **Kein Release-Workflow** — Tags, GitHub Releases, Versionierung sind E6.2; dieser Task lädt Build-Artefakte nur zur Verifikation hoch (7 Tage Retention), nicht zur Distribution.
- **Kein Linting-Zoo** — kein `golangci-lint`, kein `staticcheck`, kein `ruff`/`flake8`. `gofmt` + `go vet` + `py_compile` sind der bewusst schmale Anfang; mehr Linter sind ein separater Task mit eigener Baseline-Diskussion.
- **Keine Docker-Image-Builds** — `docker compose build` / Multi-Arch-Images (buildx/QEMU) sind teuer und redundant zur Cross-Build-Matrix; ggf. Teil von E6.2.
- **Kein Coverage-Upload** — kein Codecov/Coveralls, keine Coverage-Schwellen.
- **Keine Branch-Protection-Konfiguration** — dass `ci` als Required Check auf `main` erzwungen wird, ist eine GitHub-Repo-Einstellung (Manager/Owner), keine Datei in diesem Repo.
- **Keine ARM-Test-Ausführung** (QEMU/emulierte Runner) — nur Builds; die Test-Suite läuft auf amd64.

Negativprobe gofmt (AC5 — nicht committen, rückstandslos aufräumen): eine Go-Datei absichtlich falsch formatieren → gofmt-Zeile aus AC2 muss mit Exit ≠ 0 fehlschlagen und die Datei listen → Stand zurücksetzen → wieder grün.

### Gate L2 — Echter CI-Lauf (NACH Push, Folgeschritt des Managers)

Der Runner ist lokal nicht ausführbar — Gate L1 beweist nur die Kommandos, nicht den Workflow. Nach dem Merge/Push auf `main`:
- alle drei Jobs grün auf `ubuntu-latest`,
- vier Artefakte im Run sichtbar (`server-linux-{arm64,armv6,armv7,amd64}`),
- Golden-Suite grün auf dem Linux-Runner (erster Cross-OS-Beweis, siehe Risiken).

### Gate L5 — Review

Review gegen diese Spec, insbesondere:
- `gofmt` kommt aus der setup-go-Toolchain, `go-version-file` statt hartkodierter Version (AC2, Fakt 2),
- Python auf 3.11 gepinnt, nur `requests` + `Pillow` installiert, Begründung als Workflow-Kommentar (AC3, Fakt 5),
- Matrix enthält alle vier Targets mit `CGO_ENABLED=0`, `fail-fast: false`, `retention-days: 7` (AC4),
- `concurrency` cancelt nur PRs, nie `main` (AC1),
- alle Actions auf Major-Version gepinnt, keine `@master`-Referenzen, keine Secrets (AC1/AC6).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Golden Files weichen auf dem Linux-amd64-Runner ab (anderes OS/Arch als Dev-Mac) | Per Design ausgeschlossen: Font-Pinning auf eingebettete `goregular`, Stdlib-PNG-Encoder und Floyd-Steinberg sind plattformdeterministisch (E1.1, Fakt 3). **Restrisiko bleibt: der erste CI-Lauf ist der Beweis.** Fallback, falls der Runner doch abweicht: Diff-Statistik aus `TestGoldenRender` analysieren (Ursache identifizieren — Font-Pfad-Leck vs. Stdlib-Differenz), als Bug in E1.1-Nachfolger behandeln; NICHT blind `-update` auf dem Runner-Ergebnis, und die Golden-Konvention (Update nur mit Sichtprüfung, im selben Commit wie die Ursache) gilt unverändert. |
| `setup-go` installiert eine neuere Patch-/Minor-Version als lokal → `gofmt`-Drift | `go-version-file` bindet an `go 1.24.0` als Untergrenze; gofmt-Regeln ändern sich nur mit Minor-Releases und dann für alle sichtbar (CI rot mit Dateiliste). Gegenmaßnahme im Alltag: lokale Toolchain aktuell halten; die CI ist die Referenz, nicht der Mac. |
| `requirements.txt` und CI-Dependency-Liste driften auseinander (neue Client-Dependency wird in CI vergessen) | Bewusster Trade-off (Fakt 5: Hardware-Pakete bauen auf ubuntu nicht). Workflow-Kommentar verweist auf `requirements.txt` als Quelle; ein Import-Fehler fällt sofort durch `py_compile`/`unittest` auf (Fail, nicht Silent-Skip). |
| `actionlint` lokal nicht installiert → Workflow-Syntaxfehler erst nach Push sichtbar | YAML-Parse als Minimal-Gate (L1); der echte Beweis ist ohnehin erst Gate L2. Ein Syntaxfehler kostet einen Fixup-Commit, keine Regression. |
| armv6-Build grün, Binary auf echtem Pi Zero trotzdem defekt (kein Testgerät!) | Ehrlich benannt: CI beweist nur Kompilierbarkeit, nicht Lauffähigkeit. Mehr ist ohne armv6-Hardware nicht erreichbar; `CGO_ENABLED=0` + pures Go minimieren das Laufzeitrisiko. Dokumentierter blinder Fleck, kein Blocker. |
| CI-Minuten/Kosten durch 4-fache Build-Matrix | Reine `go build`-Läufe (~Sekunden) mit Modul-Cache; Artefakte nach 7 Tagen gelöscht. Vernachlässigbar. |
| **Rollback** | Ein einzelner additiver Commit (`.github/workflows/ci.yml`) — `git revert <commit>` entfernt die CI vollständig, kein Produktiv- oder Testcode betroffen. |
