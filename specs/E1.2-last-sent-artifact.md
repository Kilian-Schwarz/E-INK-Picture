# E1.2: Client schreibt Last-Sent-Artefakt (`/tmp/eink_last_sent.png`)

**Status:** Draft
**Abhängigkeiten:** keine (unabhängig von E1.1 umsetzbar)
**Blockiert:** alle Hardware-Gates L3 (Vorbedingung: inspizierbares An-den-Treiber-Bild)

## Ziel

Nach jedem Hardware-Refresh liegt unter einem konfigurierbaren Pfad (Default `/tmp/eink_last_sent.png`) exakt das PNG, das an den EPD-Treiber übergeben wurde — vorher gab es keine Möglichkeit, das tatsächlich gesendete Bild zu inspizieren, wodurch die vermutete clientseitige Bild-Verfälschung (Gap G1) unsichtbar blieb.

## Kontext

### Betroffene Dateien

| Datei | Rolle |
|---|---|
| `client/client.py` | Einziger Änderungsort für Logik: Funktion `display_image(img, display_config)` |
| `client/config.py` | Neue Env-Var nach bestehendem Muster (`X = os.getenv("EINK_X", default)` auf Modulebene) |
| `client/test_client.py` | Neue Unit-Tests (unittest, `MockEPD`-Pattern vorhanden) |
| `client/.env.example` | Neue Env-Var dokumentieren |
| `.env.example` (Repo-Root) | Neue Env-Var dokumentieren (Root dokumentiert bereits `EINK_*`-Client-Variablen) |

### Bestehende Muster in `display_image()` (client/client.py)

Ablauf heute:

1. `epd.init()`
2. Resize auf `epd.width` × `epd.height` (LANCZOS), falls Größe abweicht
3. Modus-Konvertierung, abhängig von `display_config["colors"]`:
   - `len(colors) > 2` (epd7in3e, 6-Farb): `img.convert("RGB")`
   - sonst (epd7in5_V2, B/W): `img.convert("L").point(lambda x: 0 if x < 128 else 255, "1")`
4. `epd.display(epd.getbuffer(img))`
5. `epd.sleep()`

**Zentrale Beobachtung:** Beide Panel-Pfade (epd7in3e und epd7in5_V2) münden in denselben einen Aufruf `epd.display(epd.getbuffer(img))`. Ein einziger Schreibpunkt unmittelbar vor diesem Aufruf — also NACH Resize und Modus-Konvertierung — deckt beide Treiber ab und erfüllt „exakt das Bild, das an den Treiber geht".

**No-Hardware-Pfad:** Wenn `epd is None` (Preview-only, z. B. Mac), speichert `display_image()` heute `preview_output.png` und returnt früh. Dieser Pfad ruft keinen Treiber auf und bleibt unverändert.

**Config-Zugriff:** `display_image()` und andere Funktionen greifen zur Laufzeit via `config.<NAME>` zu (nicht per `from config import ...`), damit Tests `client.config` patchen können. Das neue Feature muss dieses Muster beibehalten (`config.LAST_SENT_PATH` im Funktionskörper lesen).

**Test-Muster:** `test_client.py` nutzt `unittest`, eine `MockEPD`-Klasse (800×480, `display()` speichert Buffer, `getbuffer()` gibt `list(image.tobytes())`) und setzt `client.epd` direkt bzw. patcht `client.config`.

## Akzeptanzkriterien

Alle AC sind durch die unter „Verifikation" genannten Kommandos maschinell prüfbar; pro AC ist der beweisende Test benannt.

1. **AC1 — Konfigurierbarer Pfad:** `client/config.py` definiert
   `LAST_SENT_PATH = os.getenv("EINK_LAST_SENT_PATH", "/tmp/eink_last_sent.png")`
   nach dem bestehenden Modulebenen-Muster.
   *Beweis:* `test_last_sent_path_default` — Import von `config` ohne gesetzte Env-Var liefert `"/tmp/eink_last_sent.png"`; mit gesetzter Env-Var (via `unittest.mock.patch.dict(os.environ, ...)` + Modul-Reload) den gesetzten Wert.

2. **AC2 — Schreibpunkt:** `display_image()` schreibt das finale `img` als PNG nach `config.LAST_SENT_PATH` — nach Resize und Modus-Konvertierung, unmittelbar vor `epd.display(epd.getbuffer(img))`. Das gespeicherte PNG ist pixel-identisch mit dem Bild, das an `epd.getbuffer()` übergeben wird (PNG ist verlustfrei).
   *Beweis:* `test_last_sent_artifact_color` — `MockEPD.getbuffer` wird so erweitert/gepatcht, dass es das übergebene PIL-Image festhält; Assertion: Artefakt existiert, `Image.open(path)` hat `size == (800, 480)`, `mode == "RGB"` und `tobytes()` identisch zum festgehaltenen Bild.

3. **AC3 — Beide Panel-Pfade:**
   - 6-Farb-Pfad (`len(colors) > 2`): Artefakt hat `mode == "RGB"`.
   - B/W-Pfad (`len(colors) <= 2`): Artefakt hat `mode == "1"` (nach PNG-Roundtrip: 1-Bit-PNG, Pixelwerte nur 0/255 nach `convert("L")`).
   *Beweis:* `test_last_sent_artifact_color` (RGB) und `test_last_sent_artifact_bw` (mode "1", Eingabe z. B. RGB-Grau 128, Artefakt-Pixel identisch zum an `getbuffer` übergebenen Bild).

4. **AC4 — Atomares Schreiben:** Das PNG wird zuerst in eine temporäre Datei **im selben Verzeichnis** wie `LAST_SENT_PATH` geschrieben und dann per `os.replace()` auf den Zielpfad verschoben. Kein direktes `img.save(LAST_SENT_PATH)`. (Grund: paralleler `scp`-Abruf darf nie ein halbes PNG sehen; `os.replace` ist nur innerhalb desselben Dateisystems atomar.)
   *Beweis:* `test_last_sent_artifact_atomic` — `client.os.replace` (bzw. `os.replace` im Client-Modul) wird gemockt; Assertion: aufgerufen mit (Temp-Pfad, Zielpfad), Temp-Pfad liegt via `os.path.dirname` im Zielverzeichnis, und der Zielpfad wird nicht direkt per `save` beschrieben (z. B. via Spy auf den Save-Aufruf oder: bei Exception im Save existiert der Zielpfad nicht).

5. **AC5 — Fehler bricht Refresh nie ab:** Schlägt das Artefakt-Schreiben fehl (z. B. Zielverzeichnis nicht vorhanden/nicht schreibbar), loggt der Client eine Warnung (`logger.warning`) und fährt fort: `epd.display()` wird trotzdem aufgerufen, `display_image()` returnt `True`, Heartbeat-Verhalten unverändert.
   *Beweis:* `test_last_sent_artifact_write_failure_does_not_break_refresh` — `config.LAST_SENT_PATH` zeigt auf nicht existierendes Verzeichnis (unter `tempfile.mkdtemp()`-Basis, danach gelöscht) oder Save wird per Mock zum Raisen gebracht; Assertions: Rückgabewert `True`, `MockEPD.displayed_buffer is not None`, `assertLogs(level="WARNING")` enthält den Fehler.

6. **AC6 — No-Hardware-Pfad unverändert:** Bei `epd is None` wird KEIN Last-Sent-Artefakt geschrieben; das bestehende `preview_output.png`-Verhalten bleibt exakt wie heute.
   *Beweis:* `test_display_no_hardware` (bestehend) bleibt grün; neuer Assert-Zusatz oder eigener Test `test_no_artifact_without_hardware`: nach Aufruf existiert `LAST_SENT_PATH` (auf frischen tmp-Pfad gesetzt) nicht.

7. **AC7 — Tests schreiben nicht nach `/tmp`:** Alle neuen Tests setzen `client.config.LAST_SENT_PATH` (Patch) auf einen Pfad unter einem Test-Temp-Verzeichnis (`tempfile.TemporaryDirectory`). Kein Test berührt das Default-`/tmp/eink_last_sent.png`.
   *Beweis:* Code-Review der Tests (L5) + Tests laufen auf dem Mac grün ohne Seiteneffekte.

8. **AC8 — `.env.example` aktualisiert:** Sowohl `client/.env.example` als auch `/.env.example` (Root) dokumentieren `EINK_LAST_SENT_PATH` mit Default-Wert `/tmp/eink_last_sent.png` und einem Kommentar (Zweck: Debug-Artefakt, letztes an den Treiber gesendetes Bild). Kein echter/abweichender Wert.
   *Beweis:* `grep -q EINK_LAST_SENT_PATH client/.env.example .env.example` liefert Treffer in beiden Dateien.

9. **AC9 — Bestand bleibt grün:** Alle vorhandenen Tests in `client/test_client.py` laufen unverändert durch.
   *Beweis:* pytest-Lauf unter „Verifikation".

## Non-Goals

- **KEIN Entfernen der clientseitigen Bild-Konvertierung (Passthrough).** Das ist ein separater Task und darf erst NACH E1.1 (serverseitige Quantisierung) passieren — sonst bekäme das Panel unquantisierte Bilder. E1.2 verändert die Konvertierungslogik in `display_image()` nicht, es beobachtet sie nur (bindende Scope-Entscheidung des Managers).
- Keine Server-Änderungen (Go), keine neuen oder geänderten API-Endpoints.
- Kein HTTP-Endpoint zum Abrufen des Artefakts — Abruf erfolgt via `scp`.
- Keine Änderung am No-Hardware-/Preview-Pfad (`preview_output.png`).
- Keine Artefakt-Historie/Rotation — es existiert immer nur die letzte Datei.
- Keine Änderung an Timing, Poll-Loop, Heartbeat oder Treiber-Laden.

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Halbes PNG bei parallelem `scp`-Abruf | Durch AC4 (temp + `os.replace`) ausgeschlossen; `os.replace` ist atomar im selben Dateisystem — deshalb Temp-Datei zwingend im Zielverzeichnis. |
| Artefakt-Schreiben schlägt fehl (voller/fehlender `/tmp`, Rechte) und reißt den Refresh mit | Durch AC5 ausgeschlossen: Warnung loggen, Refresh läuft weiter. Worst Case: veraltetes Artefakt — akzeptabel für ein Debug-Werkzeug. |
| SD-Karten-Verschleiß durch Schreiben pro Refresh | Refresh-Intervall Default 3600 s ⇒ ~24 Schreibvorgänge/Tag à <1 MB — vernachlässigbar. Falls `/tmp` auf dem Pi kein tmpfs ist, kann `EINK_LAST_SENT_PATH` auf ein tmpfs-Verzeichnis zeigen (Env-Var, AC1). |
| RAM-Spitze auf Pi Zero 2 W (512 MB) durch zusätzliches PNG-Encode | 800×480 in RGB ≈ 1,2 MB roh; PNG-Encode-Peak wenige MB, einmalig pro Refresh — unkritisch gegenüber ~17 MB Client-Budget. |
| PNG-Roundtrip verändert Pixel | PNG ist verlustfrei; mode "1" und "RGB" werden von Pillow nativ als PNG unterstützt. AC2/AC3 beweisen Pixel-Identität per `tobytes()`-Vergleich. |
| Tests verschmutzen `/tmp` auf Dev-Maschinen | Durch AC7 ausgeschlossen (Temp-Verzeichnisse, gepatchter Pfad). |
| **Rollback** | Ein einzelner, rein additiver Client-Commit ohne persistente Formate oder API-Änderungen — `git revert <commit>` genügt. Zurückbleibende `/tmp/eink_last_sent.png` auf dem Pi ist harmlos und verschwindet spätestens beim Reboot (tmpfs) bzw. kann manuell gelöscht werden. |
