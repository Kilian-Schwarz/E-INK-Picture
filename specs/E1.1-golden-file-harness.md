# E1.1: Golden-File-Harness + exakte Palette-Assertions für den Renderer

**Status:** Draft
**Abhängigkeiten:** keine
**Blockiert:** alle nachgelagerten Renderer-Änderungen (Passthrough-Umbau, Dithering-Tuning) — ohne diese Harness sind Renderer-Regressionen unsichtbar
**Zuständig:** test-engineer — NUR `*_test.go` + `testdata/`; falls ein Seam im Produktivcode nötig wird: als Blocker an den Manager melden, NICHT selbst bauen

## Ziel

Jede Regression im Renderpfad (`PreviewService.Render` → Supersampling → Downscale → `quantizeToPalette`) bricht ab sofort den Build, und die Palette-Treue des quantisierten Outputs wird maschinell bewiesen statt nur der Rückgabetyp geprüft.

### Verifizierte Fakten zum Renderpfad

1. **Tests liegen im Package `services`** (kein `_test`-Suffix-Package) — `quantizeToPalette` und alle unexportierten Helfer sind direkt erreichbar. Es ist KEIN Produktivcode-Seam nötig.
2. **Display-Profil kommt aus `settings.json`:** `Render` liest `s.settings.GetDisplayConfig()`; das Profil wird über `data/settings.json` (`display_type`) gesteuert, `render_quality` ebenso (`GetRenderQuality`, Default `high` bei leerem Feld). Tests steuern beides, indem sie pro Subtest eine passende `settings.json` ins `t.TempDir()` schreiben — Muster wie `setupTestServices`, aber mit **korrekten** Werten: `waveshare_7in5_v2` / `waveshare_7in3_e` (Achtung: das bestehende `setupTestServices` schreibt `"waveshare_7in5_V2"` mit großem V — das matcht die Konstante NICHT und fällt nur zufällig über `models.GetDisplayConfig` auf das B/W-Profil zurück; für die neuen Tests nicht kopieren).
3. **Font-Falle (Determinismus über Maschinen):** `loadFontFace` probiert VOR dem eingebetteten Go-Font echte Systempfade (`/usr/share/fonts/noto/...`, DejaVu etc., Z. 960–967). Auf dem Mac existieren die nicht (→ `goregular`), im Docker-/CI-Linux ggf. schon → andere Glyphen → andere Golden Files. Mitigation ohne Produktivcode-Änderung: jedes Text-Element der Test-Designs setzt `fontFamily` auf eine Font-Datei, die der Test selbst nach `<tmpDir>/fonts/` schreibt (z. B. `goregular.TTF` aus `golang.org/x/image/font/gofont/goregular` — bereits Dependency — als `testfont.ttf`). `loadFontFace` → `s.image.GetFontPath("testfont.ttf")` (= `fontsDir` + sanitisierter Name) → `loadTTFFace` → deterministisch, egal welche Systemfonts installiert sind.
4. **Deterministische Element-Typen:** `text`/`i-text`/`textbox` (`fillTextContent` liest nur Props), `shape`, `image` (Datei aus `<tmpDir>/uploaded_images/`, siehe `TestRenderImageWithCrop` Z. 321–387 als Muster). NICHT deterministisch: `widget_clock` (`time.Now`), `widget_weather`/`widget_forecast` (synchrone HTTP-Calls im Renderpfad), `widget_calendar`, `widget_news`, `widget_system`, `widget_timer` — alle für diesen Task tabu.
5. **PNG-Encoding (`image/png` der Stdlib) und Floyd-Steinberg sind für identischen Input deterministisch** — Golden-Vergleich auf Byte-Ebene ist damit zulässig; Pixel-Diff dient nur der Diagnose.

## Akzeptanzkriterien

1. **AC1 — Testdata-Layout:** Es existieren
   - `server/internal/services/testdata/designs/*.json` — mind. 2 deterministische Test-Designs als `models.DesignV2`-JSON (Canvas 800×480):
     (a) `basic.json`: Text-Elemente (verschiedene Größen/Ausrichtungen, `fontFamily: "testfont.ttf"`), Shapes (Fill, Stroke, `rx`), alle 6 Palettenfarben als Fill-Farben vertreten;
     (b) `gradient.json`: ein `image`-Element, das ein foto-artiges Testbild referenziert, welches der Test deterministisch generiert (programmatischer RGB-Verlauf über beide Achsen, z. B. `R=x*255/w, G=y*255/h, B=(x+y)*255/(w+h)`, als PNG nach `<tmpDir>/uploaded_images/` geschrieben) — Zwischentöne erzwingen echtes Dithering.
   - `server/internal/services/testdata/golden/<design>__<display_type>.png` — Referenz-Renderings, NUR für `render_quality: high` (4 Dateien: 2 Designs × 2 Profile).
   *Beweis:* Dateien im Commit; `go vet ./...` ignoriert `testdata/` automatisch.

2. **AC2 — Golden-Diff-Test:** `TestGoldenRender` rendert jedes Design-JSON für beide Display-Profile mit `render_quality: high` über die öffentliche API (`previewSvc.Render(design, false)`) und vergleicht **byteweise** (`bytes.Equal`) gegen das Golden-PNG. Bei Abweichung: Fail mit Diff-Statistik — Anzahl abweichender Pixel, Gesamtpixel, Prozent, Koordinaten der ersten ≤10 Abweichungen (dazu beide PNGs decodieren; weichen schon die Dimensionen ab, das als Fehlermeldung).
   *Beweis:* Test grün auf sauberem Stand; manuell verifizierbar durch Mutieren einer Konstante in `preview.go` (z. B. Padding Z. 173) → Test rot mit Pixel-Statistik.

3. **AC3 — Update-Mechanismus:** Package-Level-Flag `var updateGolden = flag.Bool("update", false, "rewrite golden files")` in der neuen Testdatei. `go test ./internal/services -run TestGoldenRender -update` schreibt die aktuellen Renderings als neue Referenz und markiert den Test als bestanden (mit `t.Log` je geschriebener Datei). Die Konvention steht als Kommentar über dem Flag: **Golden-Update nur bewusst und im selben Commit wie die verursachende Renderer-Änderung.**
   *Beweis:* `-update`-Lauf schreibt die 4 Dateien; anschließender Lauf ohne Flag ist grün.

4. **AC4 — Palette-Exaktheit (ersetzt `TestPaletteQuantization`):** Neuer Test (z. B. `TestPaletteExactness`) rendert `gradient.json` (Dithering wird wirklich gefordert) für **alle 3×2 Kombinationen** aus `render_quality` (`fast`, `medium`, `high`) × Display-Profil als Subtests, decodiert das PNG und sammelt die Menge der unique RGB-Werte aller Pixel:
   - `waveshare_7in3_e`: Menge ist Teilmenge von {`#000000`, `#FFFFFF`, `#FF0000`, `#FFFF00`, `#00FF00`, `#0000FF`} — **kein einziger Zwischenton**; Fail listet jeden fremden RGB-Wert mit Pixelanzahl.
   - `waveshare_7in5_v2`: Menge ⊆ {`#000000`, `#FFFFFF`} (≤ 2 Werte).
   Zusätzlich bleibt die bisherige Typ-Assertion erhalten: decodiertes Bild ist `*image.Paletted`. Das alte `TestPaletteQuantization` (preview_test.go Z. 837–876) wird entfernt — es prüfte nur den Typ.
   *Beweis:* Subtests grün; `grep -n "func TestPaletteQuantization" preview_test.go` liefert keinen Treffer mehr.

5. **AC5 — Determinismus:** `TestRenderDeterminism` rendert dasselbe Design (a) zweimal mit derselben Service-Instanz und (b) einmal mit einer frisch konstruierten zweiten Instanz (deckt Font-Cache-kalt/warm ab) — alle drei Byte-Slices identisch (`bytes.Equal`). Mindestens für `high` und beide Profile.
   *Beweis:* Test grün.

6. **AC6 — Kein Netz, keine Zeit:** Die Test-Designs enthalten ausschließlich `text`/`shape`/`image`-Elemente (keine `widget_*`-Typen), der Testcode macht keine HTTP-Aufrufe und liest keine Wanduhr in Assertions. Widgets mit Live-Daten sind Non-Goal (separater Task, braucht Injection-Seams).
   *Beweis:* Review (Gate L5) + Tests laufen mit deaktiviertem Netz grün (Stichprobe lokal).

7. **AC7 — Laufzeit + Hygiene:** Der komplette Suite-Lauf `go test ./...` (im `server/`-Verzeichnis) bleibt unter 60 s auf dem Dev-Mac; `gofmt -l .` leer; `go vet ./...` sauber. Alle Bestandstests bleiben grün.
   *Beweis:* Kommandos unter „Verifikation".

8. **AC8 — Nur Test-Artefakte:** Der Diff des Tasks berührt ausschließlich `server/internal/services/*_test.go` und `server/internal/services/testdata/**`. Kein Produktivcode, keine `go.mod`-Änderung (alle benötigten Module — `goregular`, `image/png` — sind vorhanden).
   *Beweis:* `git diff --stat` zeigt nur diese Pfade.

## Non-Goals

- **Keine Golden Files für Widgets mit Live-Daten** (clock/weather/forecast/calendar/news/system/timer) — `fillClockContent` nutzt `time.Now()`, `fillWeatherContent` & Co. machen synchrone HTTP-Calls; das braucht Injection-Seams im Produktivcode und ist ein separater Task.
- **Keine Produktivcode-Änderung** — auch keine „kleine". Stellt sich heraus, dass ein Seam unvermeidbar ist (z. B. weil ein Renderpfad ohne Codeänderung nicht deterministisch zu bekommen ist), ist das ein Blocker-Report an den Manager, kein Eigenbau.
- Keine Golden Files für `medium`/`fast` (Referenz-Menge klein halten; Palette-Assertion AC4 deckt diese Stufen ab).
- Keine Client-Tests (Python), keine CI-Pipeline-Konfiguration.
- Keine Bildähnlichkeits-Metriken (SSIM etc.) — Vergleich ist exakt oder rot.

```sh

```

Negativprobe (manuell, nicht committen): eine Render-Konstante in `preview.go` ändern → `TestGoldenRender` muss mit Pixel-Diff-Statistik fehlschlagen.

### Gate L5 — Review

Review gegen diese Spec, insbesondere:
- Test-Designs enthalten keine `widget_*`-Typen, kein `time.Now`/HTTP im Assertion-Pfad (AC6),
- jedes Text-Element pinnt `fontFamily` auf die ins tmpDir geschriebene Font-Datei (Kontext-Punkt 3),
- Golden-Vergleich byteweise, Diff-Statistik nur als Diagnose (AC2),
- `-update`-Konvention als Kommentar dokumentiert (AC3),
- Palette-Test nutzt den Verlaufs-Content, nicht nur Flächenfarben (AC4),
- Diff berührt nur Testdateien + testdata (AC8).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Golden Files sind maschinenabhängig wegen System-Font-Fallback in `loadFontFace` | Größtes Risiko, durch Font-Pinning (Kontext Punkt 3, Review-Punkt in L5) ausgeschlossen: jedes Text-Element referenziert die vom Test geschriebene TTF; Systempfade werden nie erreicht. |
| Dependency-Bump (Go-Stdlib `image/png`-Encoder, `golang.org/x/image` Rasterizer/CatmullRom) ändert Bytes → Golden-Suite rot | Erwartet und gewollt sichtbar: `-update` im selben Commit wie der Bump, mit Sichtprüfung der neuen PNGs. Konvention in AC3 verankert. |
| Floyd-Steinberg propagiert Fehler global — jede noch so kleine RGBA-Änderung kippt viele Pixel | Kein Bug, sondern der Zweck der Harness; die Diff-Statistik (AC2) unterscheidet „1 Pixel" von „30 % des Bildes" für die Diagnose. |
| Repo-Größe durch eingecheckte PNGs | 4 Dateien, 800×480 paletted (≤ 6 Farben) — typisch je zweistellige KB; vernachlässigbar. |
| Golden-Update kaschiert echte Regression („blind `-update` laufen lassen") | Prozess-Risiko; Konvention (AC3) + Review-Gate. Mehr ist testseitig nicht erzwingbar. |
| Flakiness durch parallele Tests / geteilten State | Jeder (Sub-)Test nutzt eigenes `t.TempDir()` + eigene Service-Instanzen (bestehendes Muster `setupTestServices`); AC5 beweist Instanz-Unabhängigkeit. |
| **Rollback** | Rein additiver Test-Commit (plus Entfernen eines schwachen Alt-Tests) — `git revert <commit>` stellt den Vorzustand inkl. `TestPaletteQuantization` wieder her; Produktivverhalten ist zu keinem Zeitpunkt betroffen. |
