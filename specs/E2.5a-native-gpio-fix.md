# E2.5a — Native install must bring up the GPIO/panel on kernel 6.12 (RaspiOS Bookworm/Trixie)

## Problem (found by HIL run 3, 2026-07-15, Pi 3B, Debian/kernel 6.12.47)
The native one-command install (`install.sh` → `setup.sh --headless`) FAILS at the
display-driver check on a current Raspberry Pi OS:
- The `lgpio` Python wheel fails to build during `setup.sh` (compile error, currently
  logged non-fatally).
- Without `lgpio`, `gpiozero` falls back to its **NativeFactory** (legacy sysfs
  `/sys/class/gpio/export`), which is broken on kernel ≥ 6.6 (`OSError: [Errno 22]
  Invalid argument`).
- Therefore `from waveshare_epd import epd7in3e` raises, `setup.sh` (correctly) refuses
  the preview-only completion on real hardware, and **no systemd units are installed** —
  the native path never comes up.

This breaks a core v1.0 promise ("fresh SD → one command → panel shows a design,
natively"). It is a SOFTWARE/toolchain bug — the panel, SPI and wiring are proven good
(under Docker the identical `epd7in3e` driver loads and writes the panel in ~19.7 s).

## Goal
`setup.sh --headless` on a stock current Raspberry Pi OS (Bookworm/Trixie, kernel ≥ 6.6)
brings up the native systemd stack with a WORKING panel driver — no manual intervention.

## Acceptance criteria
- **AC1 — lgpio is mandatory and must be installed.** HIL-3 proved that on kernel ≥ 6.6
  (test-Pi: kernel 6.12, Debian 13, Python 3.13) **`lgpio` is the ONLY working gpiozero
  pin factory** — `native`/sysfs is gone (`OSError [Errno 22]`), `rpigpio` (RPi.GPIO
  0.7.1) fails edge-detection (`RuntimeError: Failed to add edge detection`), `pigpio`
  needs a daemon. There is therefore **no viable fallback** — `setup.sh` MUST make
  `lgpio` importable. Prefer the distro package (`sudo apt-get install -y python3-lgpio`)
  or a pinned prebuilt wheel over a source build; if source-building, install the required
  build deps first (inspect `artifacts/hil-3/setup-native-FAILED.log` for the exact wheel
  build error). Then force `GPIOZERO_PIN_FACTORY=lgpio` in the client env (`.env` / systemd
  `Environment=`). Never leave the factory on the broken `native`. On an older kernel where
  `lgpio` is genuinely unavailable, a documented fallback to `rpigpio` is acceptable, but
  the kernel-≥6.6 path (the real target) must use `lgpio`.
- **AC1b — Remove the Jetson.GPIO contamination.** The pinned `waveshareteam/e-Paper`
  package pulls in `Jetson.GPIO`, whose `RPi/GPIO/` shim shadows the real `RPi.GPIO`
  (HIL-3 finding). `setup.sh` must ensure the venv does not end up with `Jetson.GPIO`
  shadowing `RPi.GPIO` (uninstall it, or install the driver in a way that doesn't drag it
  in — the Docker client uses `waveshare/e-Paper` without Jetson.GPIO and works). Verify
  `import RPi.GPIO` resolves to the real package after install.
- **AC2 — Driver import verified before unit install.** The existing driver-import check
  in `setup.sh` stays as the gate, but now passes on a healthy install; if it still fails,
  `setup.sh` emits a clear, actionable error naming the pin-factory cause (not a generic
  abort) and points to `--allow-preview-only` as the explicit escape.
- **AC3 — systemd units come up.** After a successful `setup.sh` the two units
  (server + client) are installed, enabled and active; the client loads `epd7in3e` and
  drives the panel once; `/health` returns 200. (This AC is the L3 hardware gate — proven
  on the Pi, not just in CI.)
- **AC4 — No regression to Docker or to the preview-only path.** Docker mode is
  unaffected. `--allow-preview-only` still tolerates a truly headless/driverless box.
- **AC5 — Idempotent + documented.** Re-running `setup.sh --update` is idempotent w.r.t.
  the pin-factory/env change. The chosen factory and the kernel-6.6 rationale are noted in
  a short comment and in the README/INSTALL native section.
- **AC6 — Script tests.** Extend `scripts/test-setup.sh` (shellcheck-clean) to cover the
  new pin-factory logic (dry-run assertions: the env/unit gets the factory set; the
  lgpio-missing branch falls back rather than aborting). `go`/python static checks stay
  green.

## Verification
- L1: shellcheck + `scripts/test-setup.sh` dry-run matrix.
- L3 (the real gate): hardware-validator runs `setup.sh --headless` on the test-Pi
  (10.33.0.106, kernel 6.12) from a clean-ish state and confirms AC3 — native units up,
  panel driven. This is also the deferred E2.5 gate and the native RAM measurement (E5.6
  L3, Server RSS vs 25 MB) that HIL-3 could not take.

## Notes
- Pin factory is NOT Kilian's call — HIL-3 settled it empirically: on kernel ≥ 6.6 only
  `lgpio` works, so the fix is "make lgpio install + force it", not "pick a preference".
  `lgpio`/`RPi.GPIO`/`python3-lgpio` are all free system packages — no keyed/paid
  dependency, no approval gate.
- The L3 hardware re-run starts from a venv the HIL-3 Path-1 attempt already modified
  (Jetson.GPIO removed, RPi.GPIO rebuilt by hand). For a clean E2.5 gate the
  hardware-validator should let `setup.sh --update` rebuild the venv (or recreate it) so
  the fix is proven from a reproducible state, not from the hand-patched one.
- Do not weaken the driver-import gate into always-preview-only; the panel must actually
  come up on real hardware.
