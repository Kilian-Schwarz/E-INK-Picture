# E2.5a ‚Äî Native install must bring up the GPIO/panel on kernel 6.12 (RaspiOS Bookworm/Trixie)

## Problem (found by HIL run 3, 2026-07-15, Pi 3B, Debian/kernel 6.12.47)
The native one-command install (`install.sh` ‚Üí `setup.sh --headless`) FAILS at the
display-driver check on a current Raspberry Pi OS:
- The `lgpio` Python wheel fails to build during `setup.sh` (compile error, currently
  logged non-fatally).
- Without `lgpio`, `gpiozero` falls back to its **NativeFactory** (legacy sysfs
  `/sys/class/gpio/export`), which is broken on kernel ‚â• 6.6 (`OSError: [Errno 22]
  Invalid argument`).
- Therefore `from waveshare_epd import epd7in3e` raises, `setup.sh` (correctly) refuses
  the preview-only completion on real hardware, and **no systemd units are installed** ‚Äî
  the native path never comes up.

This breaks a core v1.0 promise ("fresh SD ‚Üí one command ‚Üí panel shows a design,
natively"). It is a SOFTWARE/toolchain bug ‚Äî the panel, SPI and wiring are proven good
(under Docker the identical `epd7in3e` driver loads and writes the panel in ~19.7 s).

## Goal
`setup.sh --headless` on a stock current Raspberry Pi OS (Bookworm/Trixie, kernel ‚â• 6.6)
brings up the native systemd stack with a WORKING panel driver ‚Äî no manual intervention.

## Acceptance criteria
- **AC1 ‚Äî Working pin factory guaranteed.** `setup.sh` ensures `gpiozero` uses a pin
  factory that works on kernel ‚â• 6.6. Preferred: make the `lgpio` install succeed
  (install the needed build deps, or use the distro package `python3-lgpio` / a prebuilt
  wheel instead of compiling). Acceptable fallback: if `lgpio` is unavailable, explicitly
  set `GPIOZERO_PIN_FACTORY=lgpio` or `=rpigpio` (RPi.GPIO) in the client's environment
  (`.env` / the systemd unit `Environment=`), and NEVER leave it on the broken
  `native`/NativeFactory. Decide deterministically at install time which factory is
  available and pin it.
- **AC2 ‚Äî Driver import verified before unit install.** The existing driver-import check
  in `setup.sh` stays as the gate, but now passes on a healthy install; if it still fails,
  `setup.sh` emits a clear, actionable error naming the pin-factory cause (not a generic
  abort) and points to `--allow-preview-only` as the explicit escape.
- **AC3 ‚Äî systemd units come up.** After a successful `setup.sh` the two units
  (server + client) are installed, enabled and active; the client loads `epd7in3e` and
  drives the panel once; `/health` returns 200. (This AC is the L3 hardware gate ‚Äî proven
  on the Pi, not just in CI.)
- **AC4 ‚Äî No regression to Docker or to the preview-only path.** Docker mode is
  unaffected. `--allow-preview-only` still tolerates a truly headless/driverless box.
- **AC5 ‚Äî Idempotent + documented.** Re-running `setup.sh --update` is idempotent w.r.t.
  the pin-factory/env change. The chosen factory and the kernel-6.6 rationale are noted in
  a short comment and in the README/INSTALL native section.
- **AC6 ‚Äî Script tests.** Extend `scripts/test-setup.sh` (shellcheck-clean) to cover the
  new pin-factory logic (dry-run assertions: the env/unit gets the factory set; the
  lgpio-missing branch falls back rather than aborting). `go`/python static checks stay
  green.

## Verification
- L1: shellcheck + `scripts/test-setup.sh` dry-run matrix.
- L3 (the real gate): hardware-validator runs `setup.sh --headless` on the test-Pi
  (10.33.0.106, kernel 6.12) from a clean-ish state and confirms AC3 ‚Äî native units up,
  panel driven. This is also the deferred E2.5 gate and the native RAM measurement (E5.6
  L3, Server RSS vs 25 MB) that HIL-3 could not take.

## Notes
- Which pin factory to standardize on is partly Kilian's call if there's a preference, but
  the obvious default is: prefer a working `lgpio` (the modern, recommended factory), fall
  back to `rpigpio`. This is NOT a keyed/paid dependency ‚Äî no approval gate.
- Do not weaken the driver-import gate into always-preview-only; the panel must actually
  come up on real hardware.
