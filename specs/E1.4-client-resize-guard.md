# E1.4: Client-Resize-Guard — Größenabweichung darf Server-Dithering nie zerstören

**Status:** Draft
**Abhängigkeiten:** E1.2 (gemergt) — der Guard liegt im selben Funktionskörper `display_image()` wie der Artefakt-Schreibpunkt; E1.2-Verhalten bleibt unangetastet
**Blockiert:** nichts hart; Vorbedingung für verlässliche L3-Hardware-Diagnosen — ein stilles Re-Quantisieren im Client würde jede Panel-Qualitätsanalyse verfälschen
**Zuständig:** pi-client — NUR `client/client.py` + `client/test_client.py`

## Ziel

Der Server liefert ein bereits exakt auf die Panel-Palette gedithertes, indiziertes PNG. Weicht dessen Größe von der Panelgröße ab (Fehlkonfiguration), darf der Client-Resize dieses Dithering nie zerstören: Es überleben ausschließlich Palettenfarben (`Image.Resampling.NEAREST`), und die Abweichung wird als `logger.warning` mit beiden Größen sichtbar gemacht — statt wie heute still per LANCZOS zu interpolieren und dem Waveshare-Treiber die (fehlerhafte, ditheringfreie) Re-Quantisierung zu überlassen.

### Verifizierte Fakten zum Resize-Pfad

1. **Server-Output ist Mode "P":** `quantizeToPalette` (server/internal/services/preview.go Z. 789–803) liefert `*image.Paletted`; Gos `png.Encode` schreibt daraus ein indiziertes PNG (Farbtyp 3); Pillow öffnet es als Mode `"P"`. `fetch_preview()` loggt den Mode (client.py Z. 71).
2. **Ablauf in `display_image()` heute (client.py, Stand nach E1.2-Merge):**
   - Z. 114–116: bei `img.size != (epd.width, epd.height)` → `img.resize(..., Image.Resampling.LANCZOS)` + `logger.info("Resized to %dx%d", ...)` — INFO-Level, und nur die **Ziel**größe wird geloggt.
   - Z. 118–129: Modus-Konvertierung — 6-Farb-Pfad (`len(colors) > 2`): `convert("RGB")`; B/W-Pfad: `convert("L").point(lambda x: 0 if x < 128 else 255, "1")`.
   - Z. 131: `save_last_sent_artifact(img)` (E1.2).
   - Z. 133: `epd.display(epd.getbuffer(img))`.
   **Die Reihenfolge Resize → Konvertierung ist heute schon korrekt** (Threshold wirkt auf das bereits skalierte Bild); diese Spec schreibt sie fest (AC3), sie wird NICHT umgestellt.
3. **Pillow-Koerzierung (empirisch verifiziert, Pillow 12.2.0):** Für Mode-`"P"`- und Mode-`"1"`-Bilder ignoriert `Image.resize()` den übergebenen Resample-Filter still und nutzt intern NEAREST. Ein 400×240-P-Bild mit 6-Farb-Palette bleibt nach „LANCZOS“-Upscale auf 800×480 bei exakt 6 Farben. Der heutige Mismatch-Pfad überlebt also **nur durch diese stille Koerzierung** — ohne Warnung, und ohne dass der Code die Invariante ausdrückt.
4. **Der echte Killer-Pfad ist RGB-/L-Input:** Dasselbe Bild als RGB (nur Palettenfarben, hochfrequentes Muster) erzeugt nach LANCZOS-Upscale 400×240 → 800×480 **339 unique Farben** (empirisch). Solche Zwischentöne re-quantisiert der Waveshare-Treiber dann selbst, ohne sauberes Dithering. RGB-Input entsteht, sobald der Server-Output nicht paletted ist (Formatänderung, Raw-/Debug-Render, künftige Pillow-/Server-Versionen) — der Guard darf sich daher NICHT auf die Pillow-Koerzierung verlassen, sondern setzt NEAREST explizit, unabhängig vom Bild-Mode.
5. **Der Pfad tritt im Normalbetrieb nie ein:** Server rendert exakt 800×480, `epd.width/height` = 800/480. Größenabweichung = Fehlkonfigurations-Signal (falsches Display-Profil, falscher Server) — genau deshalb ist WARNING (nicht INFO) das richtige Level.
6. **Kein bestehender Test pinnt LANCZOS oder interpolierten Content:** `python3 -m unittest test_client` → OK, 25 Tests (verifiziert). `test_last_sent_artifact_color` nutzt zwar Oversize-Input (1600×960), vergleicht aber nur Artefakt gegen `getbuffer`-Bild — beide Seiten liegen NACH dem Resize, die Assertion bleibt mit NEAREST identisch gültig.
7. **E1.2-Schreibpunkt deckt den Resize-Fall bereits ab:** `save_last_sent_artifact` läuft nach Resize + Konvertierung; das Artefakt zeigt damit automatisch exakt das (ggf. resizete) Panel-Bild. E1.4 verändert an der Artefakt-Logik nichts.

### Bindende Richtungsentscheidung (Manager)

Bei Größenabweichung: `logger.warning` mit **beiden** Größen (Ist + Soll) + `Image.Resampling.NEAREST` — für alle Bild-Modes, beide Panel-Pfade. Kein Verwerfen des Bildes, kein Abbruch: Der Refresh läuft durch (degradierte Darstellung mit korrekten Palettenfarben ist besser als gar kein Bild, und die WARNING macht die Fehlkonfiguration sichtbar).

## Akzeptanzkriterien

Alle AC sind durch die unter „Verifikation" genannten Kommandos maschinell prüfbar; pro AC ist der beweisende Test benannt. Alle neuen Tests nutzen `ArtifactSandboxMixin` (AC7-Regel aus E1.2: kein Test berührt das echte `/tmp`).

1. **AC1 — NEAREST bei Paletted-Input (Server-Normalfall bei Fehlkonfiguration):** Ein Mode-`"P"`-Input in Fremdgröße (400×240, 6-Farb-Palette, nicht-uniformes Muster — z. B. Schachbrett über alle 6 Palettenindizes) durchläuft `display_image()` mit `COLOR_DISPLAY_CONFIG`: Ergebnis `True`, `epd.display()` aufgerufen, das an `getbuffer()` übergebene Bild ist 800×480 und enthält ausschließlich die 6 Palettenfarben.
   *Beweis:* `test_resize_mismatch_paletted_input_preserves_palette` — `RecordingEPD`; unique-Farb-Zählung z. B. via `getbuffer_image.convert("RGB").getcolors(1_000_000)`, Assertion: Farbmenge ⊆ {`#000000`, `#FFFFFF`, `#FF0000`, `#00FF00`, `#0000FF`, `#FFFF00`}.

2. **AC2 — NEAREST bei RGB-Input (Regressionsbeweis gegen LANCZOS):** Derselbe Testfall mit RGB-Input in Fremdgröße (400×240, ausschließlich Palettenfarben, hochfrequentes Muster): Das an `getbuffer()` übergebene Bild enthält nach Resize + `convert("RGB")` ausschließlich Palettenfarben. **Dieser Test MUSS gegen den heutigen LANCZOS-Code rot sein** (Kontext Punkt 4: 339 Farben) — er beweist die Änderung.
   *Beweis:* `test_resize_mismatch_rgb_input_preserves_palette` — Assertions wie AC1; Negativprobe siehe „Verifikation".

3. **AC3 — B/W-Pfad: NEAREST + Threshold NACH Resize:** Ein B/W-Input in Fremdgröße (Mode `"P"` mit Schwarz/Weiß-Palette oder RGB mit reinen 0/255-Pixeln) durchläuft den B/W-Pfad (`BW_DISPLAY_CONFIG`): Das an `getbuffer()` übergebene Bild hat `mode == "1"`, Größe 800×480, und enthält nach `convert("L")` nur die Werte {0, 255}. Die bestehende Reihenfolge (Resize Z. 114–116 VOR der Threshold-Konvertierung Z. 127–128) bleibt erhalten und wird hiermit festgeschrieben — der Threshold darf nie auf das Original wirken und dann interpolierend skaliert werden.
   *Beweis:* `test_resize_mismatch_bw_threshold_after_resize` — Assertions wie oben; Reihenfolge zusätzlich Review-Punkt in Gate L5.

4. **AC4 — WARNING mit beiden Größen:** Im Mismatch-Fall loggt `display_image()` genau eine Meldung auf WARNING-Level, die **Ist-Größe und Soll-Größe** enthält (z. B. `"Preview size 400x240 does not match display 800x480 - resizing with NEAREST (check server display settings)"`). Die Ist-Größe muss VOR dem Resize erfasst werden. Das bisherige `logger.info("Resized to %dx%d", ...)` entfällt bzw. geht in der WARNING auf.
   *Beweis:* in den Tests aus AC1–AC3 via `assertLogs("eink-client", level="WARNING")`; Assertion: eine Record-Message enthält sowohl `400x240` als auch `800x480`.

5. **AC5 — Kein Resize, keine WARNING bei exakter Größe:** Ein 800×480-Input erzeugt keinerlei WARNING und kein `resize()`; das Verhalten des Normalpfads ist byte-identisch zu vorher (Bild geht unverändert durch die Konvertierung an den Treiber).
   *Beweis:* `test_no_warning_on_exact_size` — `assertNoLogs("eink-client", level="WARNING")` (Python ≥ 3.10) oder Spy auf `img.resize`; plus `getbuffer_image.tobytes()`-Vergleich gegen die erwartete Konvertierung des Inputs.

6. **AC6 — E1.2-Artefakt bleibt exakt (auch im Resize-Fall):** `save_last_sent_artifact()` wird nicht verändert; im Mismatch-Fall ist das Artefakt pixel-identisch mit dem an `getbuffer()` übergebenen (resizeten, konvertierten) Bild und hat 800×480.
   *Beweis:* `test_resize_artifact_matches_driver_image` — `RecordingEPD` + `tobytes()`-Vergleich Artefakt ↔ `getbuffer_image` (Muster aus `test_last_sent_artifact_color`).

7. **AC7 — Bestand bleibt grün:** Alle bestehenden 25 Tests in `client/test_client.py` laufen unverändert durch (Kontext Punkt 6: keiner pinnt LANCZOS).
   *Beweis:* Testlauf unter „Verifikation".

## Non-Goals

- **Kein Verwerfen/Ablehnen** von Bildern in Fremdgröße — der Refresh läuft immer durch (bindende Richtungsentscheidung).
- **Keine Aspect-Ratio-Behandlung** (Letterboxing, proportionales Skalieren) — 400×240 → 800×480 wird verzerrungsfrei sein, andere Verhältnisse verzerren wie heute; das ist ein Fehlkonfigurations-Zustand, kein Feature.
- **Keine Server-Änderungen** (Go), keine API-Änderungen, keine neuen Env-Variablen.
- **Keine Änderung an der E1.2-Artefakt-Logik** (`save_last_sent_artifact`, Schreibpunkt, Atomik) und keine am No-Hardware-Pfad (`preview_output.png`).
- **Kein Entfernen der clientseitigen Konvertierung** (Passthrough) — separater Task, erst nach E1.1-Folgearbeiten.
- Keine konfigurierbare Resample-Wahl — NEAREST ist die einzig korrekte Option für vor-gedithertes Material.

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| NEAREST-Upscale wirkt blockig/pixelig | Akzeptiert und gewollt: Der Pfad ist ein Fehlkonfigurations-Zustand; Palettenintegrität schlägt Glätte. Die WARNING weist den Betreiber auf die eigentliche Ursache hin. |
| NEAREST-**Downscale** (Server größer als Panel) verwirft Pixel und kann Dither-Muster ausdünnen | Ebenfalls akzeptiert: Es überleben ausschließlich Palettenfarben (Kerninvariante); visuelle Güte ist im Fehlkonfigurations-Fall sekundär. Interpolation wäre in beiden Richtungen falsch. |
| Verhaltensänderung im heutigen P-Mode-Pfad | Praktisch keine: Pillow nutzte dort intern schon NEAREST (Kontext Punkt 3); neu sind nur Log-Level/-Inhalt und die explizite Absicht im Code. |
| WARNING-Spam im Log | Max. eine WARNING pro Refresh; Default-Intervall 3600 s ⇒ ~24/Tag — gewolltes Dauersignal, bis die Fehlkonfiguration behoben ist. |
| Künftige Pillow-Version ändert die stille P-Mode-Koerzierung | Genau dagegen härtet dieser Task: explizites NEAREST macht das Verhalten versionsunabhängig. |
| **Rollback** | Ein einzelner kleiner Client-Commit (Resample-Konstante + Log + Tests), keine persistenten Formate, keine API-Änderungen — `git revert <commit>` genügt. |
