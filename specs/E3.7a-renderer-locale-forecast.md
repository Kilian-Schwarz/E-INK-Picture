# E3.7a — German localization of forecast strings + forecast_days 4→7

## Problem
The panel renderer mixes languages: date/clock lines are German (hardcoded arrays
in `preview.go`), but forecast **weekday names** come from Go `time.Weekday().String()`
(English, `weather.go:293`) and **weather-condition descriptions** come from English
maps (`weatherDayMap`/`weatherNightMap`, `weather.go:504-530`). Result on-panel:
"Monday … Partly cloudy" next to German date lines. Separately, `forecast_days=4` is
hardcoded (`weather.go:244`) plus two structural caps (`weather.go:281` make-cap,
`weather.go:282` loop `i<4`), so the `week-planner.json` widget that asks for
`days: 7` (`week-planner.json:147`) silently renders at most 4 rows.

There is **no shared locale mechanism** — German is only two duplicated hardcoded
arrays in `preview.go`. Target language is **German** (the panel is German; this is a
consistency fix, not a new user-facing language toggle).

## Acceptance criteria
- **AC1 — German weekdays.** Forecast weekday names rendered on the panel are German.
  Where full names are shown, use full German ("Montag"…). Where the compact_row layout
  currently byte-slices `day.Weekday[:3]` (`preview.go:591`), use proper German short
  forms **Mo/Di/Mi/Do/Fr/Sa/So** (not a naive 3-byte slice of the German full name,
  which would give "Die"/"Mit"/"Don"). No English weekday text remains in any layout
  (`preview.go:591,593,597,604-605`; `weather.go:407`; `widgets/forecast.go:35`).
- **AC2 — German conditions.** Weather-condition descriptions are German (WMO code →
  German text), covering every code currently in `weatherDayMap` and `weatherNightMap`
  (day + night variants), with a German fallback replacing `"Unknown"` (`weather.go:542`).
- **AC3 — Single source of truth.** Do not add a third divergent copy. Introduce one
  German weekday map/helper and one German WMO map; the `"Weekday: min-max°C desc"`
  summary line (`weather.go:407`) uses them too. If practical, the German weekday data
  reuses/consolidates the existing arrays in `preview.go:486-497` rather than duplicating.
- **AC4 — JS renderer consistency.** `server/static/js/widgets.js` must not reintroduce
  the mix. It consumes server-provided `day.weekday`/`desc` (`widgets.js:191`), so
  Germanizing the Go source should propagate — VERIFY it does (short-form logic in JS,
  if any, must match Mo/Di/… too). No English/German mix in either the Go panel render
  or the JS preview for the same design.
- **AC5 — forecast_days 7.** `weather.go:244` URL param `forecast_days=4`→`7`; the make
  cap (`weather.go:281`) and loop bound (`weather.go:282`) → 7. `WeatherData.Daily` can
  hold 7 entries; week-planner renders up to 7 forecast rows. `fillForecastContent`
  (`preview.go:573`) already clamps by the widget `days` prop — unchanged.
- **AC6 — Goldens.** The golden-file suite must stay green. Golden PNGs that contain
  forecast/weather text WILL legitimately change (that is the point) — regenerate them
  intentionally, keep `TestPaletteExactness`/determinism/font-pinning green, and list in
  the PR which goldens changed and why. No unrelated golden churn.
- **AC7 — Tests + static checks.** New/updated unit tests assert: German full + short
  weekday output; German condition text for ≥2 representative WMO codes (one day, one
  night); a 7-day forecast parse yields 7 `Daily` entries. `gofmt -l` empty, `go vet`
  clean, full `go test ./... -count=1` green including `-race`.

## Non-goals
- No settings-driven language switch (German is fixed).
- No change to the clock/date placeholder logic (already German).
- No refactor of the widgets-package vs preview.go duplication beyond what AC3 needs.
