# E3.3: Responsive Designer-Layout für Handy & Tablet

**Status:** Draft
**Abhängigkeiten:** E3.1 (gemergt, `af69822`) — Pointer-Events-Konventionen gelten unverändert. E3.3 wird bewusst **VOR E3.2 (Gesten)** gezogen: Bei 390 px Viewport-Breite hat `#canvas-area` heute **0 px sichtbare Breite** — ohne Layout ist auf dem Handy kein Canvas da, an dem Gesten etwas nützen könnten.
**Koordination:** E3.2 (Pinch/Pan/Long-Press) baut auf dem hier eingeführten Zoom-Fit und den Bottom-Sheets auf; **Sheet-Drag (Swipe zum Öffnen/Schließen) ist explizit E3.2** — in E3.3 gibt es nur Tap-Toggle. Parallele E3.x-Tasks fassen `designer.html` an — vor Beginn `git pull --rebase origin main`.
**Zuständig:** frontend-designer — nur `server/static/js/`, `server/static/css/`, `server/templates/designer.html` + `CHANGELOG.md`. Kein Go-Code, kein `client/`.

## Ziel

Der Designer wird auf Handy (< 768 px) und Tablet (768–1024 px) benutzbar: Der Canvas ist der Star und nimmt mobil die volle Breite ein; die beiden 260-px-Seiten-Panels werden mobil zu **Bottom-Sheets** (untere Tab-Leiste: Widgets | Layers | Properties, immer nur ein Sheet offen), auf dem Tablet zu **einklappbaren Panels mit Icon-Leiste** (Default eingeklappt). Ein **Zoom-Fit** passt das 800×480-Design beim Laden und bei Resize/Orientation-Change in den sichtbaren Bereich ein. **Ab 1024 px bleibt das heutige Desktop-Layout pixel- und verhaltensidentisch** (hartes Regression-AC).

**Warum:** Beide Panels haben feste `width/min-width/max-width: var(--panel-width)` = 260 px (`panels.css:3–6`, Variable in `designer.css:21/42`); der Layout-Container `.designer-layout` clippt mit `overflow: hidden` (`designer.css:284–288`), `html, body` ebenfalls (`designer.css:56–59`). Bei 390 px Breite belegen die Panels 520 px — `#canvas-area` (flex: 1) wird auf 0 gequetscht, der Canvas ist auf dem Handy schlicht unsichtbar. Die einzige Media-Query im gesamten Frontend ist `media-dashboard.css:668` (Dashboard-Grid-Spalten, `max-width: 900px`) — der Designer selbst hat null Responsive-Verhalten, keinen Resize-Handler, keine Panel-Toggle-Logik. E3.1 hat die Eingaben touch-fähig gemacht; E3.3 macht sie auf kleinen Screens erreichbar.

### Bindende Architektur-Richtung (Manager, präzisiert nach Bestandsaufnahme)

1. **Breakpoints (bindend):** Mobil = `@media (max-width: 767.98px)`, Tablet = `@media (min-width: 768px) and (max-width: 1023.98px)`, Gemeinsames („compact") = `@media (max-width: 1023.98px)`. **≥ 1024 px gilt keine einzige neue Regel** außer `display: none` für die neuen Elemente. Alle neuen Styles leben in `responsive.css`, geladen als LETZTES Stylesheet (Kaskaden-Sieg über `panels.css` bei gleicher Spezifität — deshalb kein einziges `!important`).

2. **Mobil (< 768 px) — Canvas ist der Star:**
   - Beide Asides werden per CSS zu **Bottom-Sheets**: `position: fixed; left/right: 0; bottom: calc(var(--tabbar-h) + env(safe-area-inset-bottom, 0px)); height: 50dvh` (Fallback-Zeile `height: 50vh` davor); `width/min-width/max-width` zurückgesetzt; geschlossen `transform: translateY(100%)`, offen `transform: none`, Transition ~0.25 s; `z-index: 300`. Dadurch sind die Panels **out of flow** → `#canvas-area` bekommt die volle Breite, **der Canvas-Viewport verschiebt sich beim Öffnen/Schließen NICHT** (Sheet overlayt die untere Canvas-Hälfte — bewusste Entscheidung, kein Reflow, kein Re-Fit, kein `calcOffset`-Zwang pro Sheet-Toggle).
   - **Untere Tab-Leiste** `<nav id="mobile-tabbar">` (`position: fixed; bottom: 0`; Höhe `var(--tabbar-h)` = 48 px + `padding-bottom: env(safe-area-inset-bottom, 0px)`; `z-index: 320`) mit drei `<button type="button" data-sheet="widgets|layers|properties">`. `.designer-layout` bekommt mobil `margin-bottom: calc(var(--tabbar-h) + env(safe-area-inset-bottom, 0px))`, damit der Fit die Leiste nie überlappt.
   - **Sheet-State:** genau EIN Attribut `document.body.dataset.sheet ∈ {'', 'widgets', 'layers', 'properties'}` — „immer nur ein Sheet offen" ist damit strukturell garantiert. `widgets`/`layers` öffnen die linke Aside (CSS blendet die jeweils andere Section aus: `body[data-sheet="widgets"] #layers-section { display: none }` und umgekehrt), `properties` die rechte. Tap auf aktiven Tab = schließen (Toggle).
   - **Drag-Handle:** `.sheet-handle` (Button, erste Zeile jeder Aside, Pill-Optik, `aria-label="Close panel"`) schließt per Tap. **KEIN Backdrop:** Ein klick-fangender Backdrop würde AC2 (Canvas-Interaktion bei offenem Sheet) brechen — der unbedeckte Canvas-Bereich bleibt voll interaktiv, Schließen geht über Handle, Tab-Toggle oder Escape.
   - **Keine Auto-Öffnung:** Objekt-Selektion öffnet das Properties-Sheet NICHT (der Canvas bliebe sonst bei jedem Tap halb verdeckt). Stattdessen bekommt der Properties-Tab bei aktiver Selektion eine Indikator-Klasse `.has-selection` (Punkt-Badge); Hook über `canvas.on('selection:created'/'selection:updated'/'selection:cleared')` in `responsive-layout.js`.
   - **Topbar kompakt:** Sichtbar bleiben Logo (Dashboard-Zugang!), `#design-select` (`flex: 1; min-width: 0`), `#design-name-edit-btn` ✎, `#save-btn`, Burger `#topbar-menu-btn`. Per CSS versteckt: `#design-name-display`, `#active-indicator` (redundant zum Select). **Ins Burger-Menü wandern (DOM-Reparenting, s. Punkt 4):** `#new/clone/delete-design-btn`, `#undo-btn`, `#redo-btn`, `.zoom-controls`, `#refresh-now-btn`, `#preview-btn`, `#history-btn`, `#activate-btn`, `#settings-btn`, `#theme-toggle`. (New/Clone/Delete sind zusätzlich über das Dashboard abgedeckt: Create-Flow + Kontextmenü Rename/Duplicate/Delete, `design-dashboard.js:150–223`.)
   - `.statusbar` wird mobil ausgeblendet (rein informativ, Platz gehört der Tab-Leiste). Tablet und Desktop behalten sie.
   - Kleine mobile Härtungen (alle in der Mobil-Query): `.history-sidebar { width: 100%; }`; `.modal-content, .modal-content-lg { width: calc(100vw - 24px); }`. In der Compact-Query: `.modal-content-lg { width: min(860px, calc(100vw - 24px)); }` (860 px > 768-px-Viewport).

3. **Tablet (768–1023.98 px) — einklappbare Panels, Default eingeklappt:**
   - Beide Asides bleiben **im Flex-Flow mit fester Breite 44 px** (Icon-Leiste). Der neue `.panel-body`-Wrapper (umschließt die bestehenden `.panel-section`s, desktop styling-neutral) ist eingeklappt `display: none`; ausgeklappt wird er als **Flyout-Overlay** absolut positioniert (`position: absolute; top: 0; bottom: 0; width: 260px;` links bei `left: 44px`, rechts bei `right: 44px`; `background: var(--bg-secondary)`, Border + Schatten, `overflow-y: auto`, `z-index: 300`; `.designer-layout` bekommt in der Tablet-Query `position: relative`). **Der Canvas-Flow ändert sich beim Toggle NICHT** (kein Reflow, kein Re-Fit).
   - State: `body.dataset.panelLeft/panelRight ∈ {'', 'open'}` — beide unabhängig, Default `''` (eingeklappt), keine Persistenz (bewusst: Scope klein halten).
   - `.panel-toggle`-Buttons (je einer pro Aside, oben in der Icon-Leiste, `title="Widgets & Layers"` bzw. `"Properties"`) toggeln den State.
   - Topbar Tablet: wie mobil, aber `#undo-btn`, `#redo-btn` und `.zoom-controls` bleiben sichtbar (Reparenting-Liste = Mobil-Liste minus diese drei); `#new/clone/delete-design-btn` wandern auch hier ins Menü.

4. **Burger-Menü per DOM-Reparenting (bindend, statt CSS-Relocation):** CSS kann `#save-btn` nicht aus `.topbar-right` „herausziehen" und ein Umsortieren der HTML-Reihenfolge würde die Desktop-Optik ändern (AC3). Deshalb: `ResponsiveLayout.applyMode()` verschiebt beim Eintritt in einen Compact-Modus die definierte Button-Liste per `appendChild` in den Container `#topbar-menu` (fixed Dropdown unter der Topbar, `z-index: 350`, gruppierte Zeilen) und stellt sie beim Eintritt in Desktop **an die gemerkte Originalposition zurück** (Parent + `nextSibling` vor dem ersten Verschieben einmalig erfassen). Event-Listener überleben Reparenting; verlorene Vorfahren-Styles (`.topbar-center button`, `.topbar-right button…`, designer.css:253–281) MUSS `responsive.css` für `#topbar-menu button` gleichwertig neu definieren. `#topbar-menu-btn` toggelt `body.classList('topbar-menu-open')`; Menü schließt bei Tap auf einen Menü-Eintrag, Escape oder erneutem Burger-Tap.

5. **Zoom-Fit (`canvas-manager.js`):**
   - Neue Methode:
     ```js
     fitToViewport() {
         if (!window.matchMedia('(max-width: 1023.98px)').matches) return; // desktop untouched
         const area = document.getElementById('canvas-area');
         if (!area || !this.canvas) return;
         const M = 8; // must match centerCanvas() compact min padding
         const fit = Math.min(
             (area.clientWidth  - 2 * M) / this.displayConfig.width,
             (area.clientHeight - 2 * M) / this.displayConfig.height
         );
         this.setZoom(fit);        // clamps 0.25–4, setDimensions + centerCanvas
         this.canvas.calcOffset(); // pointer coords after layout change
     }
     ```
   - `centerCanvas()`: Min-Padding wird `const minPad = window.matchMedia('(max-width: 1023.98px)').matches ? 8 : 40;` (heute hart 40 — würde mobil 454 px > 390 px erzwingen und horizontales Scrollen in `#canvas-area` verursachen). Desktop-Pfad bleibt wörtlich 40.
   - Aufrufe: einmal in `ResponsiveLayout.init()` (nach `CanvasManager.init()`, s. Script-Reihenfolge), am Ende von `setDisplayConfig()`, sowie bei `window.resize` (debounced 150 ms), `orientationchange` und `change` der beiden matchMedia-Queries. Manuelles Zoomen (Buttons/Ctrl+Wheel) bleibt möglich; der nächste Resize-Event fittet neu (dokumentierte, akzeptierte Semantik).
   - **Moduswechsel** (`matchMedia.change`): alle Responsive-States zurücksetzen (`data-sheet=''`, `topbar-menu-open` entfernen, `panelLeft/Right=''`, Reparenting rückgängig bei Desktop-Eintritt); bei Desktop-Eintritt Zoom NICHT anfassen (nur `centerCanvas()` + `calcOffset()`).

6. **iOS-Safari-Realität:** `body { height: 100vh; height: 100dvh; }` in der Mobil-Query (dvh folgt der dynamischen URL-Bar; der dabei feuernde `resize` triggert den debounced Re-Fit). `viewport-fit=cover` ins Viewport-Meta (designer.html:5), `env(safe-area-inset-bottom, 0px)` an Tab-Leiste und Sheet-`bottom`. Headless nicht testbar → statische Asserts (Gate L1) + Gerätetest (Gate L3).

7. **z-index-Plan (bindend):** Sheets/Flyouts 300 · Tab-Leiste 320 · Topbar-Menü 350 — alle UNTER `dashboard-overlay` (500), `history-sidebar` (600), Modals (1000), Notifications (9999), Kontextmenü (10000). Damit overlayen Dashboard/Modals wie heute alles Neue.

8. **Stack-Invarianten:** Vanilla CSS/JS, kein Framework, kein npm/`package.json` im Repo, kein Build-Step, keine neue Runtime-Dependency, Fabric bleibt 5.3.1 (designer.html:404). Pointer-Events-Konventionen aus E3.1 gelten (neue Interaktionen sind reine `click`-Handler — tap-kompatibel, kein `mouse*`).

## Akzeptanzkriterien

1. **AC1 — Mobil sichtbar & gefittet (390×844):** `#canvas-area` hat > 300 px sichtbare Breite (erwartet: 390 px, Panels sind out of flow) und das komplette 800×480-Design ist ohne Scrollen sichtbar: `CanvasManager.canvas.getZoom()` ≈ `min((areaW−16)/800, (areaH−16)/480)` (erwartet ≈ 0.47) ± 0.02, und die `getBoundingClientRect()` des Fabric-Wrappers `.canvas-container` liegt vollständig im Viewport UND oberhalb der Tab-Leiste (`left ≥ 0`, `right ≤ innerWidth`, `top ≥ 48`, `bottom ≤ innerHeight − 48`).
   *Beweis:* Gate L2, Viewport 390×844, JS-Asserts + Screenshot.

2. **AC2 — Sheets & Canvas-Interaktion:** Tap auf jeden der drei Tabs öffnet das jeweilige Sheet (`body[data-sheet]` gesetzt, Sheet-BCR sichtbar in der unteren Viewport-Hälfte, bei `widgets`/`layers` ist nur die jeweilige Section sichtbar); Tab-Wechsel schließt das vorige (nie zwei sichtbar — strukturell via Single-Attribut, zusätzlich per BCR-Assert); Tap auf Handle oder aktiven Tab oder Escape schließt. Canvas-Drag (synthetische Pointer-Sequenz nach E3.1-Protokoll auf einem UNBEDECKTEN Canvas-Punkt) funktioniert bei geschlossenem UND bei geöffnetem Sheet: Objekt-`left/top` ändern sich um `screenDelta / zoom` (± 2 px).
   *Beweis:* Gate L2, Viewport 390×844, DOM-Asserts + Objekt-Koordinaten-Diff.

3. **AC3 — Desktop-Regression (1440×900):** Pixel- und verhaltensidentisch. Screenshot vorher (main) vs. nachher (Branch) unter deterministischen Bedingungen: **0 abweichende Pixel** (pixelmatch, threshold 0). DOM-Asserts: `#mobile-tabbar`, `#topbar-menu-btn`, `.sheet-handle`, `.panel-toggle` haben `display: none`; beide `.panel` sind exakt 260 px breit; `#undo-btn.parentElement` ist `.topbar-center` und `#save-btn.parentElement` ist `.topbar-right` (kein Reparenting aktiv). Verhaltens-Smoke: Element-Drag per Maus + Save funktionieren (E3.1-Protokoll-A-Subset).
   *Beweis:* Gate L2, Viewport 1440×900, Screenshot-Diff + Asserts, Werte im MR.

4. **AC4 — Kein horizontales Scrollen:** In ALLEN drei Viewports (390×844, 768×1024, 1440×900) gilt `document.scrollingElement.scrollWidth <= window.innerWidth`, `topbar.scrollWidth <= topbar.clientWidth + 1` (die Topbar ist heute der Überläufer) und bei Default-Zoom `canvasArea.scrollWidth <= canvasArea.clientWidth + 1`.
   *Beweis:* Gate L2, Assert pro Viewport.

5. **AC5 — Tablet (768×1024):** Beide Panels default eingeklappt (Aside-Breite ≤ 48 px), `#canvas-area` ≥ 650 px breit, Fit-Zoom ≈ 0.83 ± 0.02; `.panel-toggle` öffnet das 260-px-Flyout als Overlay, wobei sich `#canvas-area`-Breite und `.canvas-container`-BCR NICHT ändern (kein Reflow); beide Flyouts unabhängig toggelbar; Widget-Klick im offenen linken Flyout fügt weiterhin ein Element hinzu, Properties-Inputs im rechten funktionieren.
   *Beweis:* Gate L2, Viewport 768×1024, BCR-Vergleich vor/nach Toggle.

6. **AC6 — Re-Fit bei Resize/Orientation:** Nach Viewport-Wechsel 390×844 → 700×390 (mobiles Querformat) ist das Design erneut vollständig sichtbar und `getZoom()` entspricht dem neuen Fit ± 0.02; nach Wechsel zurück ebenso. Moduswechsel 390→1440 räumt alle States (kein `data-sheet`, kein `topbar-menu-open`, Buttons an Originalposition).
   *Beweis:* Gate L2, `page.setViewport`-Sequenz + Asserts.

7. **AC7 — iOS-Viewport-Grundlagen (statisch):** `designer.html:5` enthält `viewport-fit=cover`; `responsive.css` enthält `100dvh` MIT `100vh`-Fallback-Zeile davor und `env(safe-area-inset-bottom` an Tab-Leiste und Sheet-Positionierung.
   *Beweis:* Gate L1 grep; Laufzeitverhalten auf echtem Gerät = Gate L3.

8. **AC8 — Stack- und Scope-Invarianten:** Kein `package.json`/npm im Repo, keine neue Dependency, kein Framework; `responsive.css` enthält **0 × `!important`** und außerhalb von `@media`-Blöcken ausschließlich Selektoren auf die NEUEN Elemente mit `display: none`; Bestands-CSS-Dateien byte-identisch; der Diff berührt nur die in „Betroffene Dateien" gelisteten Pfade; `cd server && go build ./... && go test ./... -count=1` grün; `CHANGELOG.md` aktualisiert.

## Non-Goals

- **Kein Pinch-Zoom, kein Two-Finger-Pan, KEIN Sheet-Swipe/-Drag** — E3.2. Der `.sheet-handle` ist in E3.3 reine Tap-Fläche mit Drag-Handle-OPTIK.
- **Keine Template-Galerie** — E3.5.
- **Keine Funktions-Änderungen an Panel-Inhalten** — Widgets/Layers/Properties werden nur anders positioniert/ein-geblendet; Markup und Logik der Inhalte bleiben (einzige HTML-Änderungen: Wrapper + IDs + neue Buttons).

## Risiken

| Risiko | Bewertung / Mitigation |
|---|---|
| Fabric-Offsets stimmen nach Layoutwechsel nicht mehr (Klicks/Taps treffen daneben) | `fitToViewport()` ruft `canvas.calcOffset()` nach jedem `setZoom`; Sheets/Flyouts sind Overlays — die Canvas-Position ändert sich bei Panel-Toggles NIE (Kern-Designentscheidung). Fabric selbst ruft `calcOffset` nur bei `window.resize` (`_onResize`). AC2/AC5 beweisen Treffgenauigkeit per Drag-Assert. |
| CSS-Spezifität: `panels.css .panel { width: 260px }` gewinnt gegen neue Regeln | `responsive.css` wird als letztes Stylesheet geladen (Kaskade bei gleicher Spezifität); Review-Punkt in L5; verboten: `!important` (AC8) und Edits an Bestands-CSS (L1 Schritt 1). |
| Neue Basis-Regeln oder der `.panel-body`-Wrapper verändern Desktop-Rendering minimal | Basis-Regeln sind ausschließlich `display: none` auf neue Elemente; `.panel-body` ist ein style-loser Block-Wrapper im Flex-Column-Panel (rendering-neutral). Beweis ist nicht Argumentation, sondern der 0-Pixel-Diff (AC3). |
| DOM-Reparenting verliert Styles oder verwirrt Selektoren | Listener überleben `appendChild`; verlorene Vorfahren-Styles werden für `#topbar-menu` explizit nachgebaut (Architektur-Richtung 4); kein JS-Code selektiert Buttons über `.topbar-*`-Vorfahren (Bestandsaufnahme: alle per `getElementById`). Restrisiko deckt Protokoll M/R ab. |
| Sheet-z-Index kollidiert mit Modals/Kontextmenü/History | Fester Plan (Richtung 7): 300/320/350 liegen unter 500/600/1000/9999/10000 — Modals und Kontextmenü overlayen die neuen Flächen wie heute. |
| iOS-dvh-Sprünge (URL-Bar) verschieben Tab-Leiste/Fit | dvh + debounced Re-Fit auf `resize`; falls L3 Springen zeigt, ist der dokumentierte Fluchtweg `svh` statt `dvh` (eine Zeile). Headless nicht abbildbar — bewusstes L3-Thema. |
| Properties-Sheet + Bildschirmtastatur (Input-Fokus im 50-dvh-Sheet) | Bekanntes Mobile-Web-Thema; E3.3 akzeptiert das Browser-Default-Verhalten (Viewport schiebt), L3 sammelt Findings. Kein Sonder-Code in E3.3. |
| Nicht-800×480-Displayprofile | Fit rechnet generisch über `displayConfig` (beide unterstützten Panels sind 800×480; `setDisplayConfig`-Hook fittet bei Profilwechsel neu). |
| Puppeteer-Pixel-Diff flaky (Fonts, Notifications, Widget-Inhalte, Statuszeile) | Deterministisches Setup ist Teil des Protokolls: statisches Testdesign ohne Zeit-/Wetter-Widgets, frisches Datenverzeichnis (Statuszeile „Display: waiting"), `document.fonts.ready` + 4 s Wartezeit, identisches Chrome für beide Läufe. |
| Paralleler E3.x-Churn an `designer.html` | Trunk-based, kleiner Diff, früh mergen; `git pull --rebase origin main` vor Beginn; semantische Konflikte (z. B. E3.2 fasst `canvas-manager.js` an) an den Manager. |
